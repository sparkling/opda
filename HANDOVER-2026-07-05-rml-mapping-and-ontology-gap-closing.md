# Handover — RML Mapping Adoption, morph-kgc → RMLMapper Migration, Ontology Gap-Closing

**Date:** 2026-07-05

## TL;DR

Adopted [RML](https://rml.io) (RDF Mapping Language) as OPDA's independent, machine-checkable trace between the PDTF v3 JSON Schema and the OPDA ontology (ODR-0035, ADR-0057), then spent the bulk of this session closing the gap between "resources the ontology has minted" and "resources the RML mapping can actually trace back to real source JSON." Coverage went from 0 to **444/464 schema-generated resources mapped (95.7%)**. Along the way: found and fixed a real bug in the chosen RML engine, migrated the whole mapping to a different engine when that engine's remaining limitations became blocking, found and fixed ~10 genuine pre-existing ontology defects (several "ratified in an ODR but never actually implemented" cases), and did a real (if bounded) investigation into whether RuvNet's MetaHarness Darwin Mode tooling could accelerate this kind of work (ADR-0058 — short answer: yes, but not the way it was first tried, twice).

Everything is green: `make ci-ontology` (repo root) and `make rml-test` / `rml-pytest` / `provenance-test` / `dct-audit` (from `source/03-standards/rml/`) all pass. The remaining 20 gap items are all genuinely characterized (data-absence, architectural-by-design, or real policy decisions) — this session swept every "should be easy" case it could find.

## Why this session happened

OPDA's ontology (`source/03-standards/ontology/`, generated by `tools/opda-gen/`) is built from ~15 prior Council sessions modelling the PDTF v3 property-transaction JSON Schema. Over time it accumulated a real risk: does every `opda:` class/property still trace to something real in the source schema, or has drift crept in (a term minted for a JSON field that changed shape, a term whose citation is stale, a genuinely fabricated binding)? ODR-0035 answers this by adopting RML as a **bidirectional verification mechanism** — not an executable ETL pipeline, a piece of **declarative provenance documentation** that a harness can materialise against real PDTF instances and validate with SHACL. `docs/adr/ADR-0057-rml-mapping-implementation.md` is the engineering record of building this; `docs/adr/ADR-0058-metaharness-darwin-mode-for-rml-gap-closing.md` is a side-investigation into using evolutionary tooling to accelerate authoring; this handover is the connective narrative.

## What changed (63 commits, categorized)

### 1. RML mapping standard adopted (early commits)
- `source/03-standards/rml/` created: `mapping/opda-pdtf.rml.ttl` (the mapping, now ~2900 lines / 35 numbered "M-blocks"), `harness/` (validation scripts), `testdata/` (4 fixtures: 01-conformant-full, 02-minimal, 03-multi-participant, 04-negative-invalid), `provenance-index.json`/`.md`, `gap-register.md`, `CONTRACT.md`, `README.md`.
- Initial engine choice: **morph-kgc** (Python). Chosen because Apache Jena `arq`/`shacl` (ADR-0036/0037) already gate this project's RDF toolchain and morph-kgc supports RML-FNML (function execution) without adding rdflib/pyshacl.

### 2. A real morph-kgc bug found and fixed, then the engine migrated anyway
- Found: `morph_kgc.data_source.data_file._read_json`'s final row filter (`dropna(how='any')`, no `subset=`) drops any row with a NaN in *any* column of the fully-flattened dataframe — not just the columns a given `TriplesMap` uses. Since real JSON arrays almost always have items with different optional fields, this silently discarded valid data across many "this can't be mapped" cases. Patched via `harness/morph_kgc_patch.py` (a runtime monkeypatch, kept for its bug-investigation history, no longer imported).
- The fix unblocked several gaps, but two remained genuinely blocked under morph-kgc even after: `opda:hasParticipant`'s role-filter (needs a JSONPath filter embedded mid-reference; morph-kgc only supports filters as the whole iterator) and a severe performance cliff (~15-20s/rule) binding onto the root `Transaction` iterator.
- **Migrated the whole engine to RMLMapper** (Java reference implementation) — see ADR-0057's Amendments for the full reasoning and the several real engine-behaviour deltas found along the way (JSONPath dialect differences, a classic-vs-modern FNML vocabulary mismatch, an `xsd:date` truncation regression, a genuinely dangerous multi-placeholder cross-product bug). `harness/run_mapping.py` now self-provisions RMLMapper (downloaded + SHA-256-verified into the gitignored `.rmlmapper/` cache) and shells out to it directly — no INI generation, no Python engine dependency.

### 3. Gap-closing (the bulk of the commit list)
Closed properties/classes region by region: ownership/leasehold (41 props), valuationComparisonData, localSearches, waterAndDrainage/electricity/buildInformation, RiskAssessment's `subCategory`/`riskIndicator`/`peril`/`hasSubAssessment`, `hasRoomDimension`, `constructionType`, `planningStartDate`/`buildingControlStartDate`, `ownerType` (co-typing legal owners as `opda:Proprietor`), `TransactionChain`/`Milestone`/`Claim`/`Evidence`/`VerificationActivity` (four classes that were wrongly written off as "structurally impossible" — see "Lessons" below), `Address` (6 structural properties + real occurrence binding), `TrustFramework`, `recordsEstate`, `leaseTerm`/`dependsOnTransaction` (both "ratified in an ODR but never actually declared" bugs), `schoolType`, `Survey`, `Proprietorship`'s `mediates` conditional, an IAO-ontology crosswalk, `opda:DescriptiveProperty` (a meta-class + CI-gate fix), and four more small ratified-but-undeclared terms (`Verifier`, `hasSteward`, `variantPredicate`, `variantValue`).

### 4. A real, pre-existing SHACL bug found and fixed
`MilestoneVarianceRule`'s SPARQL WHERE clause queried a same-node co-location of `occurredAtTime`/`plannedAtTime` that Council session-007's own ratified Q6 verdict had explicitly rejected in favour of a separate `prov:Plan` node — the rule had been silently, permanently dormant. Fixed to traverse the real, ratified `prov:qualifiedAssociation → prov:Association → prov:hadPlan` structure.

### 5. MetaHarness Darwin Mode investigated (ADR-0058)
A genuinely useful side-quest: does RuvNet's evolutionary "Darwin Mode" tooling help author RML/ontology fixes? Tried the packaged `metaharness_evolve` tool naively — failed (flat, uninformative result). Challenged twice by the user, corrected twice: the architecture *can* target arbitrary repo files (not just its own seven policy surfaces), and the actual way to author a working custom task is via the package's public `bench.makeSuite()`/`saveSuite()` API, not the CLI's bare scaffold or a hand-edited JSON file (which trips a real, working tamper-check). `ADR-0058` was rewritten as a general-purpose playbook (`## Rules` section) for any future ontology-authoring session that wants to use this — not scoped to this session's specific gaps. Four real trials used the simpler hand-rolled version of the same idea (`opda:EPCCertificate`, `opda:NearbyFacility`, `opda:Proprietorship`, `opda:Survey`) — scripts in `source/03-standards/rml/harness/darwin-poc/`.

## Key lessons (worth internalising before touching this area again)

1. **A "confirmed gap" or "intractable" finding is a claim, not a fact — re-verify it before trusting it, especially your own prior session's characterization.** Four classes (`TransactionChain`, `Milestone`, `Claim`/`Evidence`/`VerificationActivity`) were wrongly written off earlier in this session as having no real JSON basis. They did — real, rich data existed; the classes had just been minted through ODR council deliberation rather than mechanical JSON extraction, and that citation-provenance distinction got conflated with "no data exists." Re-reading the actual JSON schema directly settled it in minutes. Same pattern hit `Address` (a "held-as-live dissent" in its ratifying ODR turned out, on actually reading the ODR's own text, to explicitly *not* block implementation) and `schoolType` (a "RML can't express this" claim that was simply wrong once re-tested against the current engine).
2. **"Ratified in an ODR but never actually implemented" is a real, recurring bug class in this codebase**, distinct from "genuinely unmodelled." Found and fixed 6+ instances this session (`leaseTerm`, `dependsOnTransaction`, `Address`'s 6 properties, `Verifier`, `hasSteward`, `variantPredicate`/`variantValue`, `lawfulBasis`'s D1/D2/D3 compound defect). The tell: a class/property's own `rdfs:comment` names a specific term that a `grep` against the merged ontology shows was never declared.
3. **When re-testing an "impossible" finding after an engine migration, get a genuinely new angle, don't just re-run the old technique.** The `Proprietorship` fix worked because RMLMapper has a different (bracket-index null-skip) behaviour than morph-kgc had — re-attempting the *same* technique on a new engine would have failed the same way.
4. **External-ontology citations (OBO, DPV, etc.) must be verified via a real lookup, never asserted from memory** — this session's own IAO crosswalk work did this correctly (WebFetch + OLS4 cross-check) after being explicitly warned to.
5. **A background agent's "done" report is a claim, not a fact.** This session independently re-verified essentially every dispatched agent's work (re-running the actual test/CI commands, checking real materialized triples, cross-checking cited SHACL shapes exist) before accepting it — and caught real issues doing so (a missing `rdfs:isDefinedBy`, a mis-ranged property, at least one agent's "done" report that undercounted its own progress).

## Known issues / gotchas

1. **63 commits on `main`, not pushed to `origin/main`.** This repo's own convention is work-directly-on-main (no feature branches), but pushing is a separate, explicit action this session correctly held back on per standing instructions. **Whoever picks this up should push (or ask the user to) before doing anything else** — a huge amount of uncommitted-to-remote work is sitting here.
2. **`agentdb.rvf`/`agentdb.rvf.lock` at repo root are untracked** — machine-local ruflo/AgentDB runtime state, never meant to be committed (matches this repo's own memory note on the subject).
3. **`.rmlmapper/` (the self-provisioned RMLMapper JAR cache) and `.metaharness/` (if present) are gitignored, local-only** — regenerate on demand, no action needed.
4. **`public/ontology/artefacts/opda-merged.ttl` and `opda-shapes-merged.ttl` are gitignored, local-only, and must be regenerated after ANY ontology-generator change** before running the RML harness — they're the Jena-merged form the SHACL validator actually loads. The exact `riot` invocation is in this session's own commits' history (search for `RIOT=.jena/apache-jena-6.1.0/bin/riot`); there's no dedicated script yet, which is a real, minor gap someone could fix (a `make regen-merged` target would remove a manual step every session seems to rediscover).
5. **`make ci-ontology` occasionally fails with a transient `ModuleNotFoundError: No module named 'opda_gen'`** despite a correct `PATH="$PWD/tools/opda-gen/.venv/bin:$PATH"` prefix — a PATH-resolution flake (mise shim interaction), not a real bug. Retry once with an absolute path prefix if it happens.
6. **`harness/darwin-poc/*.mjs` scripts are non-idempotent** — they APPEND candidate `TriplesMap`s onto the mapping as it stood *before* the winning genome was folded in. Re-running them now will not reproduce their original results (each script's own comment says this).

## What's left (20 gap items, all genuinely characterized — no more easy wins)

None of these are "should be easy, just hasn't been done" — every one was investigated this session and needs either new source data that doesn't exist, a real policy/design decision, or is architecturally out of RML's reach by the ontology's own doctrine:

- **`AssuranceLevel`** — needs a real policy decision (the only real field, pass/fail/consider, measures verification *outcome*; the property needs verification-*method-rigor*, a different axis — mapping one to the other would fabricate a compliance-consequential judgement).
- **`NameChangeEvent`/`UPRNSuccessionEvent`/`LeaseExtensionEvent`** — confirmed zero JSON basis; cross-instance/historical concepts a single transaction snapshot structurally cannot carry (they're populated by SHACL-AF materialization across a corpus, not RML).
- **`founds`/`playedBy`/`plays`/`roleNotation`** — UFO Role/Relator pattern connectives, design-time-only by this ontology's own doctrine, never meant to hold PDTF instance data.
- **`Proprietorship`'s `numberOfSellers`/`numberOfNonUkResidentSellers`/`hasRegisteredTitle`** — a separate, real gap even after `mediates` itself was closed this session.
- **`digest`/`identifiesSameProperty`** — no source data anywhere.
- **`potentialCost`** — confirmed ratified, permanent free-text/`MonetaryAmount` tension (Council session-028 Q3) — a deliberate, accepted design cost, not a bug.
- **`attestedBy`/`evidenceType`/`hasEvidencedAuthority`** — the real value-space lives in a separate PDTF schema envelope (`verifiedClaims/`) never `$ref`'d from the transaction schema — out of ODR-0035's single-instance mapping scope by construction.
- **`inspireFeatureId`** — the property is real and correctly declared; no tracked fixture happens to exercise INSPIRE-variant address data.
- **`Verifier`** — confirmed genuine data-absence (checked directly against both schemas): the real verifying-agent field lives in the same out-of-scope `verifiedClaims/` envelope as `evidenceType`.

If someone wants to push further, the highest-leverage next moves are probably: (a) a real EC/Council decision on `AssuranceLevel`'s policy mapping (this is a genuine "someone needs to decide" item, not an engineering one), and (b) whether `TrustFramework`/`chainMembers`-style derived-constant patterns are worth applying more broadly now that this session established the precedent.
