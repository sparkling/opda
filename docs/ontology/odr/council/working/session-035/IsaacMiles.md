## OPENING

### Frame: the proposition bundles three distinct claims. Two have merit; one is overreach. SKOS adjudicates all three.

I separate them deliberately, because a single FOR/AGAINST ballot on the whole proposition would smear a correct diagnosis (the alias) against an incorrect remedy (delete the OWL subclasses).

1. **The `owl:equivalentClass` short-name alias is a SKOS misuse** — a controlled-vocabulary labelling problem solved with a TBox-identity axiom. **I AFFIRM the anti-pattern claim, on SKOS grounds the brief did not anticipate.**
2. **Evidence-kind IS a controlled vocabulary** and *already half-exists* as a `skos:ConceptScheme` (`opda:EvidenceMethodScheme`). **I AFFIRM, with a precise mechanism.**
3. **"Rigid subclass inheritance is wrong; retire the three OWL subclasses in favour of a facet"** — this is the overreach. SKOS §3.5.1 does **not** force a choice between a concept scheme and OWL classes; it explicitly permits **both over the same resources**. **I REVISE: keep the subclasses, add the facet property, bind the two with `skos:exactMatch`. Facet-OR-inheritance is a false dichotomy.**

### Q1 — Is the alias an anti-pattern to retire?

**The artefact.** `opda-claim.ttl` declares three aliases (ADR-0011 "within-engineering option (b)"):

```
opda:Document            owl:equivalentClass opda:DocumentEvidence .
opda:ElectronicRecord    owl:equivalentClass opda:ElectronicRecordEvidence .
opda:Vouch               owl:equivalentClass opda:VouchEvidence .
```

The stated *purpose* (rdfs:comment, verbatim): "Alias … retained for **exemplar compatibility** (the diagnostic exemplar set uses the short name). owl:equivalentClass binding ensures one OWL identity; downstream shapes + annotations target the long name." So the entire job this axiom does is: *let an instance say `a opda:Document` and still count as a `DocumentEvidence`.* That is **a labelling/synonymy requirement** — "Document" and "Document Evidence" are two **labels for one concept** — dressed up as a class-identity assertion.

**Why this is a SKOS misuse — the load-bearing citation.** The SKOS Reference §5.3 defines exactly the construct for "same concept, alternative lexical form": **`skos:altLabel`**. S13 makes `skos:prefLabel`/`skos:altLabel`/`skos:hiddenLabel` *pairwise disjoint*; S14 caps `skos:prefLabel` at one per language. The intended pattern for "DocumentEvidence is the canonical name, Document is an accepted shorter synonym" is:

```
opda:evidenceMethod/Document
    skos:prefLabel "Document Evidence"@en ;   # canonical (downstream targets this)
    skos:altLabel  "Document"@en .            # exemplar-friendly short form
```

`owl:equivalentClass` is the **wrong tool**: it is a logical TBox axiom asserting two classes share *every extension in every model* — a statement about set identity, not about names. Using it to paper over a naming convenience is precisely the "SKOS misuse where `skos:altLabel` belongs" my lens is charged to flag. **I flag it.**

**The strongest count against me — and why it still loses.** The defender will say: ADR-0011's validation (commit `3d941c4`) *independently affirmed* the alias as "within-engineering," and ODR-0026 §R3 *confirms it is harmless* because under the OWL-RL-safe closure `owl:equivalentClass` **is not evaluated** (it is "documentation, not an entailment bridge"). True. But this is the **fatal tell**, not a defence:

- ODR-0026 §R3 itself records the consequence — "**an instance typed only with the short name is not inferred to be the canonical class.**" So the axiom **does not even do the one job it was minted for.** An exemplar that says `a opda:Document` is, under opda's own ratified entailment regime (ODR-0025 §R1/R2), **not** a `DocumentEvidence`. The shapes target the long name; the entailment that would bridge them is switched off. The alias is *inert*. A `skos:altLabel` is also inert at the logical level — but it is **honest**: it never claimed to bridge types, and it makes the short string dereference to the same concept, which is all anyone wanted.
- An *inert `owl:equivalentClass`* is worse than a no-op: it is a **latent landmine**. The moment anyone points an OWL-DL reasoner at the class graph (ODR-0026 §R2 explicitly contemplates "external OWL-DL tooling… may evaluate them independently"), the three equivalences fire bidirectionally and transitively, collapsing `opda:Document ≡ opda:DocumentEvidence` into the reasoner's class-identity machinery — the exact "uncontrolled bidirectional propagation + transitive explosion" ODR-0025 §R2 lists as the *reason* `equivalentClass` is excluded. opda is carrying, in its canonical TBox, three axioms it has declared unsafe to evaluate, to achieve an effect (short-name dereference) that SKOS achieves safely and natively.

**Verdict Q1.** The alias is an anti-pattern: a labelling need mis-encoded as a logical-identity axiom that (a) doesn't deliver the typing bridge it was minted for under opda's own regime, and (b) is a dormant unsafe-construct under any DL consumer. Retire all three `owl:equivalentClass` aliases; re-express "Document/ElectronicRecord/Vouch" as `skos:altLabel` on the corresponding concepts, with `…Evidence` as `skos:prefLabel`. Migrate the diagnostic exemplars to type against the canonical class (`a opda:DocumentEvidence`); the short string survives as a dereferenceable label, which is all "exemplar compatibility" ever required.

> `Q1: AFFIRM — ballot: FOR — The short/long alias is a controlled-vocabulary synonymy requirement ("Document" is an alternate label for "Document Evidence") encoded as owl:equivalentClass. SKOS Reference §5.3 (S13 pref/alt/hidden disjoint; S14 one prefLabel/lang) supplies the correct construct: skos:altLabel. The axiom is doubly defective — inert under opda's ratified OWL-RL-safe regime (ODR-0026 §R3: short-name instances are NOT inferred to the canonical class, so it fails its stated "one OWL identity" purpose) yet a latent transitive-explosion hazard under any external DL reasoner (the very risk ODR-0025 §R2 cites to exclude equivalentClass). Retire the three aliases → skos:altLabel; canonical name stays skos:prefLabel; re-type exemplars.`

### Q2 — Does evidence-kind belong in a SKOS concept scheme (vs OWL subclasses), and how does it relate to the existing `EvidenceMethodScheme`?

**The decisive repo fact the proposition missed.** The evidence-kind facet does not "half-exist" as a scheme — it **fully exists**. `opda-vocabularies.ttl` already carries `opda:EvidenceMethodScheme` ("the method by which identity evidence was obtained, per the OIDC4IDA `evidence` taxonomy") with **exactly three member concepts, member-for-member identical to the three OWL subclasses**:

```
#evidenceMethod/Document          skos:prefLabel "Document"@en ;
    skos:definition "OIDC4IDA Document evidence…" ; skos:inScheme opda:EvidenceMethodScheme .
#evidenceMethod/Electronic-Record skos:prefLabel "Electronic-Record"@en ;
    skos:definition "OIDC4IDA ElectronicRecord evidence…" ; skos:inScheme opda:EvidenceMethodScheme .
#evidenceMethod/Vouch             skos:prefLabel "Vouch"@en ;
    skos:definition "OIDC4IDA Vouch evidence…" ; skos:inScheme opda:EvidenceMethodScheme .
```

So opda **already models evidence-kind twice**: once as a SKOS concept scheme (the OIDC4IDA `evidence.type` value-space, correct per ODR-0011), and once as an `rdfs:subClassOf` tree under `opda:Evidence` (ODR-0009 R5). The proposition's framing — "*should* evidence-kind be a facet *instead of* subclasses" — is already overtaken by the artefact: the facet vocabulary is emitted and conformant.

**Does evidence-kind belong in a scheme? Yes — ODR-0011 already settled the general case, and this instance is textbook.** ODR-0011's whole thesis (§Decision, citing SKOS Reference §3): every closed JSON `enum` is a controlled vocabulary → a `skos:ConceptScheme`, *not* an `owl:oneOf` bag and *not* a subclass tree. The OIDC4IDA `evidence.type` discriminator (`document`/`electronic_record`/`vouch`) is **exactly** such an enum — three human-curated, regulator-defined, dereferenceable concepts with prefLabels and definitions sourced verbatim from the OpenID spec (ODR-0011 §4a). It is a **Quality Value / Method axis**, not a sortal hierarchy. SKOS is purpose-built for it. **Affirmed.**

**But "belongs in a scheme" does NOT entail "must not also be OWL subclasses." This is the proposition's category error, and SKOS §3.5.1 is dispositive.** The SKOS Reference §3.5.1 ("SKOS Concepts, OWL Classes and OWL Properties") states plainly: *"This specification does not make any additional statement about the formal relationship between the class of SKOS concepts and the class of OWL classes,"* and gives worked examples (Ex. 3, Ex. 4) declaring it **consistent for one resource to be both a `skos:Concept` and an `owl:Class`.** SKOS does not say "use a concept scheme *instead of* classes." It says the two layers are orthogonal and may coexist over the same subject matter. So the framing "facet vs. inheritance" sets up a dichotomy that the very standard the proposition invokes refuses to endorse.

**How they relate — the mechanism, which is also already-ratified.** ODR-0011 §8a defines the **"Substance Kind label" cross-scheme consistency check**: where a scheme's members correspond to an OWL sub-class hierarchy, *each member MUST carry `skos:exactMatch` to its OWL class — NEVER `owl:sameAs`* (ODR-0005 Anti-pattern §5). This is exactly the bridge evidence-kind needs, and it is the construct the alias should have used in the first place:

```
opda:evidenceMethod/Document        skos:exactMatch opda:DocumentEvidence .
opda:evidenceMethod/Electronic-Record skos:exactMatch opda:ElectronicRecordEvidence .
opda:evidenceMethod/Vouch           skos:exactMatch opda:VouchEvidence .
```

`skos:exactMatch` (SKOS Reference §10; S44 symmetric, S45 transitive, S46 disjoint-with-broadMatch) is a **mapping-layer** assertion — "these denote the same thing for retrieval/interoperability" — *without* the model-theoretic class-identity force of `owl:equivalentClass` or `owl:sameAs`. It is the standard, ODR-sanctioned way to connect a concept-scheme member to an OWL class. (Strictly, SKOS §10.6.1 notes `exactMatch` is *intended* for cross-scheme mapping; the internal scheme↔class binding is the established **opda house extension** of it — `TenureKindScheme`, `OwnerTypeScheme`, `PropertyTypeScheme` all already do precisely this internally per ODR-0011 §8a. I endorse continuing that house convention here; it is the project's settled answer to "concept ↔ its OWL class.")

**Relation to `EvidenceMethodScheme` specifically.** No new scheme is needed — `opda:EvidenceMethodScheme` **is** the evidence-kind facet vocabulary. The two open items are: (1) does it stay open-ended or become closed (`sh:in`)? OIDC4IDA's `evidence.type` is a **closed** three-value enum in the spec → it should drive a `sh:in` over its three concepts per ODR-0011 §Rules (closed-scheme discipline) and ODR-0009's `sh:xone`-on-evidence.type dispatch. (2) the three `skos:exactMatch` links to the OWL subclasses are **not yet emitted** — that is the concrete gap this session should direct. I defer the property-shape mechanics (does the facet ride a datatype `opda:evidenceMethod` property `sh:in` the scheme, or the `skos:exactMatch`-to-class binding, or both) to **Cagle's SHACL ruling** — see cross-talk below.

> `Q2: AFFIRM — ballot: FOR — Evidence-kind is a controlled vocabulary (OIDC4IDA evidence.type) and ALREADY exists as opda:EvidenceMethodScheme with the correct three concepts (ODR-0011 §Decision/§4a). It belongs in a scheme. BUT SKOS Reference §3.5.1 explicitly permits a resource to be both a skos:Concept and an owl:Class and makes "no statement" forcing one over the other — so "scheme vs subclasses" is a false dichotomy. The scheme relates to the OWL subclasses via skos:exactMatch (SKOS §10; ODR-0011 §8a Substance-Kind-label check — NEVER owl:sameAs/equivalentClass), the bridge the alias should have used. Action: emit the three missing scheme→class skos:exactMatch links; close the scheme with sh:in over its three concepts (OIDC4IDA enum is closed). No new scheme needed.`

### Q3 — Disposition / migration + ODR-0025/0026 + ADR-0011

**On the proposition's third claim ("rigid subclass inheritance is wrong; evidence typing should be a facet, not a hierarchy").** I **REJECT** this as stated, and the alternative I offer is *keep both layers, each doing its proper job*:

- **The OWL subclasses earn their place — ODR-0009 R5 is right, and it is not "rigid."** `opda:DocumentEvidence ⊑ opda:Evidence` (and the ElectronicRecord/Vouch siblings) exist because each subtype **carries type-specific facets and SHACL obligations**: ODR-0009 R5 — "Each subtype carries type-specific facets; SHACL `sh:xone` dispatches on subtype" (e.g. `electronic_record` → `sh:minCount 1` on `record.source.name`; `VouchEvidence` → `opda:attestedBy` an Agent). `opda:DocumentEvidence` additionally sits under `opda:AttachedDocument` (ODR-0024 R7) so that the neutral document Kind and the evidence-role are distinguished. **These are structural, shape-bearing distinctions** — they are not labels. SKOS concepts cannot carry `rdfs:domain`/property-shape obligations; OWL classes are the right home for "this kind of evidence must have these properties." Deleting the subclasses to leave only a facet string would **destroy the `sh:xone` dispatch** that is the entire validation contribution of ODR-0009.
- **The facet (scheme) and the subclasses are not redundant — they answer different questions.** The scheme answers *"what is the controlled, dereferenceable, regulator-cited value-space of evidence methods, reusable as a `sh:in` and a form picker?"* (a vocabulary/interoperability question). The subclasses answer *"what properties and constraints does an instance of this evidence kind bear?"* (a structural/validation question). SKOS §3.5.1 is the warrant for running both. The `skos:exactMatch` links keep them in lockstep so they cannot drift.
- **"Facets-vs-inheritance was never weighed" — partly fair, and worth recording.** The brief is correct that no ODR explicitly stages a *facet-vs-subclass bake-off* for evidence. ODR-0009 chose the PROV-O subclass backbone (S009); ODR-0011 minted the scheme independently; ADR-0011's validation waved the alias through as "within-engineering" (commit `3d941c4`) **without** noticing it had thereby created a *third, redundant, unsafe* encoding of the same axis on top of the scheme and the subclasses. That blind spot is real. But the remedy is not to demolish a layer — it is to **name the layering explicitly** (scheme = vocabulary; subclasses = structure; `skos:exactMatch` = the bridge; `skos:altLabel` = the short names) and retire the one encoding that is both redundant and unsafe (the alias).

**Empirical confirmation (verified in the repo, not asserted) — INCLUDING a correction to my own earlier reasoning (credit: Baker).** Facts from the artefact:
- *The subclasses bear subtype-specific DATA the facet cannot — but the SHACL that would govern it is NOT yet emitted.* The document exemplar (`exemplars/claim-with-document-evidence.ttl:44`) types its evidence `a opda:Document , prov:Entity` and carries `opda:documentType`, `opda:issuerAuthority`, `opda:documentReference`, `opda:issuedOn` — the document-evidence-specific facets. **CORRECTION:** in my opening I claimed the subclasses' structure "already exists" via the ODR-0009 `sh:xone`-on-`evidence.type` dispatch (electronic_record→record.source.name; vouch→attestedBy). **Baker checked `opda-claim-shapes.ttl` and that dispatch is NOT emitted** — I verified: the file (ADR-0012 emission) carries only `ClaimIdentityKeyShape`, `EvidenceIdentityKeyShape` (a lone `opda:digest sh:maxCount 1` on `sh:targetClass opda:Evidence`), and two SHACL-AF Info rules. **Zero `sh:xone`, zero per-subtype shape, no `record.source.name` constraint.** So as emitted, the three subclasses are presently structure-thin; the `sh:xone` is a *ratified-but-unemitted* ODR-0009 obligation, not a live artefact. My conclusion (keep all three) stands, but the **correct reason** is (a) the ODR-0009 R5 per-subtype obligations are ratified and remain on the emit list (the subclasses are their designated home), and (b) the §8a Substance-Kind-label consistency check forbids a facet-only member (it would split one scheme's members into concept+subclass vs concept-only). NOT "the structure already exists."
- *This is itself a session finding — a documentation OVERCLAIM of the same defect class as the alias.* `opda:Evidence`'s `rdfs:comment` (`opda-claim.ttl:83`) advertises "SHACL `sh:xone` dispatches on subtype at validation time (ADR-0012 emits the shape)" — **but ADR-0012 did not emit it.** A comment asserting validation the artefact never performs is exactly the alias's pathology (an axiom/annotation claiming semantics the regime doesn't deliver). Baker flagged this to Kendall for the remediation pass; I concur it belongs in the session record as a second instance of the through-line.
- *The short names appear in exactly three places* beyond the alias definitions — the three diagnostic exemplars (`a opda:Document`/`opda:ElectronicRecord`/`opda:Vouch`). **No shape file references them** (verified); shapes target `opda:Evidence`/the `…Evidence` long names. So the consumer-side blast radius of retiring the alias is precisely *three exemplar re-typings, zero shape edits.* (Confirms Cagle Q2 pre-emptively.)
- *The facet vocabulary is currently unexercised.* The exemplars carry evidence-method as bare strings (`opda:verificationMethod "document-inspection"`, `opda:documentType "grant-of-probate"`) and the kind via the OWL type — **none reference an `#evidenceMethod/*` concept.** So the scheme today is a conformant-but-dormant parallel encoding; this session is what wires it in (the `skos:exactMatch` links + the disposition below).

**Caveat on the §8a "forbids facet-only" reason (Baker's sharpening, which I accept):** §8a is not an absolute SKOS prohibition on label-only sub-Kinds — SKOS Reference §3.5.1 happily permits concepts with no OWL class. What makes a facet-only member wrong *here* is (i) prospective `sh:xone` dispatch the subclasses are the home for, and (ii) near-zero cost to keep all three uniform. The panel should not over-read §8a as "SKOS forbids facet-only members" — it is a *local* uniformity argument, not a universal law.

The Council (modelling altitude) now rules the alias a vocabulary-layer concern mis-encoded at the logic layer. This is a clean engineering→Council escalation, not a re-litigation of within-engineering latitude.

### From Baker exchange + OIDC4IDA source verification — two refinements to Q2 (both strengthen my position):

**(i) The existing scheme is MISLABELLED — rename, do not mint a second.** I fetched the OIDC4IDA spec. It layers **three orthogonal axes**: `type` (the kind: document / electronic_record / vouch / electronic_signature — top-level classifier); `method`/`verification_method` (*how* obtained — pipp / sripp); `check_details`/`validation_method` (*what checks* — vpiruv / bvr). `opda:EvidenceMethodScheme`'s three members are `Document`/`Electronic-Record`/`Vouch` — those are **`evidence.type` values, not method values** (method would be pipp/sripp). So the scheme is the **kind** scheme wearing a "Method" name (its scopeNote even says "method by which evidence was obtained" — wrong axis). **Disposition refinement:** *rename/repurpose* `EvidenceMethodScheme` → the evidence-**kind** scheme (prefLabel "Evidence Kind"; fix scopeNote to the `type` axis); do **NOT** mint a parallel `EvidenceKindScheme` (Baker's instinct) — that would put `Document`-the-concept `skos:inScheme` two schemes, violating the ODR-0011 §1a `ConceptInExactlyOnePrimaryScheme` IC (SKOS Reference §4.6 `skos:inScheme`; S9 `ConceptScheme` disjoint-with `Concept`). A *genuine* future `EvidenceMethodScheme` (pipp/sripp) + `ValidationCheckScheme` (vpiruv/bvr) may be minted **iff** a method/check leaf appears in PDTF data (generator-first). **One scheme per axis; the kind axis already exists, just needs its name/category corrected.**

**(ii) ufoCategory = Substance-Kind label, NOT Quality Value.** The kind axis partitions Evidence into sub-Kinds that ARE OWL subclasses (Document/ElectronicRecord/Vouch Evidence). That is sortal sub-Kinding → the ODR-0011 §8a **"Substance Kind label"** row, which is the row that *carries the `skos:exactMatch`-to-OWL-subclass mandate* (TenureKind/OwnerType template). Quality Value (addressVariant/S015) is for a Quality particularising a Kind *without* a subclass partition — wrong fit here. Tagging it Substance-Kind label **fires the §8a cross-scheme consistency check**, which MANDATES the three `skos:exactMatch` links — exactly the binding the alias failed to deliver. (Note: the substrate currently tags the scheme "Quality Value"; this is a *correction*, not an inheritance.) Snag I own: `ElectronicRecordEvidence` is property-thin. It stays a subclass anyway, because (a) it is the ratified HOME for the as-yet-**unemitted** ODR-0009 R5 obligation (record.source.name) + the eIDAS-Substantial anchor, and (b) per §8a uniformity all three kind concepts should exactMatch a subclass — a facet-only carve-out would split one scheme into concept+subclass vs concept-only members. **(Caveat, per Baker: §8a is a *local* uniformity argument here, not an absolute SKOS prohibition — SKOS §3.5.1 permits label-only concepts. What makes facet-only wrong here is prospective dispatch + near-zero cost to keep all three uniform.)**
