## Q1 — Adopt hm ODR-0014 repeated-`rdfs:domain` "any-of": sound for OPDA? what documentation makes it legible not a shibboleth? SHACL `sh:or` dual unaffected?

**Verdict: AFFIRM the adoption (per directing authority), CONDITIONAL on a mandatory, CI-gated documentation discipline. Ballot: FOR (a) repeated-`rdfs:domain` — under conditions.**

I accept (a) as the decided form. But I will not let the panel record it as if it were semantically innocent, because it is not, and the danger is exactly the one that produced the session-047 error in the first place.

**The semantic fact (the shibboleth named).** Multiple `rdfs:domain` triples on one property are **conjunction**, not union, under standard semantics. RDF Schema 1.1 §3.2 (rdfs:domain) is explicit — each `rdfs:domain` assertion independently licenses rule `rdfs2` (`(?p rdfs:domain ?c), (?x ?p ?y) ⊨ (?x rdf:type ?c)`; RDF 1.1 Semantics §9.2.1) — so `playedBy rdfs:domain opda:Role ; rdfs:domain opda:RoleMixin` *means* "every player is **both** a Role **and** a RoleMixin." For sibling meta-classes that is the **negation** of the disjunctive intent, and if Q3 makes them disjoint, it is **unsatisfiable** (`owl:Nothing`). *Semantic Web for the Working Ontologist*, 3rd ed., Ch. 7 (Allemang, Hendler, Gandon) teaches precisely this additive/intersective reading as the canonical trap. hm ODR-0014 itself concedes the point in its own §Bad: "Technically misleading to OWL purists (multiple `rdfs:domain` looks like intersection)" — hm is *aware* it is publishing a convention that contradicts the standard reading, and accepts it. So my brief's framing is exactly right: hm's "any-of" is **the standards conjunction dressed as a convention.** It is a shibboleth — a token that means "we are hm and we read our comma-lists our own way," legible to the in-group and a landmine to everyone else.

**Why it is nonetheless SOUND for OPDA (the only reason, and it is load-bearing).** An axiom that is never fed to the rule that would draw the unwanted conclusion draws no unwanted conclusion. OPDA's closure is frozen to exactly 7 rules (ODR-0025 §R1); `rdfs:domain`/`range` type-entailment (rules `rdfs2`/`rdfs3`) is **excluded** (ODR-0025 §R2, named in the R2 table); and ADR-0035 ships a consistency test proving the closure adds **zero** domain/range-derived triples (the EPC case, ODR-0025 §R7, is the worked proof the trap does not fire). This is the session-048 correction I made on the record: my own session-047 "everything-becomes-a-Person" example silently assumed `rdfs2`/`rdfs3` were live; **they are provably not.** `model-but-don't-evaluate` (ODR-0026 §R2) is not a hopeful convention — it is the precise statement that OPDA evaluates only the R1 closure, and that closure is sound-but-RL-incomplete (ODR-0029 §R5). So for *OPDA's loader*, repeated-`rdfs:domain` is inert and therefore harmless — the conjunction never fires. **AFFIRM on soundness.**

**But soundness ≠ legibility, and this is where I press as DA.** The session-048 test cuts BOTH ways. It rescued `owl:unionOf` (correct under the external reader) — but it *indicts* repeated-`rdfs:domain`, because the very external/AI reader ADR-0049 courts as a stated driver will read the comma-list as **conjunction** and conclude the property is contradictory or unsatisfiable. A *competent* LLM that has internalised RDFS §3.2 (precisely the AI-signal audience the ADR wants) gets it **wrong** in the dangerous direction; only a *weak* pattern-matcher reads the comma as "or" and gets lucky. So the directing authority's choice optimises for **cross-project corpus consistency with hm** (real, legitimate value — one convention, one meta-shape, boolean-constructor-free) at the cost of **external-DL fidelity** (which my session-048 ballot weighted higher). That trade is the authority's to make and it has made it. My DA duty is to make the chosen option *carry its own cost honestly*: if OPDA publishes the shibboleth, the convention MUST be impossible to misread by accident, which means the documentation discipline is **not optional decoration — it is the load-bearing mitigation that converts an incorrect-by-standards axiom into a legible one.**

**The documentation discipline that makes "any-of" legible rather than a shibboleth (this is my real Q1 deliverable as DA — I am hardening the adoption, not blessing it):**

1. **Module-header `skos:editorialNote` — MANDATORY and CI-gated.** Every `owl:Ontology` header in a file that carries any multi-domain object property MUST declare the convention verbatim, e.g.: `"OWL documentary mode (ODR-0026 §R2 model-but-don't-evaluate; ADR-0049). Multiple rdfs:domain on one property is read as domainIncludes 'any-of' (the intended subject is a member of AT LEAST ONE listed class), NOT the standard RDFS §3.2 conjunction. OPDA never evaluates domain/range (closure frozen to 7 rules, ODR-0025 §R1); the intersection entailment never fires. SHACL sh:or carries 100% enforcement (ODR-0029 §R3)."@en`. hm requires this header (ODR-0014 §Bad: "Requires header comments in every ontology file") — OPDA must too, and must **gate it**: a new `ci-domainIncludes-header` check fails any class-graph file with a multi-domain object property whose owl:Ontology header lacks the note. *Without the gate, the convention is a verbal agreement that the next modeller (or the session-047 reviewer) never sees — exactly how the original error happened.*

2. **Per-property `rdfs:comment` — MANDATORY on every multi-domain object property.** Each MUST spell out the disjunction explicitly: e.g. `playedBy` → "any-of subject typing: opda:Role OR opda:RoleMixin (NOT both — domainIncludes convention, ODR-0026 §R2); bearer disjunction Person∪Organisation enforced in opda:RolePlayShape sh:or; NOT a conjunction; never reasoned." hm ODR-0014 mandates exactly this for multi-use properties (Rules: "Multi-use properties carry `rdfs:comment` explaining the per-class usage pattern"; §Confirmation). This is the per-axiom antidote to the comma-conjunction misread.

3. **Reverse the now-wrong inline comments.** The current corpus carries comments that are now *false* under the adopted doctrine: `opda:plays` (`opda-agent.ttl:247`) says asserting `rdfs:domain` "would re-introduce the union-entailment anti-pattern"; `opda:playedBy` (`:239`) attributes the "would entail every bearer is a Person" warning to me. Those were correct under session-047's mistaken premise and are **wrong** under ADR-0049's adopted doctrine. They MUST be reversed to the domainIncludes-convention wording, or they will re-teach the very error the ADR exists to correct. (This is outstanding-task #1 in the ADR; I am confirming it is doctrinally required, not optional.)

4. **The meta-shape (Q-doctrine) must POSITIVELY permit multi-domain and still EXCLUDE the boolean constructors.** Because OPDA adopts (a) not (b), the excluded-construct meta-shape (hm ODR-0030) must continue to fail `owl:unionOf`/`intersectionOf`/`complementOf`/`owl:Restriction`/cardinality/`oneOf`/`hasKey` — repeated-`rdfs:domain` introduces **no** blank-node class expression, which is the *one* genuine engineering advantage of the directing authority's choice over my session-048 (b). I credit it: (a) keeps the corpus boolean-constructor-free, so the meta-shape stays a pure blacklist with no "admit unionOf only in domain-position" carve-out. That is cleaner to gate than (b) would have been. Recorded as the honest win for (a).

**SHACL `sh:or` dual — CONFIRMED unaffected, and it is doing ALL the actual work.** ODR-0029 §R3 ("`domain`/`range`-as-SHACL-constraint pattern") is untouched: every `rdfs:domain C` gets its `sh:targetSubjectsOf <pred> ; sh:class C` dual, and a genuine disjunction is enforced as `sh:targetSubjectsOf <pred> ; sh:or ([sh:class A][sh:class B])` (already the corpus pattern — `opda:RolePlayShape`, `opda:RolePlaySubjectShape`, `opda:HasAddressBearerShape`; verified `opda-agent.ttl` comments + `scripts/ontology-model.mjs` recovery). The closed-world `sh:or` is the **only** layer that actually rejects a mis-typed node; the documentary repeated-`rdfs:domain` rejects nothing (it can't — RDFS has no violation concept; RDF Semantics, open-world). So the honest framing for the record: under (a), **SHACL `sh:or` carries 100% of enforcement AND 100% of the correct disjunctive semantics; the repeated-`rdfs:domain` is pure documentary signal whose standard reading is wrong and which is salvaged only by the mandatory convention + the non-evaluation guarantee.** That is a coherent position — but only with limbs 1–3 mandatory and gated. Re-run the ADR-0035 zero-triple test after authoring (no rule consumes multiple-domain; it will pass — that passing is the formal proof the conjunction never fires).

**DA disposition: WITHDRAW my session-048 `owl:unionOf` advocacy** (the directing authority decided cross-project consistency over external-DL fidelity; that is a legitimate values call above my pay grade, and (a)'s boolean-constructor-freedom is a real countervailing good). **HOLD one principled, non-blocking dissent:** the documentation discipline (limbs 1–3) is **not optional** — if the panel or implementation treats the module-header note or per-property `rdfs:comment` as nice-to-have rather than CI-gated MUST, then OPDA has published a bare shibboleth and I revert to opposing the adoption. **Named re-open trigger:** an external/AI consumer (or a future OPDA modeller in a review) demonstrably reads a multi-domain object property as the RDFS §3.2 conjunction and either (i) concludes a property is unsatisfiable, or (ii) re-derives a session-047-style "strip the domain/range" correction — at which point the convention has failed its legibility burden and Q1 reopens to reconsider local-omission or `owl:unionOf` for the genuine-disjunction cases specifically. The trigger is concrete and falsifiable, which is what distinguishes a principled hold from foot-dragging.

## Q2 — THE LIVE TENSION: adopt hm ODR-0054 FP/IFP WHOLESALE, or CARVE OUT (SHACL-only)? — argue the carve-out hard.

**Verdict: REVISE the adoption — CARVE OUT FP/IFP; do NOT adopt hm ODR-0054 wholesale. Ballot: AGAINST authoring `owl:FunctionalProperty`/`owl:InverseFunctionalProperty` as axioms; FOR `sh:maxCount 1` / `dash:uniqueValueForClass` + annotation as the home.**

This is the question my brief tells me to "argue hard," and I will, because it is the one place where "adopt hm's coverage wholesale" would make OPDA **publish to the Web the one inference its constitution bans by name.** I reached AGAINST in session-048 (moved there by Gandon and Cagle from my opening FOR-scoped); I now hold it as DA with the carve-out argument made affirmatively rather than conceded. The carve-out is not a quibble — it is the difference between OPDA's published TBox being honest and it asserting a falsehood OPDA spent ODR-0005 establishing it does not hold.

**hm ODR-0054's actual claim, and why it does not transfer.** hm ODR-0054 R1/R2 authors `owl:FunctionalProperty` (~60–80 properties) and `owl:InverseFunctionalProperty` (~15–20: ISO codes, product numbers, currency codes) as "AI-readable domain documentation," on the rationale "OWL describes the domain; SHACL validates the data; declaring both is not duplication." That rationale is coherent **for hm**, whose ODR-0005-equivalent does not exist — hm has no constitutional ban on identity-merge. **OPDA does.** The transfer fails at exactly the axis hm never had to consider.

**Argument 1 — IFP's entire model-theoretic content IS the banned merge. There is no separable "identity-key" proposition above it.** `owl:InverseFunctionalProperty` means, by OWL 2 Direct Semantics §2.3.5 (and Primer §4.2, the canonical `foaf:mbox` key pattern): *for any two subjects sharing a value of the property, those subjects are `owl:sameAs`.* That is not a side-effect of IFP — it **is** IFP; the axiom has no other content. OPDA bans `owl:sameAs` materialisation outright and by name (ODR-0005 §R5, ODR-0017 §R6, and the ODR-0025 §R2 table lists FP/IFP precisely because they "Trigger `owl:sameAs` identity-merge"). So adopting IFP as a *published documentary axiom* publishes the merge rule to every external reader and AI consumer the ADR is courting. You cannot author IFP "for documentation" and disclaim the merge — the merge is the documentation. **To name it is to arm it** (Cagle's phrase, session-048; I now own it as DA).

**Argument 2 — the engineering-act ≠ ontological-act asymmetry. This is the decisive, OPDA-specific test.** Apply session-048's own test consistently — the same test that *permits* `owl:disjointWith` (Q3) and *permitted* `owl:unionOf` in session-048: does the construct's published W3C meaning AGREE with OPDA's enforced behaviour? 
  - **`rdfs:domain`/`range` (Q1):** external mis-inference is *benign-redundant typing* — at worst the reader adds an extra `rdf:type`. Tolerable. (This is why Q1 survives at all.)
  - **`owl:disjointWith` (Q3):** external reading is *detect-and-reject on erroneous data* — exactly what OPDA wants; the reader draws nothing bad on correct data. PASSES.
  - **FP/IFP:** external reading is *merge two individuals into one* — the single operation OPDA's identity doctrine forbids, on exactly the identifier spine (UPRN, title number, companies-house number) where you would be tempted to author it. **FAILS.** A wrong domain adds a wrong type; a wrong IFP **fuses distinct records.** The asymmetry is not rhetorical — it is the difference between additive (recoverable) and destructive (irreversible) external inference.

**Argument 3 — even a TRUE IFP asserts a proposition OPDA holds FALSE for its domain.** This is the argument that closes it, and it survives every "but it's inert in OPDA's loader" rejoinder. UPRN genuinely identifying a Property is a defensible open-world key claim *in the abstract*. But OPDA's master-data semantics **deliberately do not hold shared-key ⇒ same-individual** — ODR-0005 establishes bounded-context identity with no global `sameAs`. Two records sharing a UPRN through (i) a re-numbering, (ii) a data-entry error, or (iii) a deliberately maintained bounded-context boundary are **NOT the same individual** under OPDA's model — and a published `owl:InverseFunctionalProperty hasUPRN` tells the entire Web that they are. That is a **correctness failure in the published contract**, independent of whether OPDA's own loader ever evaluates it. hm can author IFP because hm holds shared-key ⇒ identity (or doesn't care); OPDA constitutionally holds the opposite. *Copying hm's axiom here imports a proposition OPDA's foundational identity ODR exists to deny.*

**Argument 4 — session-035 (ODR-0026 §R3, 8–0–0) is binding precedent and it points straight at the carve-out.** Session-035 retired the three `owl:equivalentClass` evidence aliases — not because they were unsound under the closure (they were inert, exactly like FP/IFP), but because a **safe, evaluable, native substitute carried the same signal** (`skos:altLabel` + `skos:exactMatch`), so `model-but-don't-evaluate` did **not** apply: *keep the information, drop the unsafe-and-redundant axiom.* FP/IFP are the identical case: 
  - The Functional signal ("single-valued") → `sh:maxCount 1` (already in the corpus, already fires in CI, closed-world).
  - The IFP signal ("this value identifies the subject") → `dash:uniqueValueForClass` — which *flags the collision* instead of *fusing the individuals*, carrying the identifying-key meaning **without** the merge. 
  The substitute is complete and safe; therefore session-035's rule mandates dropping the axiom. This is not "the everything-becomes-a-Person reflex" (that reflex is *wrong*, as I conceded for Q1) — it is the session-035 test passing for FP/IFP *specifically*, on the merge hazard + complete-safe-substitute grounds.

**The honest scope of the carve-out (so the record is precise, not a blanket reflex):**
  - **IFP: OUT ENTIRELY.** No exceptions. Its content is the merge; the merge is banned; the substitute is complete. (Effective Council A unanimity in session-048.)
  - **FP: no general documentary layer.** `sh:maxCount 1` is the whole single-valuedness signal. I can tolerate — but as DA I do **not** advocate — a hand-curated `owl:FunctionalProperty` opt-in on a tiny set of genuine single-valued-*by-domain-truth* world-facts (Allemang's session-048 carve-out), each individually justified in `rdfs:comment` with the ontological-vs-data-shape distinction marked, **provided** (i) it never becomes a sweep over every `sh:maxCount 1`, and (ii) the SHACL is never deleted "because OWL says the same." If the panel cannot guarantee (i)/(ii) by gate, FP goes out too. My ballot is AGAINST FP/IFP *as a documentary layer*; the home is SHACL + annotation.
  - **Separate bucket, flagged so no one conflates it (the session-047 class of error):** the closure *does* enable `owl:SymmetricProperty`/`TransitiveProperty` (R1 rules 6/7), so those, *if authored*, would actually fire. That is a dormant **safe capability**, NOT a mandate to author — OPDA types nothing Symmetric/Transitive today and should keep it that way absent a real such relation. "Enabled" ≠ "should-author." This is outside Q2's FP/IFP scope but I name it because conflating capability with mandate is precisely the error ADR-0049 exists to prevent, and a wholesale "adopt hm ODR-0054" would invite it.

**DA disposition: HOLD — this carve-out is BLOCKING.** I do not withdraw on Q2. Adopting hm ODR-0054 wholesale for FP/IFP would publish the `owl:sameAs` merge OPDA bans (Arg 1–2) and assert a proposition OPDA holds false (Arg 3), against binding session-035 precedent (Arg 4). The carve-out (SHACL-only for FP/IFP) is the condition of a sound adoption, not a preference. **Named re-open trigger (the ONLY thing that would move me):** a named, real consumer demonstrates it requires the *open-world identity-key entailment itself* (shared value ⇒ same individual) AND OPDA's bounded-context identity doctrine (ODR-0005) is first formally revised to permit global `sameAs` — i.e. the ban itself is lifted by a council. Absent that two-part trigger, FP/IFP stay out. Note this trigger requires overturning ODR-0005, so it is a genuine constitutional re-open, not a routine one — which is the correct bar for re-admitting a banned inference.

## Q3 — Adopt hm ODR-0030's permitted/excluded table + excluded-construct CI meta-shape, AND hm ODR-0028b disjointness (rigid Kinds only, never anti-rigid Roles, pairwise) as-is?

**Verdict: AFFIRM both, with two OPDA-specific corrections to "as-is." Ballot: FOR.**

These two adoptions are where "adopt hm's coverage" is genuinely *right* — but "as-is" is not quite right, and as DA I name the two deltas where OPDA's regime requires a deviation from hm's table.

**hm ODR-0030 permitted/excluded table + meta-shape — ADOPT, with TWO corrections.**

The construct-governance principle transfers cleanly: a permitted/excluded list governs which OWL constructs may appear, enforced by an excluded-construct SHACL-SPARQL meta-shape (CI-only). hm ODR-0030's excluded list (`owl:disjointUnionOf`, `owl:Restriction`, `owl:unionOf`/`intersectionOf`/`complementOf`, all cardinality, `owl:oneOf`, `owl:hasKey`) maps almost exactly onto ODR-0025 §R2's excluded set, and the meta-shape implementation (hm ODR-0030 §Meta-Shape) is directly reusable. **AFFIRM the mechanism.** But two corrections, because OPDA's regime is *not* hm's:

  - **Correction 1 — the PERMITTED column diverges, and the divergence is the whole point of OPDA.** hm ODR-0030 permits FP/IFP and `equivalentClass`/`equivalentProperty` as documentary-but-not-inferenced. **OPDA must move FP/IFP from PERMITTED to EXCLUDED** per Q2 (the merge ban). And `equivalentClass` is already retired by session-035 for the alias case and stays admissible-but-unevaluated only for true co-extension (ODR-0026 §R3) — narrower than hm's permission. So OPDA's table is **not a copy of hm's** — it is hm's *mechanism* applied to OPDA's *different boundary*. Recording it "as-is" would re-import the FP/IFP permission Q2 just carved out. This is the single most important place the panel must NOT cargo-cult hm.
  - **Correction 2 — OPDA's meta-shape must additionally FAIL FP/IFP** (which hm's permits), and must **permit multi-domain `rdfs:domain`** (Q1, which hm also permits — agreement here) while continuing to fail all boolean constructors. The meta-shape is therefore OPDA-authored from hm's template, not lifted verbatim. (Implementation note for the record: OPDA runs SHACL-1.2-via-Jena per ODR-0036, and the meta-shape is SHACL-SPARQL — confirm `jena-shacl` executes the `sh:sparql` constraints; hm's note that this is "CI only, not runtime" applies to OPDA identically.)

**hm ODR-0028b disjointness (rigid Kinds only / never anti-rigid Roles / pairwise) — ADOPT; this is the construct that PASSES the engineering-act = ontological-act test.** 

This is the one R2-set construct whose published standard semantics **agree** with OPDA's enforced behaviour, which is why it adopts cleanly where FP/IFP cannot. `owl:disjointWith`'s standard meaning (OWL 2 §2.3.2 `DisjointClasses`; Primer §5.3) is exactly the *inconsistency check* OPDA **already runs** — ODR-0025 §R1: "A disjointness consistency check is permitted but is a validation gate, not an entailment — it produces no domain triples." Authoring it asserts to the Web precisely what OPDA enforces: zero divergence between published meaning and behaviour.

The point I made in session-048 that defeats any "don't bother" objection: **a check needs an axiom to check against.** OPDA authors zero `owl:disjointWith` today (verified corpus count, session-048), so the safe gate is currently **vacuous** — an empty shape of exactly the kind session-047's verification pass caught. Authoring `Person owl:disjointWith Organisation` is what **arms** the existing safe gate; it produces no triples and no entailment under R1 (the `owl:Nothing` consequence *is* the inconsistency the gate reports — ADR-0035:132 is a local pairwise closed-world COUNT-query, not a global ex-falso surface; verified session-048). Smallest delta of all three questions.

**The scope guards, which I hold as NON-NEGOTIABLE and which are exactly hm ODR-0028b's own scoping rule (so "as-is" is correct *here*):**
  1. **Rigid/sortal Kinds only** — author only between classes carrying their own identity criterion (Person/Organisation; Relator Kinds). hm ODR-0028b's "NOT Disjoint (9-0)" analysis reaches the identical conclusion for country roles via Guarino's identity-continuity argument: anti-rigidity alone is insufficient; *shared identity principle* is the formal basis for non-disjointness.
  2. **NEVER between anti-rigid Roles sharing a bearer's identity (HARD carve-out).** `Seller`/`Buyer` as Roles are anti-rigid; one Person can play both across different transactions (Guarino & Welty 2002 OntoClean; Guizzardi 2005 §4.x on anti-rigidity). So it is **NOT** `Seller owl:disjointWith Buyer` at the Role level — that exclusivity is *transaction-scoped* and lives in SHACL on the qua-individuals (session-047 Q4 `playedBy` distinct-node discipline). Getting this wrong re-creates, in disjointness form, the exact role/relator confusion session-047 untangled. This matches hm ODR-0028b's "Anti-Pattern: Disjointness on Anti-Rigid Roles" verbatim — the one place hm and OPDA's OntoClean discipline are identical and both correct.
  3. **Pairwise `owl:disjointWith`, NOT `owl:AllDisjointClasses`.** hm ODR-0028b uses both; **OPDA must use pairwise only** — this is a third deviation from hm "as-is", because OPDA's closure lists `AllDisjointClasses` but **no rule consumes it** (verified), so `AllDisjointClasses` would re-introduce the engineering-act ≠ ontological-act gap (a published axiom OPDA's gate does not actually check). Pairwise `owl:disjointWith` is what the ADR-0035 pairwise COUNT-gate actually consumes. So on this sub-point OPDA is **stricter than hm**, and rightly.

**DA disposition: WITHDRAW.** I have no live dissent on Q3 — it is the construct where adopting hm is correct *and* where my own session-048 analysis already landed FOR-scoped. The carve-outs (rigid-Kinds-only, anti-rigid-Role exclusion, pairwise-not-AllDisjointClasses) are binding regardless of any other question's outcome — they were the unanimous (4–0) Council A product and the most important thing Q3 produced. **One recorded caveat, not a hold:** Cagle's session-048 cost-discipline note (a named union class serves no current OPDA consumer) survives only as a *non-blocking* preference and is now **moot** under the directing authority's choice of repeated-`rdfs:domain` over `owl:unionOf` — there is no union *class* to back, so Q3's `disjointWith` stands on its own as the partition statement, enforced by `sh:not`/`sh:xone`. The session-048 Q1(b)-coupling dissolves entirely. **Named re-open trigger:** a disjointness pair is proposed between two classes where at least one is anti-rigid (a Role/Phase/RoleMixin) — that is the bright line; any such proposal reopens Q3's scope, because it is the carve-out failing.

## Q4 — Consequences: relationship-layer object properties, the gate, ontology-model.mjs; confirm supersession of session-048 + fold into ADR-0049.

**Verdict: AFFIRM the consequences as the ADR-0049 §Outstanding-work list states them, with the per-construct dispositions above applied. Ballot: FOR.**

  - **Relationship-layer object properties** (`founds`/`mediates`/`playedBy`/`plays`/`hasParticipant`/`hasAddress`): under the adopted doctrine these regain **documentary** domain/range — **repeated-`rdfs:domain` "any-of" for the genuine disjunctions** (`playedBy`/`plays` bearer Role∪RoleMixin; `hasParticipant` co-domain Seller∪Buyer; `hasAddress` bearer Property∪Person∪Organisation), **plain `rdfs:domain`/`range` for the single-class single-bearer spine** (`founds`/`mediates` → their Relator/Role types; `concernsProperty`/`hasRegisteredTitle` already plain).
