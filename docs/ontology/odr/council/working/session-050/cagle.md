## Opening positions

### Q1 — Repeated-`rdfs:domain` "any-of" + SHACL `sh:or` carries 100% enforcement

**Verdict: AFFIRM. Ballot: FOR.**

The OWL `rdfs:domain`/`rdfs:range` is documentary; the SHACL `sh:or` is the enforcement. This is not theory in OPDA — it is **already the emitted state of the corpus**. Look at `source/03-standards/ontology/opda-agent-shapes.ttl` as built today:

- `FoundsRangeShape` — `sh:targetObjectsOf opda:founds ; sh:or ( [sh:class opda:Role] [sh:class opda:RoleMixin] ) ; sh:severity sh:Violation`.
- `PlaysRangeShape`, `RolePlaySubjectShape` — the same `sh:or` co-domain disjunction on the focus node, Violation severity.
- `RolePlayShape` / `SellerShape` / `BuyerShape` — `sh:property [ sh:path opda:playedBy ; sh:or ( [sh:class opda:Person] [sh:class opda:Organisation] ) ]`, Violation.

Every one of these carries **`sh:severity sh:Violation`**. Under the SHACL Core Recommendation (Knublauch & Kontokostas eds. 2017, §3 — Validation, and §4.6.1 `sh:or`), a focus node that fails an `sh:or` whose members are `sh:class` constraints produces a validation **result** at the declared severity; at `sh:Violation` the data graph does **not conform**. That is 100% enforcement of the co-domain disjunction — a non-conforming graph is rejected at the gate. The disjunction is enforced *exactly*, value-by-value, with no reliance on any reasoner.

By contrast `rdfs:domain` / `rdfs:range`, even read as "any-of", enforce **nothing**: RDF Schema 1.1 (§2.3.1 `rdfs:domain`, §2.3.2 `rdfs:range`) defines them as entailment rules — they *infer* a type onto the subject/object; they never *reject* a mis-typed node. And OPDA never even fires that entailment (ODR-0025 §R1 frozen 7-rule closure; ODR-0026 §R2 model-but-don't-evaluate). So the OWL axiom is pure documentation: a machine-legible, Protégé-renderable, LLM-readable signal of modeller intent that costs zero triples in the closure (ADR-0035 zero-domain/range-triple test is the proof obligation).

This is precisely the **ODR-0029 §R3** "domain/range-as-SHACL-constraint pattern": domain/range sit on the *validation* side; "RDFS only ever put them on the wrong side." Adopting hm ODR-0014's repeated-domain is sound for OPDA **because** the SHACL dual already exists and already enforces — confirmed FOR.

**One SHACL-practitioner refinement (REVISE-flavoured, non-blocking):** the "any-of" reading of repeated `rdfs:domain` is, to an OWL reader, the conjunction (RDFS 1.1 §3.2; multiple `rdfs:domain` = intersection). hm makes it legible with a module-header note + per-property `rdfs:comment`. OPDA must do the same — but I want it *machine-checkable*, not just prose. See Q3: the excluded-construct meta-shape should be paired with a **positive** lint that asserts every multi-domain object property in the relationship layer has a corresponding `sh:or` co-domain/bearer shape. Prose conventions rot; a shape does not.

### Q2 — FP/IFP: carve out to SHACL-only. Functional → `sh:maxCount 1`; InverseFunctional → `dash:uniqueValueForClass`

**Verdict: REVISE (do not adopt hm ODR-0054 wholesale for FP/IFP; carve them out to the SHACL home). Ballot: FOR the carve-out.**

This is the question I carry, and the SHACL home is the right home for three reasons.

**(1) IFP's semantics ARE `owl:sameAs`, which OPDA constitutionally bans.** OWL 2 (Structural Specification & Functional-Style Syntax, §9.2.8 Inverse-Functional Object Properties; and OWL 2 Primer §2.3.5) defines `owl:InverseFunctionalProperty` so that if `x P z` and `y P z`, then `x owl:sameAs y` is entailed. The *meaning of the axiom* is identity-merge on shared key. OPDA bans `owl:sameAs` across exactly these surfaces — ODR-0005 §R5 ("No `owl:sameAs` across the UPRN surfaces — unanimous, irreversible inference propagation") and ODR-0017 §R6 (a data-quality rule MUST NOT produce `owl:sameAs`). To author `owl:InverseFunctionalProperty hasUPRN` (or `hasLEI`, `hasNINumber`) is to publish to the Web the very proposition the corpus holds false: that two records sharing a UPRN are the same individual. OPDA's whole identity doctrine (ODR-0005's three-class Property pattern; bounded-context identity) is that a shared UPRN is a **join key with graceful degradation**, NOT an identity assertion — succession is `prov:wasDerivedFrom` + a reified `UPRNSuccessionEvent`, never `sameAs`. IFP contradicts that at the axiom level. **IFP is out.**

**(2) The SHACL dual does what we actually want — it FLAGS the collision instead of FUSING the individuals.** This is the decisive practitioner point. `dash:uniqueValueForClass` (DASH constraint-components vocabulary — Knublauch, DASH data shapes; the W3C-track companion to SHACL) validates that, within a target class, the value of a property is unique across all focus nodes; a duplicate produces a **validation result** (configurable severity). That is the *opposite behaviour* to IFP under the same input: where IFP silently concludes "same individual" and propagates every property of one onto the other irreversibly, `dash:uniqueValueForClass` raises a *visible, reversible* "two nodes claim the same key — investigate." For master data — where a duplicated UPRN is a data-quality defect to surface to a human/downstream tool, not a license to merge — flag-not-fuse is exactly the semantics OPDA wants. The SHACL home is not a grudging substitute for IFP; it is the *correct* construct for the requirement.

**(3) Functional → `sh:maxCount 1`, and the redundant OWL axiom should be dropped (session-035 rule).** `owl:FunctionalProperty` says "at most one value in all possible worlds" (open-world). `sh:maxCount 1` says "at most one value in this graph" (closed-world). For OPDA — which never reasons, and whose master-data records are closed-world conformance targets — `sh:maxCount 1` is the operative constraint and **already carried** in the shapes (e.g. `hasAssertedCapacity` `sh:maxCount 1`, `ownerType` `sh:maxCount 1` in `opda-agent-shapes.ttl`). Authoring `owl:FunctionalProperty` *in addition* adds an OWL axiom that (a) is never evaluated and (b) has a safe, evaluable, native substitute already present. That is exactly the **session-035 drop-redundant-axiom-where-safe-substitute rule** (8–0–0): "keep the *information*, drop the *unsafe-and-redundant axiom*." There the axiom retired was `owl:equivalentClass` (substitute: `skos:altLabel` + `skos:exactMatch`); here it is `owl:FunctionalProperty` (substitute: `sh:maxCount 1`). Same logic, same disposition. **FP → SHACL-only; do not author the redundant OWL axiom.**

**Distinguishing this from Q1.** A fair challenge (I expect Allemang to press it): "you ACCEPT documentary domain/range under R2 — model-but-don't-evaluate — so why REJECT documentary FP/IFP? Both are unevaluated OWL signal." The answer is the per-construct discipline ADR-0049 already adopts ("engineering-act = ontological-act": author the construct only where its W3C semantics AGREE with OPDA's enforced behaviour). Domain/range's documentary "any-of" reading **agrees** with what SHACL enforces (the disjunctive co-domain) — the OWL signal and the SHACL act point the same way. IFP's documentary reading **disagrees** with what SHACL enforces: IFP says "merge," SHACL says "flag." Publishing IFP would document a behaviour OPDA does not have and a proposition (`sameAs`) OPDA bans. That is not model-but-don't-evaluate; that is model-a-falsehood. FP is the milder case — not a falsehood, just redundant — and falls to session-035, not to the sameAs ban. So: **domain/range IN (documentary, agrees), FP SHACL-only (redundant, session-035), IFP OUT (contradicts, sameAs ban).**

This lands where session-048's panel already landed (4–0 against authoring FP/IFP; IFP out, FP→`sh:maxCount 1`/`dash:uniqueValueForClass`). The directing authority's "adopt hm" decision was about **Q1's form** (repeated-domain over unionOf); it did not reach into Q2. OPDA's stronger local constraints (the `sameAs` ban) override hm ODR-0054 here. hm could afford author-FP/IFP because hm does not constitutionally ban `sameAs`; OPDA does.

### Q3 — Adopt hm ODR-0030 excluded-construct CI meta-shape + ODR-0028b disjointness dual

**Verdict: AFFIRM (both), with a SHACL-side amendment. Ballot: FOR.**

**The excluded-construct CI meta-shape — adopt, this is my native territory.** hm ODR-0030 §Meta-Shape is a `sh:NodeShape` with `sh:sparql` constraints that fail CI if any excluded OWL construct (`owl:disjointUnionOf`, `owl:Restriction`, `owl:unionOf`/`intersectionOf`/`complementOf`, cardinality restrictions, `owl:oneOf`, `owl:hasKey`) appears. This is the correct mechanism (SHACL Core §5.2.6 SPARQL-based constraints) and it is exactly what keeps OPDA's "boolean-constructor-free, never-reasoned" corpus honest. **For OPDA, the excluded set must additionally include `owl:InverseFunctionalProperty` and `owl:FunctionalProperty`** — because of Q2. hm permits FP/IFP (its ODR-0030 table lists them as permitted-but-not-inferenced); OPDA must *exclude* them at the meta-shape, so that a future contributor cannot reintroduce `owl:InverseFunctionalProperty hasUPRN` and silently publish the `sameAs` OPDA bans. The meta-shape is the enforcement that makes the Q2 carve-out durable rather than a one-time cleanup. So: adopt hm's meta-shape **and add `owl:FunctionalProperty` + `owl:InverseFunctionalProperty` to OPDA's excluded list** (OPDA's exclusion is stricter than hm's — by design).

**Disjointness dual (`owl:disjointWith` + `sh:not`/`sh:xone`) — adopt, scoped.** The dual pattern is right and I have argued it before (I am on record in hm ODR-0028b §Expert Positions: "Concrete `sh:xone` patterns, both layers"). The OWL `owl:disjointWith` documents the formal commitment (Protégé/LLM-visible); the SHACL `sh:not` (identity-incompatible classes, no common superclass) / `sh:xone` (sibling subtypes under a common superclass) enforces it. Concretely:

- **Rigid Kinds only, NEVER anti-rigid Roles.** Seller/Buyer exclusivity is anti-rigid (a Person can be Seller in one transaction and Buyer in another) — it stays **transaction-scoped SHACL**, exactly as session-047 Q4 ruled, never a class-level `owl:disjointWith`. This is the OntoClean rigid/anti-rigid carve-out (Guarino & Welty 2002) and I defer to Guarino on the formal grounding; from the SHACL side I simply confirm: do not author `opda:Seller owl:disjointWith opda:Buyer` and do not emit a class-level `sh:not` between them — the exclusivity is a per-transaction `sh:xone`/`sh:not` within the transaction shape.
- **Pairwise `owl:disjointWith`, NOT `owl:AllDisjointClasses`.** No OPDA rule consumes the list form, and `owl:AllDisjointClasses` is a list-bearing construct that buys nothing here. Pairwise is simpler and self-documenting (hm ODR-0028b §"Pairwise `owl:disjointWith` — use for 2-3 classes without a common superclass").
- Authoring the `owl:disjointWith` axioms **arms the currently-vacuous ADR-0035 disjointness gate** — right now there is nothing for it to check.

**My one held concern from session-048, now largely moot, restated honestly:** I dissented on a class-level *partition* axiom (`owl:disjointUnionOf` / `owl:AllDisjointClasses`) on cost-discipline grounds — a consumer that reads a class-level partition off the axiom and cannot reconstruct it from `sh:xone`+SHACL. Under this session's disposition there is **no partition axiom** (pairwise `disjointWith` only, no `AllDisjointClasses`, no `disjointUnionOf` — the latter is excluded by the meta-shape), so the concern does not arise. I withdraw it as moot, contingent on the disposition staying pairwise-only. Re-open trigger: if anyone proposes `owl:AllDisjointClasses` or `owl:disjointUnionOf` for a covering partition, my cost-discipline dissent revives (the partition is not reconstructible from the per-class `sh:not` shapes alone).

### Q1 — Repeated-`rdfs:domain` "any-of" + SHACL `sh:or` carries 100% enforcement

**Verdict: AFFIRM. Ballot: FOR.**

The OWL `rdfs:domain`/`rdfs:range` is documentary (the "any-of" reading, valid because OPDA never evaluates it — ODR-0025 §R1 frozen 7-rule closure; ODR-0026 §R2 model-but-don't-evaluate). The SHACL `sh:or` carries 100% of enforcement, and this is **already the emitted state of the corpus**: `FoundsRangeShape`/`PlaysRangeShape`/`RolePlaySubjectShape`/`RolePlayShape`/`SellerShape`/`BuyerShape` in `opda-agent-shapes.ttl` all carry `sh:or` co-domain/bearer disjunctions at `sh:severity sh:Violation` — a non-conforming graph is rejected at the gate (SHACL Core, Knublauch & Kontokostas eds. 2017, §3 Validation, §4.6.1 `sh:or`). This is exactly ODR-0029 §R3's "domain/range-as-SHACL-constraint pattern": domain/range sit on the validation side; the closed-world `sh:or` is the only layer that rejects a mis-typed node.

**Binding rider (I join Hendler's HOLD-non-blocking and Allemang/Guarino here):** the "any-of" reading is the *negation* of the standard RDFS §3.2 conjunction. It is sound for OPDA only because the closure never fires, and legible only if the convention is documented AND machine-checkable — module-header `skos:editorialNote` + per-property `rdfs:comment`, both **CI-gated** (a `ci-domainIncludes-header` check), not prose decoration. I add the SHACL-practitioner refinement: pair the negative excluded-construct meta-shape (Q3) with a **positive** lint asserting every multi-domain relationship-layer object property has the matching `sh:or` dual present. Prose conventions rot; a shape does not. **Both limbs are CI-gated MUSTs (co-signed with Hendler, DA), not SHOULDs** — they are complementary, not redundant: the header-note gate (`ci-domainIncludes-header`) makes the convention READABLE (the human/LLM reader is told "any-of, not §3.2 conjunction"); the positive `sh:or`-dual lint makes it TRUE (a documentary "any-of" domain with no `sh:or` dual is a half-authored disjunction — documented but unenforced, which lets a mis-typed node through). Header-without-lint = legible but possibly unenforced; lint-without-header = enforced but illegible; only both convert the shibboleth into an enforced convention. These two interlock with the Q4 `ci-object-property-coverage` limb-(b) inversion (where "documented any-of WITH matching `sh:or` dual" is the per-predicate pass-condition) — so the full **Q1+Q4 enforcement bundle is three coordinated checks** (header-note file-level declaration + `sh:or`-dual corpus-wide sweep + limb-(b) per-predicate gate); record them together so none is dropped in isolation. ### Q2 — FP/IFP: carve out to SHACL-only (do NOT adopt hm ODR-0054 wholesale)

**Verdict: REVISE. Ballot: FOR the carve-out — AGAINST authoring `owl:InverseFunctionalProperty` (out entirely) and AGAINST authoring `owl:FunctionalProperty` as a documentary layer; FOR the SHACL home (`sh:maxCount 1` for Functional; `dash:uniqueValueForClass` for the IFP signal).**

This is the question I carry, and the SHACL home is the *correct* home, not a grudging substitute, on three grounds:

1. **IFP's semantics ARE `owl:sameAs`, which OPDA constitutionally bans.** OWL 2 (Direct Semantics §2.3.5; Primer §4.2, the `foaf:mbox` key pattern): if `x P z` and `y P z` for IFP `P`, then `x owl:sameAs y` is entailed — the key-ness *is* the merge; there is no separable "identity-key" proposition above it. OPDA bans `owl:sameAs` across exactly the identifier spine where you would author IFP (ODR-0005 §R5, unanimous 12-0; ODR-0017 §R6; ODR-0025 §R2 lists FP/IFP precisely because they "Trigger `owl:sameAs` identity-merge"). To author `owl:InverseFunctionalProperty hasUPRN` is to publish to the Web the proposition the corpus holds false. **IFP out entirely.**

2. **The SHACL dual FLAGS the collision instead of FUSING the individuals — and flag-not-fuse IS the master-data requirement.** `dash:uniqueValueForClass` (DASH, TopQuadrant; the W3C-track SHACL companion) validates that within a target class no two focus nodes share a value, producing a *visible, reversible* validation result — a human/process resolves the duplicate, possibly **as** a UPRN-succession event (ODR-0005 §6a), possibly as a data error. That is the opposite of IFP's silent, irreversible merge (RDF 1.1 Semantics, Hayes & Patel-Schneider 2014, §6 on `sameAs` propagation). Guarino's identity-theoretic frame and mine stack: he shows IFP asserts a *false identity proposition* (an identifier promoted to a constitutive global IC — an OntoClean rigidity-clash, Guarino & Welty 2002); I show the SHACL home gives the behaviour OPDA *wants* (collision-flagging) — strictly better than IFP even setting identity theory aside. **Sortal scope (Guarino's seat grounds the identity criterion; I confirm from SHACL): `dash:uniqueValueForClass` is scoped WITHIN each identity-bearing sortal, NEVER cross-context — and the within-sortal scope IS the bounded-context identity criterion expressed in SHACL, not a SHACL implementation detail.** A UPRN legitimately recurs across the three ODR-0005 Property-family Kinds (`opda:Property`/`opda:LegalEstate`/`opda:RegisteredTitle` — three co-referring Substance Kinds, NOT identical), so an **unscoped cross-sortal uniqueness on UPRN would BE `owl:InverseFunctionalProperty` in SHACL clothing**: it presupposes the identifier individuates across the universe (the global-IC claim ODR-0005 denies) and would fire a violation precisely on the *legitimate* co-reference (Property p + RegisteredTitle t sharing a UPRN is the CORRECT state, not a collision). So the carve-out's positive half is not "use `dash:` instead of IFP" full stop — it is "use `dash:` scoped per-sortal, and NEVER widen it past the sortal," or the substitute silently becomes the axiom it replaced. `dash:uniqueValueForClass` defaults to per-`sh:targetClass`, which is the right default; the discipline is the negative one (never widen cross-class). Three layers, one verdict: identity theory says IFP asserts a false global IC; the SHACL home says flag-not-fuse is the behaviour OPDA wants; the sortal-scoping says the SHACL home must itself be sortal-bounded to STAY faithful.

3. **Functional → `sh:maxCount 1`; the redundant OWL axiom is dropped under session-035.** `sh:maxCount 1` is the operative closed-world constraint, already carried in the shapes (`hasAssertedCapacity`, `ownerType` — `opda-agent-shapes.ttl:148,211`). Authoring `owl:FunctionalProperty` in addition is an axiom that is never evaluated AND has a safe evaluable native substitute already present — exactly the **session-035 drop-redundant-axiom-where-safe-substitute rule** (ODR-0026 §R2 amendment, 8-0-0; the rule that retired the three `owl:equivalentClass` aliases). Same logic, same disposition. I do not even advocate Allemang's narrow hand-curated modal-marker FP exception; if the council wants the ontological-fact-vs-data-shape distinction marked, an annotation carries it at least as well as an axiom whose formal content `sh:maxCount 1` already states and fires.

**The asymmetry with Q1, stated for the record (this is what makes the carve-out principled, not a reflex):** ADR-0049's per-construct discipline ("engineering-act = ontological-act") authors a construct as documentary signal only where its W3C semantics AGREE with OPDA's enforced behaviour. Domain/range's documentary "any-of" AGREES with the SHACL act (the `sh:or` enforces the same disjunctive shape) → IN. IFP's documentary reading DISAGREES (IFP says merge; the SHACL dual says flag) → OUT — that is not model-but-don't-evaluate, it is model-a-falsehood. FP is the milder case (redundant, not false) → SHACL-only under session-035. hm could afford author-FP/IFP because hm has no `sameAs` ban; OPDA's stronger local constraint overrides hm here. This is the one place the "adopt hm" decision yields to OPDA's own prior ruling — and it does so *by the very test the adoption is built on*.

**Citations:** OWL 2 Direct Semantics §2.3.5, OWL 2 Primer §4.2 (IFP → `sameAs`, the key pattern); RDF 1.1 Semantics §6 (`sameAs` irreversible propagation); SHACL Core §4.2.1 (`sh:maxCount`); DASH `dash:uniqueValueForClass` (TopQuadrant); ODR-0005 §R5/§6 + §Anti-patterns (UPRN as contingent Quality, not the IC; `sameAs` rejected 12-0); ODR-0017 §R6; ODR-0025 §R2; ODR-0026 §R2-amendment / session-035 (8-0-0).
