*Lens: RDF/RDFS/OWL standards conformance + linked-data web architecture. The engineering act IS the ontological act — how you declare the term is the commitment you publish.*

## Q1 — String literal vs resource (IRI)

**Verdict: REVISE. Ballot: FOR (with the exact representation stated below).**

**Gandon:** A bare `xsd:string` is the one representation that the web architecture cannot follow. `"Substance Kind"` is opaque: you cannot dereference it, you cannot attach a definition or a `skos:scopeNote` to it, you cannot state `skos:exactMatch gufo:Kind`, and `"RoleMixin"` vs `"Role Mixin"` vs `"rolemixin"` are three different facets to a SPARQL `GROUP BY` with no machine-detectable error. The Linked Data principles (Berners-Lee, *Design Issues: Linked Data*, 2006, principles 2–4 — "use HTTP URIs so people can look things up; when someone looks up a URI, provide useful information; include links to other URIs") are a checklist a string fails on every line. RDF 1.1 Concepts (W3C Rec, 2014, §3.3) is explicit that a literal is a *value*, terminal — it is not a node you can say anything further about, so a string-valued facet is a dead end by construction. The fix is to make the value an **IRI**: mint a governed SKOS concept scheme `opda:UFOCategoryScheme` whose members (`opda:SubstanceKindCategory`, `opda:RelatorCategory`, …) carry `skos:prefLabel`, `skos:definition`, the DOLCE-lineage `skos:scopeNote` ODR-0030 Rule 7 already mandates, and `skos:closeMatch`/`skos:exactMatch` to the gUFO IRIs (`gufo:Kind`, `gufo:Relator`, `gufo:RoleMixin`). That is the textbook SKOS modelling pattern — Allemang, Hendler & Gandon, *Semantic Web for the Working Ontologist*, 3rd ed. (2020), ch. 10 — and it is what makes the category a first-class citizen of the web rather than a private code. A resource value is precisely what makes the term reusable and linkable, which is the whole point of doing this in RDF instead of a column in a spreadsheet.

**To Baker (literal vs VES resource):** I read your standing position as the DCMI line — a Vocabulary Encoding Scheme is the correct home for a closed value space, and an undocumented literal is a value with its semantics deleted. I agree, and I want to sharpen the split you will draw, because the BRIEF's established fact 2 is decisive here: `opda:ufoCategory` is doing **double duty**. The 9 endurant/perdurant categories on classes (Substance Kind, Relator, Role, …) are *genuine UFO categorial work* and earn full SKOS concepts with gUFO `closeMatch`. But the 35 "Quale-in-Region" scheme tags are, by ODR-0030 Rule 2, **register-deference** — EPC/HMLR/INSPIRE closing the value space, *not* UFO categorising. Those are exactly your VES case: a controlled value drawn from an external authority, which should resolve to (or `skos:closeMatch`) the *register's* scheme, not be dressed as a UFO concept. So I'd refine "literal → resource" into a two-scheme answer: one UFO-category scheme (mine), one register-deference encoding the externally-closed values (yours). One predicate spanning both was the original sin; promoting to resources is also the moment to stop conflating them.

## Q2 — Markup & graph placement (the decisive question)

**Verdict: REVISE — retype to `owl:AnnotationProperty`; for the class-axis IRI value, the conformant typing of an annotation whose object is an IRI is *still* `owl:AnnotationProperty`, NOT `owl:ObjectProperty`. Ballot: FOR.**

**Gandon (a) — predicate typing.** This is where standards conformance settles a question the prose has been carrying loosely. ODR-0030 (Decision Outcome; Rule 1) calls `opda:ufoCategory` "an inert provenance record … without entailing logical axioms." But the artefact types it `owl:DatatypeProperty` (`foundation.py:523`, emitted at `opda-classes.ttl:90`). Those two statements are in tension, and OWL 2 resolves it against the current typing. In the OWL 2 *Structural Specification and Functional-Style Syntax* (W3C Rec, 2nd ed., §5.5 and §10), **annotation properties carry no model-theoretic consequence** — §10.1: "Annotations … have no effect on the logical aspect of an ontology — that is, on its formal meaning." A `DatatypeProperty`, by contrast, is a *logical* property (§5.5): it lives in the object-property/datatype-property signature, it is what `rdfs:domain`/`rdfs:range`/`owl:FunctionalProperty` and the OWL Direct-Semantics entailment (§2.4) range over. So the honest, conformant way to say "this entails nothing" is not to write a `DatatypeProperty` and *promise* not to reason over it — it is to type it `owl:AnnotationProperty`, which makes its inertness a **formal property of the term**, not a deployment convention you must police forever. This is the engineering-act-is-the-ontological-act principle in its purest form: the commitment "documentary, non-axiomatic" should be *spelled in the typing*, not asserted in a comment that the typing contradicts.

This directly satisfies ODR-0030 Rule 1 in a way the current artefact does not. Right now the inertness rests entirely on the ODR-0029 regime *choosing* to exclude domain/range and on the quarantine *file placement* — Knublauch's standing condition (BRIEF fact 4) is load-bearing precisely because nothing in the term itself guarantees inertness. Retype it `owl:AnnotationProperty` and the inertness is intrinsic: it would entail nothing under *any* OWL profile or regime, not just the shallow RDFS-Plus one currently configured. That is strictly stronger than the status quo and removes a standing fragility.

**(b) — graph placement.** Here I must report a non-conformity the BRIEF foregrounds and I have verified. ODR-0030 Rule 1 says the tags "remain **annotation-graph-only** (ODR-0004 §3a)"; ODR-0004 §3a's own table is unambiguous — `opda-classes.ttl` is "OWL/RDFS class graph … **No `sh:` triples**", and the annotation graph `opda-annotations.ttl` is the home for advisory predicates. Yet every `ufoCategory` triple — the declaration *and* all 39 class tags + 47 scheme tags — is asserted in the class/vocabulary graphs (`opda-classes.ttl:32–90`, `opda-agent.ttl`, `opda-property.ttl`, …); `opda-annotations.ttl` carries **zero** (verified: `grep -c → 0`). So Rule 1's literal text is **not** satisfied by the current artefact. Whether that has *entailment* consequences today is a separate matter (ODR-0029: it doesn't — a domain-less `xsd:string` datatype property propagates nothing under the shallow regime). But ODR-0004 §3a is a *separation-of-artefacts* contract, not only an entailment one, and the CI gates it lists (the `ASK` for `sh:` triples in the annotation graph) do **not** check the converse — they never catch an annotation predicate sitting in the *class* graph. So the breach is real and currently un-gated. **My disposition:** move the declaration and every per-term tag into `opda-annotations.ttl` (and per-module `opda-*-annotations.ttl`), where Rule 1 says they belong, and retype as `owl:AnnotationProperty` per (a). The two fixes are one move: an annotation property in the annotation graph is the coherent design; a datatype property in the class graph is the incoherent one.

**On Cagle's re-open trigger (i) — and whether it has already fired.** Trigger (i): "the layer is ever made reasoned-over (moved out of the annotation-graph quarantine into a graph that drives inference or SHACL)." I want to be precise, because the literal reading and the operative reading diverge and the Council must not record a falsehood. *Literally*, the tags are not in `opda-annotations.ttl` and never have been, and they are physically in the class graph that `opda-inference.ttl` is the classes-alone export of (ODR-0004 §3a) — so on the strict wording, the quarantine has not been *maintained*, even if it was never *breached by a move*. *Operatively*, ODR-0029's regime means the class graph drives **no** inference from these triples (0 entailed triples) and no SHACL targets them. So trigger (i) has **not** fired in the sense Cagle cares about — nothing UFO-shaped is driving inference or a constraint. The right reading: the artefact has been sitting in technical non-conformity with Rule 1's *placement* clause while remaining conformant with its *inertness* intent. Retyping to `owl:AnnotationProperty` and relocating to the annotation graph closes both — and, crucially, makes trigger (i) **un-fireable by accident**: an `owl:AnnotationProperty` cannot be "made reasoned-over" without an explicit, reviewable retyping, because annotation properties are outside the logical signature by definition (Structural Spec §10.1). That converts Cagle's vigilance condition from "police the file placement and the regime config forever" into "the type system enforces it." I'd put that to Cagle as the strongest possible *answer* to his dissent, below.

**(c) — value space + SHACL `sh:in`.** Yes to a governed SKOS scheme (Q1). On the SHACL `sh:in` shape: I'd permit it but locate it carefully. A `sh:in` over the UFO-category scheme is *validation* (closed-world "is the tag drawn from the sanctioned set?"), which is the correct home for a constraint per ODR-0029 R1 and the ODR-0027 "enforcement is value-keyed not class-keyed" doctrine — it does **not** reason over the tag, it checks it. That is conformant and does not trip trigger (i): SHACL validating *that an annotation's value is in a scheme* is not "the UFO layer driving SHACL" in the categorial sense; it is the scheme policing its own closure, exactly as a VES should. But it is optional belt-and-braces, not load-bearing; if the Council wants minimal surface, the scheme membership alone (resolvable, `skos:inScheme`) already gives typo-safety via dereferenceability.

**To Cagle (keep it a thin inert string):** I expect you to argue that the thinnest inert representation is correct and that any enrichment *entrenches* a layer session-040 called "optional, droppable" — and that an IRI value plus a SKOS scheme is exactly the entrenchment your re-open triggers guard against. I rebut on conformance grounds and concede on scope grounds, and the two combine into a position I think you can sign. **Rebuttal:** a `string` is not the *inert* choice — it is the *unaccountable* choice. Inertness is a property of the OWL typing (`AnnotationProperty`, §10.1), not of the value's lexical form. A string `DatatypeProperty` in the class graph is, on the letter of the OWL signature, *more* logically committed than an `AnnotationProperty` whose object is an IRI — because the datatype property is *in* the logical signature and the annotation property is *out* of it. So your own goal — maximal inertness, minimal entailment commitment — is better served by my proposal than by the status quo string. **Concession:** I am not asking to deepen the *commitment*. The IRI does not assert `gufo:Kind` as a superclass (that would be the ADR-0034 quarantined typing, which stays quarantined); it asserts a `skos:closeMatch` — a *mapping* relation that SKOS (W3C Rec, 2009, §10) defines as carrying no class-subsumption and no logical entailment. We are making the *documentation* first-class and dereferenceable; we are not making the *foundation* heavier by one axiom. That is the opposite of entrenchment: it is making the layer **honestly inspectable and cleanly removable** — a dereferenceable SKOS scheme with `closeMatch` edges is *easier* to retire (delete one named graph) than 86 inline string literals scattered across eight class files. If your dissent is "don't let it grow teeth," `owl:AnnotationProperty` is the muzzle, and it is tighter than the one we have now.

## Q3 — Web pages / documentation IA

**Verdict: AFFIRM. Ballot: FOR.**

**Gandon:** The engineering act is the ontological act on the web too: a term's commitment is *what you serve when someone dereferences its URI*. Three things follow, all conformant with the W3C TAG `httpRange-14` resolution (2005) and ADR-0044's "ontology as web pages."

1. **`/pdtf/ufoCategory` must dereference to a term page** — and per BRIEF fact 1 it already does (ADR-0044 Phase 5c). That is exactly right: by `httpRange-14`, a `200 OK` on the property's slash-URI commits us to it denoting an information resource describing the term, which is what we want for a property. Keep it; it now describes an `owl:AnnotationProperty` (Q2) with the DOLCE-lineage `skos:scopeNote` (ODR-0030 Rule 7) rendered on the page.

2. **Each UFO-category *concept* (Q1) must dereference too** — `/pdtf/SubstanceKindCategory` etc. should `200` to a SKOS concept page showing `skos:definition`, the DOLCE/UFO lineage, and the outbound `skos:closeMatch gufo:Kind` as a real link (Linked Data principle 4 — *link to other URIs*, including off-site to gUFO). A `/ontology/category/[slug]` route already exists in the repo (`src/pages/ontology/category/[slug].astro`), so the IA is half-built; wire the category index + per-category pages to the new scheme. This is the difference between a vocabulary that participates in the web of data and one that merely mentions other vocabularies in prose.

3. **The honesty disposition (ODR-0030 Rule 7) is itself a first-class, machine-readable artefact**, not page chrome: "UFO-informed, not UFO-grounded" belongs in the `skos:scopeNote` on the property *and* surfaced on the page, so a consumer who only ever fetches the RDF gets the same caveat as a human reading the HTML. That is the test of an honest term: the disclaimer survives content negotiation.

The single `/ontology/foundational-ontology` hub stays as the *critique/alternatives* essay (it already reads the live category distribution from the TTL); the per-term badge and per-category pages are the *structured-data* face of the same facts. No conflict with ODR-0038 or ADR-0044 — this is what they ask for.

**To Baker:** your VES point cashes out as a *page*, not just a triple: the register-deference scheme (Q1, Rule 2) should dereference to a page that says, in prose and in `skos:scopeNote`, "this value is closed by EPC/HMLR/INSPIRE — UFO is not doing categorial work here," discharging Rule 2's "must stop presenting them as UFO" on the web surface, not only in the model.

## Q4 — What else to mark up

**Verdict: REVISE (markup only the deontic correlativity now; the rest stays prose or stays quarantined as-is). Ballot: FOR.**

**Gandon:** The discipline is the same throughout: mark up as a **resource/relation** only what is doing relational work a string cannot carry, and keep everything else inert. Triaging the candidates by what RDF can faithfully and cheaply represent:

- **OntoClean meta-properties (±R/±I/±D):** keep as **prose / design-time judgement**, not RDF triples. OntoClean's rigidity/identity/dependence are *meta-properties of the modelling decision* (Guarino & Welty), evaluated when we choose subclass-vs-facet (ODR-0027) — they are not data about the conveyance. Materialising them as triples would assert metaphysics into the artefact for no consumer. They are the *reason* for a modelling choice; the choice itself is already in the class graph. (This agrees with ODR-0030's own framing: "the artefact carries an inert provenance record of that governance" — governance, not metaphysics, gets recorded.)

- **gUFO `rdf:type` markers (ADR-0034):** leave **exactly where they are — quarantined**. They are the one genuinely *logical* commitment (a real `rdf:type gufo:Quality` is class membership, not an annotation), which is precisely why ADR-0034 gated and quarantined them and why ODR-0030 Rule 1 keeps them annotation-graph-only. Do **not** promote, do **not** broaden. This is the case where the string→resource argument *reverses*: here the resource (a typing axiom) is the heavy thing, and it must stay caged.

- **UFO-L Hohfeldian Claim-Right ↔ Duty correlativity:** **mark this up as a relation — this is the one promotion I affirm now.** ODR-0030 itself names the deontic core as the bounded reason UFO earns its place (Decision Outcome; Rule 6) and the relational-reification primitive as "the strictly irreducible thing." Correlativity is *inherently a relation between two resources* (a Claim-Right and its correlative Duty) — it is the textbook case where a literal is impossible and an `owl:ObjectProperty` (or, if we want it inert, a documented `skos:related`-style mapping) is the only faithful representation. Per the OWL 2 Structural Spec §5.5, an object property is the conformant typing for a resource-to-resource link, and Griffo, Almeida & Guizzardi's UFO-L is the cited source (ODR-0030 More Information). **But** note the graph discipline this triggers: a *reasoned* `owl:ObjectProperty` asserting correlativity in the class graph would be a real entailment commitment, which is fine *if* the Council wants it reasoned — it is genuine domain modelling, not advisory metadata, so it does not engage trigger (i) (that trigger is about the *UFO-category provenance layer*, not about modelling the domain's actual deontic relations). If the Council wants it inert for now, model it as an annotation-graph mapping; if it wants it to do work (e.g. SHACL checking that every charge has a correlative duty), it graduates to the class/shapes graphs as first-class domain modelling. I'd recommend the latter eventually — correlativity is exactly the kind of constraint a conveyancing file *should* validate — but that is a separate council.

- **IAO crosswalk (ODR-0030 Rule 4, adopt-now):** **mark up as `skos:exactMatch`/`closeMatch`/`rdfs:subClassOf` edges, in the annotation graph, referenced-not-imported** — exactly as Rule 4 already specifies, on the `opda:EPCCertificate rdfs:subClassOf prov:Entity` precedent. This is conformant Linked Data (principle 4: link to external URIs) and the `prov:Entity` precedent shows the mechanism is already in use and inert. Affirm Rule 4 as written; it is the model citizen of this whole question — a thin, dereferenceable, reasoned-over-nothing bridge.

**To Cagle:** on this question I am *with* you, not against — three of the five candidates I keep inert or in prose precisely to honour "optional, droppable." The only thing I promote (deontic correlativity) is the one ODR-0030 already certified as load-bearing and irreducible, so promoting it entrenches nothing that session-040 did not already affirm. **To Baker:** the IAO crosswalk (Rule 4) is your VES-to-external-authority pattern again — `skos:exactMatch` to an IAO IRI is a mapping to an external standard's encoding scheme, the same shape as the register-deference tags, and it should be governed and dereferenceable on the same discipline.
