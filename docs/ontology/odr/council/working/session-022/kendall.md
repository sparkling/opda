## Q1 — Form ↔ base association

**VOTE: AGAINST the framing that a custom predicate is needed; FOR the existing SHACL-profile-as-graph + targeting + provenance link.** The current design is already idiomatic; nothing to add.

**Named convention:** SHACL 1.2 *targeting* (`sh:targetClass` / `sh:targetNode`) + W3C DX-Prof / PROF *profile-of-a-base* relation; FIBO/OMG *module imports* (`owl:imports`) for the definitional layer. The form-profile associates to the base **two ways, both standard**: (a) operationally, a profile shape *targets* the base class — that IS the association SHACL is built for; (b) provenance-wise, `dct:source` ties each shape to its form-question IRI (ODR-0010 Rule 4). A profile graph is first-class RDF (the architect is right); it needs **no** bespoke "this profile overlays that base" predicate beyond what ODR-0010 already has (`opda:ValidationContext` + `opda:overlaysContext`).

**Where I land vs the architect:** the architect's "a SHACL profile is first-class RDF" is correct and is *already* the design. FIBO would associate a view/module to its base by `owl:imports` + shared namespace + targeting — never by a separate membership scheme. So on Q1 there is **no over-engineering**: ODR-0010 is convention-clean. The one honest nuance: `opda:overlaysContext` is a *local* reification of "this profile is the validation context for form X." That is warranted — it is exactly ODR-0010's `ValidationContext` (a named, dereferenceable context with a fixed model theory, which Guarino's withdrawal condition required). PROF would call this a `prof:Profile` with `prof:isProfileOf` the base; OPDA's `ValidationContext`/`overlaysContext` is a defensible local spelling of the same relation. **Recommend:** keep ODR-0010 as-is; optionally note PROF as the external analogue for `overlaysContext` in the catalogue. No new construct.

> **REVISED in rebuttal (supersedes the two paragraphs above on two points — see the post-cross-talk addendum for the full reasoning):** (1) My `owl:imports` alternative is **WITHDRAWN** — ODR-0010's Graph-separation rule forbids it ("Shapes reference classes via targeting, never `owl:imports`"). (2) "Keep ODR-0010 as-is, no new construct" understated it: `opda:overlaysContext` **conflates two standard relations** — `prof:isProfileOf` (→ base spec) + `dct:publisher` (→ community) — and today points at the *community*, which is why the `profiles.py:250` target reads as a "bug." Corrected Q1 vote: **AGAINST `overlaysContext` as-is; FOR `prof:isProfileOf` + `dct:publisher`** (split the two), with `prof:hasResource`/`ResourceDescriptor`/`role:validation` for the profile→shapes binding. ## Q2 — Metadata on a SHACL / profile definition

**VOTE: FOR reusing standard annotation vocabulary on the profile + its shapes; AGAINST any bespoke metadata predicate.**

**Named convention:** OMG/FIBO **Specification Metadata (`sm:`)** + **Commons Annotation Vocabulary (`cmns-av:`)** + DCMI Terms + SKOS. This is the single most settled thing in enterprise-ontology practice and OPDA should mirror it exactly.

A profile graph and its shapes carry metadata the way FIBO carries it on any ontology element:

| What it records | FIBO/OMG convention | OPDA today |
|---|---|---|
| human definition | `skos:definition` (ISO 704) | has `skos:definition` ✓ |
| provenance / origin | `cmns-av:adaptedFrom`, `dct:source` | has `dct:source` (ODR-0010 Rule 4) ✓ |
| maturity / status | `sm:maturityLevel`, `dct:status` | has `dct:status "draft"` ✓ (ODR-0004 block) |
| steward / authority | (FIBO: `sm:` author/copyright) | has `opda:hasSteward` — *local, defensible* |
| usage guidance | `fibo-fnd-utl-av:usageNote`, `cmns-av:explanatoryNote` | `rdfs:comment` |
| defining ontology | `rdfs:isDefinedBy` (implicit in FIBO IRI) | **NOT EMITTED — gap** |

**Finding:** OPDA's per-element metadata is *already* convention-aligned for everything **except `rdfs:isDefinedBy`**, which is absent corpus-wide. That gap matters for Q3. The lesson FIBO teaches here is the rule the architect is implicitly invoking: **carry metadata as reused annotation properties, never invent a predicate when `sm:`/`cmns-av:`/`dct:` already names the relation.** A profile definition is metadata-bearing exactly as any module element is — `dct:source` + `sm:maturityLevel` + `skos:definition`. **Recommend:** (i) keep the existing annotation set; (ii) **adopt `rdfs:isDefinedBy`** on emitted terms (it is the missing standard, and it is the predicate Q3 turns on); (iii) consider cataloguing `sm:maturityLevel` as the maturity vocabulary instead of only `dct:status` if a maturity gradient is ever needed. No bespoke metadata predicate is warranted.

## Q3 — "Which context a term belongs to": bespoke scheme vs standard; is `definedInContext` reinventing provenance / module-of-origin?

**VOTE: AGAINST `definedInContext` as currently specified (S021). It reinvents `rdfs:isDefinedBy` (module-of-origin) fused with `dct:source` (provenance) — both of which OPDA already has or should adopt. This is where S021, which I chaired, over-reached.**

**Named convention:** FIBO **module-of-definition** — a term's home is the ontology/module that defines it, recorded by `rdfs:isDefinedBy` and read from the IRI path; FIBO uses **no parallel "domain-of-belonging" taxonomy**. Provenance-of-conception is `cmns-av:adaptedFrom` / `dct:source`. These two standard predicates *are* the home + provenance that `definedInContext` was minted to carry.

**The honest adjudication against my own S021 synthesis.** At S021 I ratified (6–3) that every owned term gets an authored `opda:definedInContext`, **generated from its `dct:source` provenance**, as its "conceptual home." Re-examined against FIBO convention, that decision has a precise defect:

1. **`definedInContext` generated-from-`dct:source` is `dct:source` with a lossy projection in front of it.** If the home is *computed from* the term's `dct:source` (single-source → that context; many-source → foundation), then `definedInContext` carries **strictly less information than the `dct:source` it is derived from**, plus a mapping that can drift from the source vocabulary. FIBO would never mint a second predicate to restate provenance it already records on the element. Guizzardi's S021 "home is a quasi-identity relation distinct from usage" insight is *correct ontologically* — but the **standard realisation of "module-of-origin / conceptual home" is `rdfs:isDefinedBy`**, not a bespoke annotation into a bespoke SKOS scheme. S021 reached for a new predicate where convention already names two (`rdfs:isDefinedBy` for the home module; `dct:source`/`cmns-av:adaptedFrom` for the originating document).

2. **The bespoke `BoundedContextScheme` is the part to scrutinise hardest.** FIBO's verdict is blunt: it does **not** classify terms into a "business context" SKOS scheme at all — context-of-use is read from `owl:imports` and usage; home is the defining module. OPDA mints a 6-concept SKOS scheme **and** two custom predicates (`servesContext`, `definedInContext`) to do what FIBO does with module structure + `rdfs:isDefinedBy` + import graph. *However* — and this is why I do not vote to delete the scheme outright — OPDA's situation differs from FIBO's in one verified, load-bearing way: **OPDA is single-namespace by ratified decision (ODR-0004, S001 Q7 9–0), with modularity by *file* not by *IRI*.** FIBO reads module-of-definition off the IRI path *because FIBO namespaces by module*. OPDA deliberately does not. So OPDA cannot read "home" off the IRI the way FIBO does — which is the legitimate reason a *home annotation* (some predicate) is needed at all. The error is not "annotate the home"; the error is **(a) inventing `definedInContext` instead of using `rdfs:isDefinedBy`** (→ the seven module TTLs are the home; `rdfs:isDefinedBy` → the module ontology IRI is the FIBO-faithful spelling), and **(b) generating it from `dct:source` so it adds nothing the source didn't already say.**

3. **`servesContext` (usage, derived) is convention-clean and should stay.** Deriving a usage view from the profiles is exactly "read context-of-use from usage," which is what FIBO does informally and what SHACL-AF does formally. No objection. The architect's "used-by is a rule, redundant" is *half right*: it is a rule (correct — derive it), but it is not redundant — a materialised usage view is a legitimate convenience projection, shipped dormant. Keep it.

**So, precisely:**

- **`servesContext`** (derived usage): **KEEP** — idiomatic.
- **`definedInContext`** (authored home): **REPLACE with `rdfs:isDefinedBy` → the home module ontology**, OR demote to a thin alias of it. Do **not** generate the home from `dct:source` (that makes it redundant); the home is the **module the term is emitted into** — which the generator already knows at emit time, with no inference and no `dct:source` projection. The genuine "which of the 6 industry contexts conceptually owns this" question, where it is sharper than "which module file," is answered by `rdfs:isDefinedBy` pointing at a per-context module ontology if/when contexts gain their own module IRIs; until then it is foundation/shared-kernel for the descriptive majority.
- **`BoundedContextScheme`** (6 SKOS concepts): **KEEP but re-scope its job.** It is defensible as the *catalogue of the DDD context map* (the contexts are real domain objects worth naming, with stewards) — FIBO names its domains too. What it must **stop** doing is being the target of a bespoke per-term membership predicate that duplicates module-of-origin. Let `servesContext` (derived) point into it for *usage*; let `rdfs:isDefinedBy` carry *home* (to the module, not the scheme concept, unless contexts become modules).

**Verdict on the architect's Q3 claim:** **upheld in substance.** `definedInContext`-generated-from-`dct:source` *does* reinvent provenance/module-of-origin. The fix is not to abandon home-annotation (single-namespace OPDA legitimately needs it) but to **spell home with `rdfs:isDefinedBy`** (the standard the corpus is missing anyway) and **stop deriving it from `dct:source`.**

## Q4 — Over-engineering verdict + minimal idiomatic design

**VOTE: PARTIAL over-engineering. Q1 and Q2 are convention-clean. Q3's `definedInContext` layer (the S021 addition I chaired) is the over-reach. The bespoke scheme is borderline-justified by single-namespace, but its membership-predicate job is over-built.**

**Honest impact on my S021 verdict:** S021's Q1 outcome — "author a home for every term, generated-from-provenance, un-gated" — **does not survive contact with FIBO convention unchanged.** I withdraw the *un-gating + new-predicate* half. What survives: (a) the home/usage *distinction* (Guizzardi was right that they are different relations); (b) `servesContext` derived (Allemang/Davis/Cagle were right); (c) the firewall + CI guards (Baker). What I retract: minting `definedInContext` as a new predicate and **generating it from `dct:source`**. The S021 synthesis congratulated itself that "generated-from-provenance" answered Davis's anti-drift challenge — but FIBO shows the cleaner answer to the *same* challenge is "it's the defining module; use `rdfs:isDefinedBy`; don't synthesise it from the source at all." S021 found a clever bespoke route to a place convention already had a paved road to.

**Minimal idiomatic design (FIBO-faithful, ODR-0004-respecting):**

1. **Home = `rdfs:isDefinedBy`** → the module ontology that emits the term (the 7 TTL modules; or a per-context module IRI if/when a context owns one). Adopt the predicate corpus-wide (it is missing today). **No `definedInContext`, no derive-from-`dct:source`.**
2. **Provenance-of-conception = `dct:source` / `cmns-av:adaptedFrom`** (already emitted per ODR-0010). Unchanged.
3. **Usage = `servesContext`, derived** via the ODR-0020 SHACL-AF CONSTRUCT, dormant. Unchanged.
4. **Upstream = `consumesFrom` → `opda:Organisation`** (Conformist). Unchanged — this one is genuinely a domain relation, not reinvented metadata.
5. **`BoundedContextScheme`** = the named DDD context-map catalogue (6 concepts + stewards). Keep as reference data; it is **not** the home-membership mechanism. Per-element metadata follows FIBO: `skos:definition` + `sm:maturityLevel` + `dct:source`.

**Honest verdict on the bespoke scheme + custom predicates:** *two* custom predicates were one too many. `consumesFrom` earns its keep (a real Conformist relation FIBO has no exact built-in for). `servesContext` earns its keep as a derived view. `definedInContext` does **not** — `rdfs:isDefinedBy` is its name.

**ODR-0019/0020 amendments:**

- **ODR-0019 Rule 5:** replace `opda:definedInContext` with `rdfs:isDefinedBy` → home module (FIBO module-of-definition). Drop "generated from `dct:source`." Keep the home/usage distinction in prose.
- **ODR-0019 Rule 8 (the S021 amendment I chaired):** **reverse the un-gating.** There is no new home predicate to un-gate; `rdfs:isDefinedBy` is ordinary module hygiene emitted always (not "Context-Map authoring"). The polysemy-machinery gate stays.
- **ODR-0020 Rule 4/5:** the "authored ownership layer" becomes "home = `rdfs:isDefinedBy` (module-of-definition, FIBO)"; D1/D2 split survives as *kernel-module vs scaffolding-module*; keep `consumesFrom` (C) and derived `servesContext` (A/B). Firewall/CI guards stand.

**ADR-0026/0028/0029:** ADR-0026 should declare `rdfs:isDefinedBy` the home predicate (not `definedInContext`), emitted by every module emitter (trivially — each emitter knows its own module IRI), and drop the `dct:source`→home generation step from the descriptive walk (ADR-0028). ADR-0029 (profiles/`servesContext`) is unaffected. Net effect: **less** code than S021's plan, not more — we delete a predicate and a derivation step and reuse a standard one.

## Post-cross-talk addendum (after one rebuttal pass with gandon, allemang, davis)

**Four-way convergence (Kendall, Gandon, Allemang, Davis) on the core:** drop `opda:definedInContext`; record home with standard predicates; keep `servesContext` as a *derived rule* (not stored); keep the SKOS scheme. The architect's Q3 charge is upheld by all four lenses (W3C, pragmatic/derive-don't-declare, DA/YAGNI, and FIBO/enterprise). Allemang's sharpest framing, which I adopt: the design **derived** `servesContext` but **materialised** `definedInContext` though both are functions of `dct:source` — that internal double-standard *is* the over-engineering.

**Resolved fork — the `rdfs:isDefinedBy` target (the one open question my opening left loose).** Both Gandon and Allemang flagged that `rdfs:isDefinedBy` classically points at a *vocabulary/document*, not a *community concept*. My FIBO lens is decisive and I adopt the split: FIBO uses `rdfs:isDefinedBy`/module-of-definition for the **defining document/module** and never points it at a community concept. So:

- **home / vocabulary-of-origin** = `rdfs:isDefinedBy` → the defining **module ontology** (no stretch; the generator knows its module at emit time);
- **DDD community-ownership** (if separately wanted) = `dct:subject` → the SKOS context concept (Gandon's resolution; DCMI-idiomatic for "the topic this resource is about/belongs to");
- **provenance / originating authority** = `dct:source` / `prov:wasAttributedTo` (already emitted).

**Verified fact that closes the fork:** today `dct:source` points at ODR sections (`…/odr/ODR-0008#section-Q5a`) and external authority documents (legislation.gov.uk, EUR-Lex, OpenID) — **never at a context concept**. So the three predicates carry three genuinely distinct referents with **zero collision**, and `rdfs:isDefinedBy`→module is a document→document mapping with no semantic stretch. This corrects Allemang's assumption that the home-pass default resolves a `dct:source` to an *industry context* — as `dct:source` exists, it resolves to a document.

**Correction conceded to Gandon (rebuttal pass) — `rdfs:isDefinedBy` ALONE under-counts; `dct:subject` is load-bearing, not optional.** My opening leaned on "module = home." Gandon's catch, which I accept: **OPDA's six DDD contexts are NOT its seven TTL modules.** The TTL split is by *ontological concern* (foundation / agent / descriptive / claim / governance / transaction / property — confirmed in ODR-0020 §Context); the six contexts are *industry communities of practice*. The descriptive walk emits a leaf into `opda-descriptive.ttl` **by concern**, so `rdfs:isDefinedBy`→`opda-descriptive.ttl` answers "which file declares `floodRisk`" but **structurally cannot** answer "which of the six communities owns it." Two genuinely different facts — so `dct:subject`→context is **not** an optional flourish; it carries the one ownership axis `rdfs:isDefinedBy` cannot. (Two accuracy corrections from Gandon, accepted: FIBO carries module-of-origin via the **IRI path**, not actual `rdfs:isDefinedBy` triples — the precise framing is "`rdfs:isDefinedBy` is the Rec-grade reconstruction of the per-module-IRI home signal OPDA surrendered at ODR-0004," not "do what FIBO does." Residual Q3 divergence with Gandon is therefore *only* the YAGNI judgment "must OPDA carry community-ownership?" — I say yes: it is authored editorial content not recoverable from module or usage; if Davis cuts it on no-named-consumer we collapse to `rdfs:isDefinedBy` alone, a clean named fallback.)

**Correction to my own Q1 — the `owl:imports` alternative is FORECLOSED.** My Q1 framing offered SHACL targeting "+ FIBO `owl:imports`" as a route. Gandon caught that **ODR-0010's Graph-separation rule forbids exactly this**: *"Shapes reference classes via targeting, never `owl:imports`. … `sh:minCount 1` is not `owl:minCardinality`"* — banned to keep closed-world SHACL out of the open-world TBox (Guizzardi/Guarino, S001/S010). I withdraw the `owl:imports` option. The form→base association then needs a predicate that is neither `owl:imports` nor node-grain targeting — `prof:isProfileOf` (profile-grain, range `dct:Standard`) names it. By my own Q3 logic (use the standard a bespoke predicate is merely spelling), I accept `prof:isProfileOf` as the named association: `opda:overlaysContext` is a local spelling of `prof:isProfileOf` **+** `dct:publisher` (the community half it conflates); split them, fixing `profiles.py:250` as a side-effect. This does **not** collapse my hold on keeping `ValidationContext` — that hold rests on PROF's WG-Note status + VC reifying constraint-grain "required RELATIVE TO this context" (Guarino's truth-maker), which PROF's payload-grain `( prof:isProfileOf ∘ dct:conformsTo )` chain does not supply — resolved by Davis's PROF-fit spike, not an unproven rip-out.

**Two places I hold against the maximal readings (both on the same axis — going one step past what convention licenses):**

1. **Gandon/Davis: replace `opda:ValidationContext`/`overlaysContext` wholesale with PROF.** I agree PROF is the right external analogue and its load-bearing pieces (`dct:conformsTo`, the `( prof:isProfileOf ∘ dct:conformsTo )` chain, the role registry) are Rec-grade. But **PROF is a W3C Working Group Note, not a Recommendation** — adopting a Note as the spine of a UK national-standard form mechanism is a governance commitment, and `opda:ValidationContext` answers Guarino's S010 "no fixed model theory" objection in a way PROF's profile-as-*document* does not (it reifies "required *relative to* this named context" as a constraint truth-bearer). Allemang agrees VC earns its keep. **Disposition I back (Davis's own):** record PROF as the canonical recommended direction; run a one-page **PROF-fit spike** proving the `baspi5` round-trip survives re-expression; sub-property any residual OPDA-specific facet under PROF terms — *not* an immediate rip-out.

2. **Davis: cut `servesContext` entirely on YAGNI; and "the SKOS scheme may be redundant once each context is a `prof:Profile`."** On the scheme: **no** — a bounded context is a *community of practice*; a profile is a *specification that community publishes*; one community publishes many profiles (Conveyancing owns ta6/ta7/ta10/lpe1). Collapsing context→profile is a category error that breaks the moment a context has two overlays. The six contexts are authored editorial reference data, not derivable from the profiles. Keep the scheme distinct (Allemang concurs). On `servesContext`: I keep the **bare dormant CONSTRUCT** (a view definition costs nothing parked; the first term-grain consumer — "which overlays use this leaf" — is the obvious one) but I move toward Davis on the **S021 scaffolding** (cross-check `sh:Warning` + total-cover CI + F-guards built to police a thing with no consumer) — that is thinnable now.

**Ownership crux — authored-or-absent, never derived (second Allemang exchange).** Allemang's corpus tally (`dct:source` = data-dictionary 58× + ODR-section + external-authority; modules are by *ontological concern*: property/agent/claim/transaction/governance/descriptive/classes) yields three orthogonal axes — concern (`rdfs:isDefinedBy`→module), provenance (`dct:source`), usage (`servesContext`) — and he then argued context-**home** should not be stored at all but read off `servesContext` *degree* (degree-1 = single-context home). **I hold against the degree-derivation:** it makes ownership a function of usage, which is the precise OntoClean level-confusion S021 was convened to kill (a term required by only `ta6` today is degree-1, but that does not make Conveyancing its conceptual owner — degree flips 1→2 the moment a 7th profile lands, so "home" would silently move; home is by definition the thing that does *not* move when a form moves). The reconciliation, which I believe is the true convergence point: **the S021 home-pass had two parts — (a) a brittle `dct:source`→context *default* [killed: Allemang's catch is correct] and (b) a small *authored* steward residue ("Surveying owns valuation", "Conveyancing owns charge/search") [a genuine, non-derivable editorial fact].** So `dct:subject`→context is **authored-or-absent**: emitted only for the steward-adjudicated residue, NEVER generated from `dct:source` and NEVER inferred from `servesContext` degree; absent for the descriptive majority no steward has ruled on. If even the residue lacks a named consumer, the clean YAGNI floor (Allemang's position) is `rdfs:isDefinedBy` + `dct:source` + derived `servesContext`, no `dct:subject` at all — recorded as minority-acceptable. The one option I vote against is degree-derived home.

**Net minimal idiomatic design (converged, post-rebuttal):** ontological-concern=`rdfs:isDefinedBy`→module (adopt corpus-wide; this is *concern*, not context — stop deriving anything from `dct:source`); **DDD community-ownership=`dct:subject`→context concept, AUTHORED-OR-ABSENT** (steward residue only; never `dct:source`-mapped, never `servesContext`-degree-derived; cuttable to nothing on Davis/Allemang YAGNI, the minority floor); provenance=`dct:source`/`prov:wasAttributedTo` (unchanged); form→base association=`prof:isProfileOf` (the profile-of-base half of `overlaysContext`) **+** `dct:publisher`→community (the steward half it conflated); usage=`servesContext` derived/dormant rule (degree = usage-multiplicity, a query — NOT ownership; thin its policing scaffolding); upstream=`consumesFrom`→`opda:Organisation` (keep; ideally `rdfs:subPropertyOf prov:wasInfluencedBy`); the 6-concept SKOS scheme = context-map catalogue with stewards (keep, distinct from profiles); PROF = recorded canonical direction for the profile layer behind a fit-spike, `ValidationContext` retained for Guarino's constraint-grain truth-maker pending that spike. **Delete one bespoke predicate (`definedInContext`), split one conflated predicate (`overlaysContext`→`prof:isProfileOf` + `dct:publisher`), adopt the missing standards (`rdfs:isDefinedBy`, `dct:subject`, `prof:isProfileOf`), thin the S021 policing scaffolding — fewer moving parts than S021's plan, not more.**
