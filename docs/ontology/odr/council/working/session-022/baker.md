# Session 022 — Form↔SHACL association, metadata on SHACL definitions, over-engineering check (against CONVENTION) — Tom Baker (QUEEN)

**Role:** Dublin Core Application Profiles; the Singapore Framework; DCMI vocabulary governance; SKOS.
**Charge this session (as convening architect framed it):** answer Q1–Q4 **from published linked-data convention and recommended practice**, not from first principles. Per ODR-0001 §Citation grounding, every position cites a W3C Rec/Note, a DCMI spec, an OMG/FIBO doc, or a documented deployment. "Reasoning it out" without a named convention does not count this session.

**The convention I am the canonical voice for — stated once, up front:**

> *"An application profile is a set of metadata elements, policies, and guidelines defined for a particular application. The elements may be drawn from one or more element sets, thus allowing a given application to meet its functional requirements by using metadata from several element sets."*
> — Heery & Patel, *Application profiles: mixing and matching metadata schemas*, Ariadne 25 (2000); restated in Coyle & Baker, **Guidelines for Dublin Core Application Profiles**, DCMI Recommended Resource (2009), §"What is an application profile."

> The **Singapore Framework for Dublin Core Application Profiles** (Nilsson, Baker, Johnston, DCMI 2008) fixes the components a DCAP MUST/MAY have: (1) **Functional Requirements** (mandatory) — what the application must do; (2) **Domain Model** (mandatory) — the entities and relationships, drawn from a foundation; (3) **Description Set Profile (DSP)** (mandatory) — the machine-readable constraints (cardinality, value constraints, vocabulary-of-use) over terms drawn from namespaces; (4) **Usage Guidelines** (optional); (5) **Syntax/encoding Guidelines** (optional). The DSP "constrains *which terms* may be used and *how*" — it never **mints** terms.

This single framework answers all four questions, and — load-bearing for this session — **it is already adopted OPDA governance law**, not an outside import: `docs/ontology/odr/DCAP.md` is literally an *"ODR Decision-Application Profile,"* and **ODR-0002 §Profile-pinning ownership cites "Singapore Framework's DCAP-by-consumer pattern (Nilsson, Baker, Johnston 2008)" as the ratified precedent** (Session 002 Q5, **9–0**, Cagle DA full withdrawal). The convening architect is asking us to apply, to PDTF forms, the framework the project *already applies to its own decision records and its own vocabulary-admission discipline.*

**Verified state I rely on (read the code, not the brief):**

- `tools/opda-gen/src/opda_gen/emitters/profiles.py:250-254` — confirmed: `vctx opda:overlaysContext <https://w3id.org/opda/profiles/foundation>` (profile-LAYER IRI, hardcoded), plus `opda:sourcedFrom <BASPI5_FORMS_AUTHORITY>`, `opda:formVersion "5.0.3"`, `dct:source <ODR-0010>`.

## Q1 — FORM↔BASE ASSOCIATION: the convention for representing a "form" as an application profile and associating it with the base ontology + the terms it draws on

**Vote: FOR — opda's `ValidationContext` / `overlaysContext` / `requires` IS the idiomatic DCAP/Singapore Description-Set-Profile construct, in everything but the *name and the namespace of two predicates*. It does NOT need to be reshaped into a new standard construct; it should be made standard-*labelled* (typed and linked to the convention) and have its base-association predicate fixed.** Named convention: **Singapore Framework (DCMI 2008), §Description Set Profile; Coyle & Baker, Guidelines for DCAPs (DCMI 2009).**

**Argument.**

(1) *A PDTF form IS a Dublin Core Application Profile — definitionally, not analogically.* Map opda's overlay onto the Singapore Framework's five components and the fit is exact:

| Singapore Framework component | opda overlay realisation | Status |
|---|---|---|
| **Functional Requirements** (mandatory) | "validate + re-generate the BASPI5 form" (ODR-0010 §Q7 round-trip) | present (prose) |
| **Domain Model** (mandatory) | the fixed `opda:` TBox the overlay targets (ODR-0008 declare-once) | present, shared |
| **Description Set Profile** (mandatory) | the SHACL shapes: `sh:minCount`, merged `sh:in`, `sh:xone`, `dash:` UI hints, `dct:source` | **present — this is what `opda:ValidationContext` reifies** |
| **Usage Guidelines** (optional) | the `dash:` editor/viewer/group rendering hints | present |
| **Syntax Guidelines** (optional) | Turtle/SHACL serialisation | present |

The overlay is a **complete, well-formed DCAP**. ODR-0010's core decision — *"overlays are named, dereferenceable SHACL profile graphs over a fixed TBox; they constrain, they do not declare"* — is **verbatim the Singapore Framework's DSP definition**: a DSP "constrains which terms may be used and how" over "terms drawn from namespaces," and (Coyle & Baker 2009, §"Application profiles do not declare new terms") an application profile **MUST NOT mint new terms** — terms are *declared* in their owning namespace and *reused/constrained* in the profile. So the architect's framing is correct: **the SHACL profile is first-class authored RDF (a DCAP), not a lesser "form layer."** ODR-0010 already got this right by published convention; it simply did not *name* the convention (it cited SHACL/DASH mechanics, not DCAP/Singapore).

(2) *The association predicate: convention says a profile points at the base it profiles. opda has the right shape, the wrong target.* The Singapore Framework's DSP is *"a profile **of** a domain model"*; W3C **PROF** (Profiles Vocabulary, W3C Working Group Note, 2019) gives this the modern machine-readable predicate: **`prof:isProfileOf`** ("the base specification or specifications that this profile restricts/extends"), and **`prof:ResourceDescriptor` / `prof:hasResource` with `prof:hasRole` `role:constraints`** for the constraint artefact. opda's `opda:overlaysContext` is *exactly this association predicate* — but it is **mis-targeted** at a profile-LAYER IRI (`…/profiles/foundation`, confirmed profiles.py:250) instead of at what is actually being profiled. **Convention adjudication:** `overlaysContext` should point at the **base specification the profile is a profile of** — and per PROF that is the OPDA core ontology / the relevant domain-model module, expressible idiomatically as `prof:isProfileOf <opda-foundation>`. The S020/ADR-0026 decision to re-point it at an *industry-context concept* is **not** what the convention prescribes for the *profile→base* association — a DCAP is a profile of a *base vocabulary/domain model*, not "of a community." (Which-community is Q3; it is a different relation and must not be conflated with profile-of.) **This is the single most important convention correction this session surfaces** (see Q4).

(3) *The terms-drawn-from association: convention = the DSP enumerates the terms it constrains; opda's `opda:requires` is exactly that.* Coyle & Baker §"Description Set Profiles" and DCMI's DSP model (Nilsson, *Description Set Profiles: A constraint language for Dublin Core Application Profiles*, DCMI Working Draft 2008) define a DSP as a set of **Description Templates → Statement Templates**, each Statement Template naming the **property it constrains** and its value/cardinality constraints. opda's `opda:requires <term>` + the per-shape `sh:path <term>` + cardinality **IS the Statement-Template enumeration.** So "which terms a form draws on" is **already** carried, idiomatically, two ways: structurally (every `sh:property [ sh:path … ]`) and as a digest (`opda:requires`). The architect's Claim — *"which terms a form draws on already lives in the profile"* — is **correct and convention-backed**: it is the DSP's Statement Templates. Nothing new is needed to record term-usage; it is the DSP.

(4) *Is conneg-by-profile relevant?* **Tangentially — it governs *delivery*, not *modelling*.** W3C *Content Negotiation by Profile* (W3C Working Group Note, 2019) standardises how a client requests "give me this resource **conforming to** profile X" via the `Accept-Profile` HTTP header and `Link rel="profile"`. It presupposes that each profile has a **token URI** (the profile is dereferenceable and identified) — which opda satisfies (`…/profiles/baspi5/` must dereference, ODR-0010 §Enforcement). It is **not** a reason to change the modelling, but it *is* a reason to keep each `ValidationContext` a **dereferenceable, individually-identified resource** (which ODR-0010 already requires) so a future API can serve "the transaction conforming to the BASPI5 profile." Record as forward-compatible, not as a change.

**ADJUDICATION Q1:** opda is **idiomatic** — a PDTF form is a DCAP, and the SHACL shapes + `opda:requires` are its DSP. Convention-alignments, not a reshape: **(a)** the profile needs a first-class, identified, conneg-able profile *resource* (Singapore Framework; conneg-by-profile needs a token) — and per PROF that is **`prof:Profile` (⊑ dct:Standard ⊑ dcat:Resource)**; **(b)** fix `overlaysContext` to point at the **base/domain-model** it profiles via `prof:isProfileOf`, not a layer IRI and not a community concept (the community is Q3).

**Knublauch concession (cross-talk — sharpens my opening).** I had kept the bespoke `opda:ValidationContext` individual (typed as a profile). Knublauch pushed: a *separate* `ValidationContext` node is largely redundant with the profile graph's own `owl:Ontology` header + `sh:targetClass`. **He is right, and the convention-clean resolution is to FOLD `ValidationContext` into the shapes-graph `owl:Ontology` node and type THAT `prof:Profile`** — then `prof:isProfileOf <base>` (the corrected `overlaysContext`), `prof:hasResource`/`ResourceDescriptor`/`hasRole` (the SHACL/DASH artefacts), and `opda:requires` (the term digest) all hang off the one ontology node, while `sh:targetClass` does the per-shape base-binding. This reconciles three views at once: the **architect's** "the profile *graph* is first-class" (yes — its `owl:Ontology` node IS the profile), **Knublauch's** "`ValidationContext` is redundant" (yes — collapse it), and the **DCMI/Singapore** requirement that a DCAP be a typed, identified resource (yes — `prof:Profile` on the ontology node). Critically, **Guarino's ODR-0010 accepted-withdrawal condition is preserved** — a `sh:minCount 1` remains "a constraint *of a named context*" because the `owl:Ontology`/`prof:Profile` node IS that named context; we simply stop minting a *second* node beside it. **Net: bespoke `opda:ValidationContext` individual → RETIRED; shapes-graph `owl:Ontology` node → typed `prof:Profile`.**

**Davis (DA) confirmation — PROF discharges the exact S010 reservation + carries the residual predicates.** Davis ran the invention test against the live TTL and the PROF spec and confirmed the fit, adding two load-bearing facts. **(i)** PROF's `dct:conformsTo` **property-chain axiom `( prof:isProfileOf dct:conformsTo )`** supplies the *conformance inheritance* that `opda:ValidationContext` was minted at S010 to give Guarino — so PROF does not merely match the wrapper, it **discharges the precise reservation the bespoke term was invented for.** **(ii)** The two remaining bespoke profile predicates carry over to standards: `opda:formVersion "5.0.3"` → **`owl:versionInfo`** (or `dct:hasVersion` / `prof:hasToken`); `opda:sourcedFrom <forms-authority>` → **`prov:wasAttributedTo`** + `dct:source`. Both vocabularies are already in ODR-0002's catalogue (DCT Core; PROV-O Conditional; DCAT Conditional) — this is the *boring continuation of decisions already made*, not a new dependency. Davis's decisive "tell" on Q3 (quoted in synthesis): **S021 already reaches for `rdfs:isDefinedBy` to record where a *predicate* is defined, while minting a new term to record where a *term* is defined — "same arrow, different target."** One caveat I broke in his spike: PROF's `prof:Profile` "within a community of users" is the profile's **audience** (`dct:audience`), NOT a term-ownership axis — so PROF retires the *profile↔base* predicates but does **not** retire the SKOS scheme or answer *term→community* (that stays `dct:subject`, gated).

## Q2 — METADATA ON A SHACL DEFINITION: what a profile/shape legitimately carries, and what does NOT belong on a shape

**Vote: FOR the DCAP/PROF metadata set below; AGAINST loading *domain-ownership* assertions onto a shape (that is not profile metadata — it is domain-model metadata, Q3).** Named convention: **DCMI Metadata Terms; W3C PROF §"Describing a Profile"; SHACL Core §"sh:declare / sh:prefixes" + W3C Data Catalog conventions (Coyle & Baker §"Administrative metadata").**

**Argument — convention enumerates exactly what a profile/DCAP records:**

| Metadata facet | Convention-prescribed predicate | On the shape/profile? | opda today |
|---|---|---|---|
| **Identification** (the profile is a thing with a URI, label, definition) | `dct:title`, `rdfs:label`, `dct:description`/`skos:definition`, `dct:identifier` | **YES** — profile-level | partial (has IRI; add title/label) |
| **The base it profiles** | **`prof:isProfileOf`** (PROF); historically Singapore "Domain Model" reference | **YES** — profile-level | mis-targeted (`overlaysContext` → layer IRI) |
| **Role of each artefact** (this graph = the constraints; that = guidance) | `prof:hasResource`/`prof:hasRole` (`role:constraints`, `role:guidance`, `role:vocabulary`) | **YES** — profile-level | implicit (could be made explicit) |
| **Provenance / attribution** (who authored, from what source) | `dct:source`, `prov:wasAttributedTo`, `dct:creator`, `dct:publisher` | **YES** — profile- AND statement-level | **present** — `dct:source` per shape + per ValidationContext (confirmed) |
| **Applicability / community / functional requirement** | `dct:audience`, `dct:conformsTo`, `dcat:keyword`; Singapore "Functional Requirements" prose | **YES — as *applicability*, prose or `dct:audience`** | prose (ODR-0010 §Q7) |
| **Version** | `owl:versionInfo`, `pav:version`, `dct:hasVersion`, `prof:hasToken` | **YES** — profile-level | `opda:formVersion "5.0.3"` (confirmed) |
| **Cardinality / value / vocabulary constraint** | `sh:minCount`, `sh:in`, `sh:datatype`, `sh:xone` | **YES — this IS the DSP** | present (ODR-0010 canonical mapping) |
| **UI rendering hints** | `dash:` (DASH); Usage-Guidelines tier | YES — but ADR-0010/ODR-0013 already exiles non-constraint advisory annotations to a separate graph | present, correctly separated |

**What does NOT belong on a shape (convention is explicit):**

- **Term *definitions*.** Coyle & Baker §"Application profiles do not declare new terms": a profile **MUST NOT** carry `rdfs:domain`/`rdfs:range`/`owl:Class` declarations for the terms it uses — those live in the **owning namespace's TBox**. A shape `sh:path opda:floodRisk` reuses `opda:floodRisk`; it must not *define* it. opda honours this (ODR-0010 graph-separation: "shapes reference classes via targeting, never `owl:imports`"; ODR-0008 declare-once). **Correct.**
- **Identity / keys.** ODR-0010's no-identity-override gate is the convention-correct rule: a DSP constrains *use*, it never restates the *identity criterion* of a Kind (that is domain-model/TBox metadata). **Correct.**
- **🔴 Domain *ownership* ("which community conceptually owns this term").** This is the crux for Q3/Q4. **Ownership is NOT profile metadata and NOT shape metadata.** The Singapore Framework places the entities-and-their-origin in the **Domain Model** component, *not* the DSP. A DSP records *"BASPI5 uses `opda:floodRisk` with minCount 1, sourced from form-question B-whatever"* — it does **not** record *"`opda:floodRisk` is conceptually owned by Surveying."* The latter is a fact about the **term in the domain model**, carried by the term's own **vocabulary-of-origin / provenance** metadata (Q3). Putting `definedInContext`-style ownership onto a `ValidationContext` (the profile) would be the category error the convention forbids: **mixing Domain-Model metadata into the DSP.**

**ADJUDICATION Q2:** opda's per-shape and per-profile metadata is **convention-idiomatic and already substantially present** — crucially **`dct:source` provenance is already carried per term**. The legitimate additions are profile-*identification* and the *base-it-profiles* link (`prof:isProfileOf`), both PROF-standard. The thing that must **NOT** migrate onto the shape/profile is **conceptual ownership** — that belongs to the term's vocabulary-of-origin metadata in the domain model (Q3).

## Q3 — "WHICH CONTEXT/COMMUNITY A TERM BELONGS TO": bespoke SKOS scheme + `servesContext` + `definedInContext`, or a standard construct? Is `definedInContext` reinventing provenance?

This is the heart of the architect's pushback and my central adjudication. **Split vote, by sub-question:**

### Q3a — the bounded-context SKOS ConceptScheme itself

**Vote: FOR keeping it — a `skos:ConceptScheme` of communities-of-practice is idiomatic SKOS, and a DCAP's "community" is exactly what SKOS organises.** Named convention: **SKOS Reference (W3C Rec 2009) §3 (`skos:ConceptScheme`); Coyle & Baker §"Vocabulary encoding schemes"; DCMI `dct:audience`.**

The six bounded contexts are a controlled vocabulary of *communities/audiences*. SKOS Reference §3.1 is precisely "a set of concepts, optionally including statements about semantic relationships" used for "thesauri, classification schemes, subject heading systems." A vocabulary of "the communities of practice in UK property transactions" **is** such a scheme. There is no over-engineering in modelling it as `skos:ConceptScheme` + six `skos:Concept`s — that is the *minimal* idiomatic SKOS encoding, and it mirrors opda's own 23 value-schemes (ODR-0011 house style). **Keep.** (This is the part of the architect's "bespoke scheme" worry that is **unfounded** — a SKOS scheme is not bespoke; it is the W3C-standard way to do controlled vocabularies.)

### Q3b — `opda:servesContext` (which contexts USE a term) — stored vs rule-only

**Vote: FOR rule-only / derived (NOT stored as authored data) — and convention AGREES it is redundant-as-stored.** Named convention: **Singapore Framework DSP (term-usage lives in the profile); SKOS Primer §"inScheme is not transitive" (membership is not a logical relation to over-assert); DCMI "one fact, one home" Usage-Board discipline (Baker, Bechhofer, Isaac, Miles, *Key choices in the design of SKOS*, J. Web Semantics 20 (2013)).**

The architect is **right**: "which contexts use a term" is a **2-hop query over the DSPs** (`?vc prof:isProfileOf/opda:overlaysContext ?base ; opda:requires ?term` joined to the profile's community). By the Singapore Framework, *term-usage is a fact recorded by the DSP* (Q1.3 — the Statement Templates). Materialising it back onto the term as a stored `servesContext` triple **duplicates the DSP into the domain model** — the exact "second source of truth that drifts" DCMI's one-fact-one-home discipline forbids, and which ODR-0019 Alt "hand-authored membership" + ODR-0020 Rule 5 "derive-don't-declare" already reject. **So membership as a *derived view* (SHACL-AF CONSTRUCT, dormant) is convention-fine; membership as *stored authored data* would be redundant — and S021 already ruled it derived.** S021's "servesContext = rule-only" verdict survives the convention test.

**Knublauch refinement (cross-talk — retires the `servesContext` *predicate*, not just stored-vs-derived).** Knublauch confirmed the dormant CONSTRUCT is the correct SHACL-AF use (a `sh:rule` materialising a derived view; persisting it as ground truth is the denormalisation anti-pattern) and added the sharp move I am ruling FOR: **the CONSTRUCT's OUTPUT predicate should also be `dct:subject` (term→community concept), NOT the bespoke `opda:servesContext`.** Then *derived* membership and the *authored* gated residue share ONE standard vocabulary, distinguished by **provenance** (generated graph vs source TTL), not by a different predicate. So `opda:servesContext` as a coined predicate **also retires** — the derived view emits `dct:subject` triples. The F3 guard becomes "no hand-authored term→community `dct:subject` in *source* beyond the gated residue; the bulk appears only in the generated/derived graph." (Gandon's PROF `dct:conformsTo` chain answers the orthogonal *payload-level* "what context is this DATA in" with no per-term index — complementary, different arrow.) **Citation strengthened:** DCMI's current tabular application-profile form, **DCTAP**, *compiles to SHACL* — DCMI itself treats SHACL as the DSP runtime (cleaner than "DSP stalled → SHACL realised it").

### Q3c — `opda:definedInContext` (a term's "home") — **IS it reinventing provenance?**

**Vote: AGAINST `opda:definedInContext` as a *new bespoke ownership predicate* — convention ALREADY provides vocabulary-of-origin, and `definedInContext` largely reinvents it. REPLACE it with the standard constructs.** Named convention: **RDFS §2.6 `rdfs:isDefinedBy` (W3C Rec — "the resource defining the subject resource"); DCMI `dct:source`; W3C PROV-O §"wasAttributedTo"/"wasDerivedFrom"; Coyle & Baker §"terms are drawn from namespaces" (vocabulary-of-origin).**

This is where I most strongly side with the architect against S021's own prior synthesis (which I, Baker, partly dissented from at 6–3). The convention is unambiguous:

1. **Vocabulary-of-origin is `rdfs:isDefinedBy` + the namespace.** RDFS §2.6: `rdfs:isDefinedBy` is "a sub-property of `rdfs:seeAlso`… to indicate **the resource defining the subject resource**." This is *the* standard "where does this term come from / who defines it" predicate. The Linked Data convention (and FIBO's modular methodology, which ODR-0019 itself cites) is **namespace-as-definition-home + `rdfs:isDefinedBy` → the owning module/ontology**. A term's *home* is **the ontology module that defines it**, full stop.

2. **Provenance-of-conception is `dct:source` / `prov:wasAttributedTo` / `prov:wasDerivedFrom`.** And — **this is the killer fact** — opda **already carries `dct:source` per term** (verified: every shape, and the descriptive layer is designed to). S021's own resolution *admitted this*: it ruled `definedInContext` must be **"generated FROM each term's `dct:source` provenance."** Read that plainly: **S021 conceded that the home is a deterministic function of `dct:source`.** If `definedInContext` is *generated from* `dct:source`, then `definedInContext` carries **no information `dct:source` does not already carry** — it is a *materialised projection* of provenance, exactly as `servesContext` is a materialised projection of the DSP. By DCMI's one-fact-one-home discipline, **you do not store a second predicate that is a pure function of an existing one.** You either (a) query `dct:source` directly, or (b) if you want the *context-resolved* answer, derive it with a rule — never hand-mint a parallel authored predicate.

3. **So is `definedInContext` "reinventing provenance"? — Largely YES.** Three of its claimed jobs are already standard:
   - "who defines/owns this term" → **`rdfs:isDefinedBy` → the owning `opda:` module** (vocabulary-of-origin; FIBO precedent ODR-0019 cites).
   - "what real-world source/authority grounds it" → **`dct:source`** (already emitted) + **`prov:wasAttributedTo`** for an organisation.
   - "which upstream authority do we conform to" → **`opda:consumesFrom` → `opda:Organisation`** paired with `dct:source`/`prov:wasAttributedTo` (ODR-0020 Rule 2 — this one IS a justified mint: the DDD *Conformist relationship* has no W3C predicate; PROV's `prov:wasInfluencedBy` is too broad. **Keep `consumesFrom`.**).

   What remains that *might* be irreducible is the narrow case ODR-0019 Rule 4(i) names: a genuine **homonym** where the *community-scoped* home of one local name differs and is **not** recoverable from `dct:source` alone. But the corpus attests **zero** such homonyms (verified S021). So the *residue that `definedInContext` uniquely serves is presently empty.*

4. **The DCMI orthogonal-axes point I made in S021 still holds — and it now cuts AGAINST a stored home predicate.** Module-of-definition (1:1, authored where the class is minted) and context-of-use (M:N, derived from DSPs) are different axes (Baker et al. 2013, *Key choices in SKOS*). **Module-of-definition is already encoded** — by the namespace + the TTL module the term lives in + `rdfs:isDefinedBy`. So a term's *home* is **not missing data requiring a new predicate**; it is **already carried by which module mints it**. `definedInContext` re-states, as a hand-curated-then-"generated" annotation, a fact the module structure + `rdfs:isDefinedBy` + `dct:source` already carry. That is the over-engineering (Q4).

**The three-axes decomposition (sharpened in cross-talk with Gandon — verified against the code).** My opening compressed "home" into `rdfs:isDefinedBy` + `dct:source`. Gandon correctly exposed that `definedInContext` bundles **three orthogonal facts**, each with its own standard predicate — and since *all* opda terms share one namespace (ODR-0004), the namespace alone cannot carry community, which is precisely why a vocabulary predicate (`rdfs:isDefinedBy`) is the wrong tool for the community axis:

| Axis | Standard predicate | Target | Verified state |
|---|---|---|---|
| **Module-of-definition** (ontological *concern*) | **`rdfs:isDefinedBy`** (RDFS §2.6) | the owning **module `owl:Ontology` IRI** | each TTL **is** its own `owl:Ontology` (foundation, opda-agent, opda-claim, opda-property, opda-transaction, opda-descriptive, opda-governance…) — so `isDefinedBy` is available; **emitted on ZERO terms today** (a real, mechanical gap). NB it discriminates *concern*, NOT the six communities. |
| **Provenance / source** | **`dct:source`** (+ `prov:wasAttributedTo`) | the form-question / authority IRI | **present** per shape (verified profiles.py) |
| **Community-ownership** (DDD home) | **`dct:subject`** → the community `skos:Concept` (DCMI topical/aboutness) | a bounded-context concept | **already named by ODR-0019 Rule 3** ("hand-authored `dct:subject` membership… for a term no profile references") + catalogued in ODR-0019 §References — the predicate S021's `definedInContext` *shadowed* |

`opda:definedInContext` collapses all three into one coined term — **that is the precise form of the over-engineering.** None of the three needs a new predicate.

**DCMI target ruling (in answer to Allemang's cross-talk question).** Allemang verified `rdfs:isDefinedBy` is used **zero** times across all 23 emitted TTLs while `dct:source` is on essentially every term (33 in opda-property.ttl alone), and asked whether `rdfs:isDefinedBy → a skos:Concept` is a defensible target or whether `isDefinedBy` classically demands a vocabulary/ontology *document* IRI. **My DCMI/RDFS ruling: `isDefinedBy → a skos:Concept` is a category mismatch.** RDFS §2.6: `rdfs:isDefinedBy` is a sub-property of `rdfs:seeAlso` whose object is "the resource **defining** the subject resource" — the vocabulary that mints/axiomatizes the term, i.e. the module `owl:Ontology` IRI. A `skos:Concept` is a subject/aboutness node, not a defining resource; `isDefinedBy → community-concept` would falsely assert the community *defines* the term (the module does). So the community axis uses **`dct:subject`** (aboutness), and **`rdfs:isDefinedBy` targets the module document** — never the context concept. This corroborates the architect and is independently consistent with Gandon's `dct:subject` proposal.

**ADJUDICATION Q3:**
- **Scheme (Q3a): KEEP** — idiomatic SKOS.
- **`servesContext` (Q3b): KEEP AS DERIVED RULE** — convention agrees stored is redundant; S021's rule-only verdict stands.
- **`definedInContext` (Q3c): RETIRE as a bespoke authored predicate; DECOMPOSE into the three standard predicates above** — `rdfs:isDefinedBy` (concern-module, emit mechanically), `dct:source`/`prov:wasAttributedTo` (already present), and — *only where stored community-ownership is actually wanted* — `dct:subject` → the community concept (the predicate ODR-0019 Rule 3 already names). **Community-ownership is itself largely derivable** (`servesContext` answers "which communities use it"; a single-community term's home is recoverable), so even `dct:subject` is **gated** to the non-derivable residue (genuine homonym / disputed ownership), which is **empty today** (Rule 8 doing its job — mint nothing until a named case exists). This **reverses** the S021 "un-gate `definedInContext`, generate-from-provenance for every term" decision: convention says *if it is generated from `dct:source`, don't store it as a separate predicate at all — emit `rdfs:isDefinedBy` to the module, read provenance from `dct:source`, and store community via `dct:subject` only where it is not derivable.*

## Q4 — OVER-ENGINEERING VERDICT + MINIMAL IDIOMATIC DESIGN

**Vote: YES, S021 over-engineered ONE thing (the always-emitted `definedInContext` home-pass) — and the fix is to delete it, not to keep it. The rest of the design is idiomatic and stays. This DOES change S021's verdict and the ODR-0019/0020 + ADR-0026/0028/0029 amendments.** Named convention: **the Singapore Framework's component separation (Domain Model vs DSP); DCMI one-fact-one-home (Baker et al. 2013); RDFS `rdfs:isDefinedBy`; the YAGNI/named-consumer discipline opda already ratified as ODR-0019 Rule 8.**

**The over-engineering, named precisely.** S021 reasoned — soundly, but from DDD-ownership + UFO-identity *first principles* — that "home" and "usage" are different relations, therefore a term needs an *authored home predicate* (`definedInContext`) distinct from the *derived usage* (`servesContext`). The convention agrees home ≠ usage. **But convention then says: the home is already recorded** — by the **namespace + owning module + `rdfs:isDefinedBy`** (vocabulary-of-origin) and **`dct:source`/`prov:` provenance**. S021 even conceded `definedInContext` must be *generated from* `dct:source`. A predicate that (a) duplicates information `dct:source`/`rdfs:isDefinedBy` already carry, (b) is generated, not observed, and (c) serves an **empty** residue of genuine homonyms, is **textbook YAGNI over-engineering** — and it violates DCMI's one-fact-one-home discipline (don't store a pure function of an existing fact). The architect's instinct is **correct**.

**What is NOT over-engineered (keep — all idiomatic):**
- The **SHACL overlay = DCAP/DSP** (Q1) — verbatim Singapore Framework. Keep.
- **`opda:requires`** as the DSP Statement-Template digest (Q1.3). Keep.
- The **bounded-context SKOS scheme** (Q3a). Keep — standard SKOS.
- **`servesContext` as a derived/dormant CONSTRUCT** (Q3b). Keep — convention agrees stored is redundant; derived is fine.
- **`consumesFrom` → `opda:Organisation`** for upstream conformist sources (Q3c). Keep — justified mint, no W3C equivalent for the DDD Conformist relationship.
- **Per-shape `dct:source`** provenance. Keep — already convention-correct and already present.
- The **firewall CI guards** (F1/F2/F3) I specified in S021. Keep — they enforce the Domain-Model/DSP separation the Singapore Framework demands.

**The MINIMAL IDIOMATIC DESIGN (convention-grounded):**

**Concrete keep / simplify / reshape — and the amendment impact:**

  - **ODR-0019 amendment:** Rule 5 (`definedInContext`) is **withdrawn**; vocabulary-of-origin is `rdfs:isDefinedBy` + namespace. Rule 8's gate is **restored in full** for any community-scoped home (no carve-out) — there is no always-emitted home layer.

- **`opda:overlaysContext` → RESHAPE the target.** It is the profile→base association (Q1.2). The form↔community association, if wanted as stored metadata, is a **separate** predicate (`dct:audience` → the community `skos:Concept`) on the profile — but per Q3b that community-usage is itself derivable, so even this is optional.
  - **ADR-0026 amendment:** the `CONTEXT_OF` map re-pointing `overlaysContext` at `…Context` concepts is **wrong by convention**; `overlaysContext` → base (`prof:isProfileOf`), and the profile's *community* (if stored at all) is `dct:audience`. **Does this change the S021 verdict? — YES, on exactly one axis, and it is the axis the architect named.** S021 Q1's 6–3 sub-vote (un-gate `definedInContext`, generate for every term) is **overturned by convention**: the home is vocabulary-of-origin (`rdfs:isDefinedBy` + namespace) + provenance (`dct:source`), both standard, one already present. S021's *other* findings — servesContext derived, scheme authored, consumesFrom for upstream, firewall to CI, walk is mechanical, profiles need a generic composer — **all survive** because they were already convention-aligned (I and Allemang/Davis grounded them in DCMI/derive-don't-declare in S021). **The architect was right that S021 invented a bespoke ownership predicate where convention already had one; the architect was over-broad in suspecting the whole scheme + servesContext + the DCAP framing — those are idiomatic.**

## Queen's synthesis (Tom Baker) — convention verdict + emerging consensus

*Composed after one opening + one rebuttal pass. Four of five panellists reported (Gandon, Allemang, Knublauch, Davis); Kendall (FIBO) pending — her position is NOT pre-supposed here per ODR-0001 §9 (the Queen does not fabricate). Every quotation traces to the named teammate's message in this session.*

**The architect's instinct is vindicated by convention — with one precise scope correction.** The convening charge asked whether S021 invented a bespoke design where established linked-data convention already exists. **It did, in two places, and the panel is unanimous (4/4 reporting) on both:**

1. **The profile↔base machinery (`opda:ValidationContext` + `opda:overlaysContext`) reinvents W3C PROF.** Gandon: *"opda's `opda:ValidationContext`/`opda:overlaysContext` reinvents the weaker half of PROF"* — `prof:isProfileOf` is the form→base predicate; `prof:hasResource`/`ResourceDescriptor`/`hasRole` bundle the SHACL/DASH artefacts; `prof:Profile ⊑ dcat:Resource`. Davis ran the invention test against the live TTL and found PROF's **`dct:conformsTo` property-chain `( prof:isProfileOf dct:conformsTo )` supplies the conformance inheritance `ValidationContext` was minted at S010 to give Guarino** — PROF *discharges the very reservation the bespoke term was invented for*. Knublauch: a *separate* `ValidationContext` node is *"largely redundant with the profile graph's own `owl:Ontology` node + `sh:targetClass`."* **Resolution (settled): retire the bespoke `ValidationContext` individual; type the shapes-graph `owl:Ontology` node `prof:Profile`; `overlaysContext` → `prof:isProfileOf` the base. `formVersion` → `owl:versionInfo`; `sourcedFrom` → `prov:wasAttributedTo` + `dct:source`.** PROF + DCAT are already in ODR-0002's catalogue — *"the boring continuation of decisions already made"* (Davis).

2. **`opda:definedInContext` reinvents provenance + vocabulary-of-origin.** Davis's one-line "tell": *"S021 already reaches for `rdfs:isDefinedBy` to say where a PREDICATE is defined, while minting a new term to say where a TERM is defined — same arrow, different target."* Allemang verified `rdfs:isDefinedBy` is used **zero** times across the 23 TTLs while `dct:source` is on essentially every term, and that ADR-0028 *generates* `definedInContext` *from* `dct:source` — so by derive-don't-declare it is *"a view of provenance, not an authored fact."* **Resolution (settled): `definedInContext` decomposes into three standard predicates on three orthogonal axes** — `rdfs:isDefinedBy → module owl:Ontology` (concern-of-origin; a *defining document*, NOT a `skos:Concept` — that target is a category mismatch, my DCMI ruling to Allemang), `dct:source`/`prov:wasAttributedTo` (provenance, present), and **`dct:subject → the community `skos:Concept`** (DDD community-ownership — the predicate ODR-0019 Rule 3 *already names* and that `definedInContext` shadowed). Community-ownership is largely **derivable**, so `dct:subject` is authored only for the non-derivable residue (genuine homonym / disputed ownership) — **empty today**.

**What convention VINDICATES as already-idiomatic (the architect was over-broad in suspecting these):**
- **A PDTF form IS a Dublin Core Application Profile** (Singapore Framework) and **the SHACL overlay IS its Description Set Profile.** Knublauch confirmed as the original DSP author: *"SHACL is the de-facto successor to the DCMI DSP constraint language; a PDTF form = a SHACL profile = a DCAP, and modelling forms this way is the IDIOMATIC way, NOT over-engineering"* — DCMI's current **DCTAP compiles to SHACL** because SHACL is the runtime; TopBraid/EDG models a profile *"line-for-line what baspi5.ttl emits."* ODR-0010's "overlays constrain, they do not declare" is verbatim the Singapore-Framework DSP definition. **The shapes layer is kept verbatim.**
- **The six-context SKOS `ConceptScheme` is idiomatic SKOS** (SKOS Reference §3) — authored reference data like opda's 23 value-schemes. Gandon, Allemang and I converge that the *scheme* is sound; the objection is *only* to the coined predicates. Allemang explicitly *"DEFENDING it as sound authored reference data — not the thing to cut."*
- **`opda:consumesFrom`** (the DDD Conformist relationship → `opda:Organisation`) is the one justified `opda:`-mint — no W3C predicate names the architectural Conformist relationship (`prov:wasInfluencedBy` is too broad).

**The PROF↔DCAP relationship (Gandon's seam, resolved):** complementary, not competing. *"The Singapore Framework gives the conceptual layers (Functional Requirements → Domain Model → DSP → guidelines); PROF gives the RDF-native packaging (each layer a `prof:ResourceDescriptor` with a registry role); DCAT supplies the class lineage"* (Gandon). PROF is *how you publish a DCAP as linked data*. One caveat I broke in Davis's spike: PROF's `prof:Profile` "within a community of users" is the profile's **audience** (`dct:audience`), NOT a term-ownership axis — so PROF retires the profile↔base predicates but does **not** retire the SKOS scheme or answer term→community (that stays `dct:subject`).

**Knublauch's unifying refinement (ruled FOR):** the dormant SHACL-AF CONSTRUCT's **output predicate is also `dct:subject`** — derived membership and the authored gated residue share ONE standard vocabulary, distinguished by *provenance* (generated graph vs source TTL), not by a different predicate. So **`opda:servesContext` as a predicate also retires.**

**Net bespoke-predicate verdict:** `opda:ValidationContext` → `owl:Ontology`+`prof:Profile`; `opda:overlaysContext` → `prof:isProfileOf`; `opda:definedInContext` → `rdfs:isDefinedBy` + `dct:source` + `dct:subject`; `opda:servesContext` → `dct:subject` (derived). **Only `opda:consumesFrom` survives.** The SKOS scheme, the SHACL shapes, the 935-leaf walk, and the 14-DCAP rollout all stand.
