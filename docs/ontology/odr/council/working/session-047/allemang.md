# Allemang — Linked Data Council session 047 (ODR-0032 relationship layer)

## OPENING — Allemang

**Allemang (Q1 — framing / un-freeze):** AFFIRM, ballot **FOR**. This is implementation-gap closure, not fresh modelling. ODR-0006 §"Role-founding relator pattern" / §SHACL and ODR-0007 §Rules already ratified `plays`/`founds`/`mediates`, the Person/Org→Address join, `opda:Name`, and the chain relation; they were simply never emitted (ODR-0032 §Context — "only 7 object properties actually connect two `opda:` classes"). Re-deciding settled design would violate "reuse-before-mint" applied to *decisions* — the cheapest correct move is to build what the council already accepted (SWWO 3rd ed. ch. 3, on letting the model say what the domain needs and no more). The un-freeze is cleanly separable: ODR-0007's freeze gate is already CLEARED (§Rules "Freeze gate — CLEARED 2026-05-31"), and ODR-0006's residual freeze is the *W3C-Org-vs-bespoke Kind-layer* choice (§"Freeze gate" (b)) — orthogonal to whether the edges exist as data. §R4's scoped un-freeze is exactly right; do not let the Kind-layer choice gate the relationship layer.

**Allemang (Q2 — §R1 completeness criterion):** **REVISE**, ballot **FOR (conditional on the amendment)**. The principle — every *inter-entity* association reified, no silent drop, gate-enforced — is the correct mirror of the leaf-completeness gate I chaired (ODR-0022 G3). But §R1 as written ("every association ... whether expressed as one entity *containing* another ... MUST be reified") over-reaches by treating **JSON containment as conceptual relationship**. That is the precise error ODR-0022 was convened to refute: PDTF is "correct-for-a-form-transport, wrong-altitude-for-a-TBox" (ODR-0022 §Context); 56% of leaves were "form-ergonomics + repeated micro-structure, not conceptual richness." A `MonetaryAmount` value-node, a `time:Interval`, an `opda:Address` value-object, a SKOS-status node — these are *attribute structure*, not first-class entities, and an edge to them is a datatype/value property, not a navigable inter-*entity* relationship. The line is **identity**: reify an association as an object property iff the target is an entity bearing its own identity criterion (a Person, Organisation, Transaction, Property, Title, Proprietorship, Role) — OntoClean's sortal/IC test, which ODR-0022 §1 already used to keep checklist items out of the class tree. **Amendment (see Q2 final):** scope §R1 to associations whose *range is an IC-bearing `opda:` class*; containment to value-objects/quale-regions is explicitly out, recorded in the residue register (ODR-0022 §5), not silently dropped. With that boundary the criterion is sound and I vote FOR.

**Allemang (Q3 — `opda:Name` class vs value):** **REVISE → structured value, not a navigated class.** Ballot **AGAINST a full `owl:Class opda:Name` + `hasName` object-property treatment; FOR emitting it as a structured value-node.** ODR-0006 §"Kind layer" says `opda:Name` is "declared **once** here" as a *structured datatype* (the class diagram lists `name : opda:Name` as an attribute, alongside `address : opda:Address`). A person's name has no independent identity, no lifecycle, is never queried-to as an endpoint ("who is party to X?" never resolves to a Name) — it is the textbook case for *not* minting a class (SWWO 3rd ed. ch. 11 on when a structure earns class-hood vs staying a value; "a little semantics goes a long way," Hendler 2003). Reification earns its keep when you need to attach further assertions to the relationship/structure or query it as a node (SWWO ch. 10) — `opda:Name` clears that bar *only* weakly (given/family parts). My amendment: emit `opda:Name` as an `rdfs:Datatype`-shaped value-object (a blank-node or value class with `opda:givenName`/`opda:familyName` datatype properties) reached by `opda:hasName`, **but do not gate it as a §R1 inter-entity relationship** — it is Category-F-style attribute structure (ODR-0022 §1 row F), not a relationship-layer edge. If the WG later shows a query that resolves *to* a Name node, promote then (ODR-0022's "promote only on a named §Q4a query").

**Allemang (Q4 — `playedBy` vs role co-typing):** AFFIRM, ballot **FOR**. Not redundant. Co-typing (`?x a opda:Person, opda:Seller`, ODR-0006 §Q2) and a navigable `playedBy` edge answer *different* queries: co-typing asserts the bearer-and-role are the **same node**; `playedBy` is required exactly when they are **distinct nodes** (a `prov:Agent`-attested participant whose role-instance is reified separately) — ODR-0032 §R3 states this correctly. SWWO ch. 5: domain/range and explicit edges are what make a graph *traversable*; relying on co-typing alone means a consumer cannot follow Seller→bearer without already knowing they are co-typed, which fails the moment provenance reifies the role. And `playedBy` is the SHACL `sh:path` ODR-0006 §SHACL already wrote (`SellerShape sh:path opda:playedBy`) — the shape points at a predicate that must exist as data (this is also why Option D fails, Q5). Emit it.

**Allemang (Q5 — OWL+domain/range vs SHACL-only; rangeless ban):** AFFIRM, ballot **FOR**. Mandate OWL object properties carrying both `rdfs:domain` and `rdfs:range`; reject SHACL-only (Option D); ban rangeless stubs. Two grounded reasons. (1) **The object property is the data; SHACL constrains it** — a `sh:path opda:playedBy` to an *undeclared* predicate is not a navigable edge (ODR-0032 §Option D). The current `founds`/`mediates` rangeless stubs are the live proof: the shape references them but a consumer cannot traverse a typed edge. (2) On rangeless: I accept the SWWO framing that **domain/range INFER, not constrain** (SWWO 3rd ed. ch. 3 — "rdfs:domain/rdfs:range are not constraints; they license inferences"). That is *precisely why* a rangeless object property is defective *as data documentation*: it licenses *nothing* and tells a consumer nothing about what the edge connects. The "design-time, never reasoned" defence conflated *reasoning* with *existence* — you can decline to run a reasoner and still owe consumers a declared range. So: domain+range mandatory, understood as inference-licensing assertions (not OWL restrictions/constraints — those, where needed, are SHACL per ODR-0013). FOR.

**Allemang (Q6 — `opda:Address` reuse):** AFFIRM, ballot **FOR**. Reuse the single `opda:Address`; do not mint participant-specific addresses. ODR-0006 §"Kind layer" mandates it outright ("`opda:Name`, `opda:Address` ... declared **once** here and reused"; Consequences: "Downstream modules MUST consume `opda:Address`"), and ODR-0022 §1 row F bins all address sub-fields as "reuse ODR-0015 (Address) ... never re-mint." This is textbook reuse-before-mint (Singapore Framework; SWWO ch. 3). The only work is extending `opda:hasAddress`'s **domain** to Person/Organisation (currently `Property`-only, ODR-0032 §R2) — a domain widening, not a new property. One caveat I'll surface, not block on: widening an rdfs:domain to a union (Property ∪ Person ∪ Organisation) is an *inference-licensing* widening (anything with `hasAddress` is now inferred to be one of those) — fine here, but it must be a single property with a union/unconstrained domain, never three parallel `personHasAddress`/`orgHasAddress` predicates (that would be the anti-reuse Option this rejects).

**Q1 — Framing / un-freeze. AFFIRM. **Q2 — §R1 completeness criterion. REVISE. Ballot FOR (conditional on the three-limbed amendment).** The principle (every inter-entity association reified, no silent drop, gated) is the right mirror of the ODR-0022 leaf-completeness gate, but §R1 as written over-reaches by treating JSON containment as conceptual relationship — the precise error ODR-0022 was convened to refute ("correct-for-a-form-transport, wrong-altitude-for-a-TBox"). **Amendment (3 limbs):** (a) **over-reify guard (symmetric, per Guarino)** — reify an association as an `owl:ObjectProperty` (domain+range) IFF **both endpoints bear an identity criterion** — each a +I Substance-Kind supplying its own IC (ODR-0005: Person, Organisation, Transaction, Property, Title, Proprietorship), OR a Relator/Role well-founded on such (Seller, Buyer, Proprietor). This is the class-layer OntoClean test (Guarino & Welty 2002; ODR-0022 §1 "no-IC ⇒ not a sortal worth minting") carried to the edge layer — there it kept non-sortals out of the class *tree*, here out of the object-property *graph*. A −I target (MonetaryAmount, time:Interval, SKOS-status — quality/mode/quale-region/Phase) → a `DatatypeProperty`/`sh:in` **VALUE-SLOT**. An endpoint whose IC is genuinely **open upstream** (Address — ODR-0005 §6b deferred Mode-vs-Resource to ODR-0015) → **RESIDUE-PENDING**: emit the join predicate, record the reason, but the gate must NOT manufacture a class to satisfy coverage. Net **three-way disposition: GATED (both +I) | RESIDUE-PENDING (open upstream IC) | VALUE-SLOT (−I datatype)** — nothing silently dropped (ODR-0022 §5). (b) **consumer guard** — RESTORE ODR-0022's G3 coverage-by-test, ported faithfully: each emitted edge must be ratified by **one worked SPARQL competency query that traverses it** (parties-of-transaction traverses `hasParticipant`; address-of-participant traverses `hasAddress`; role-bearer traverses `playedBy`), not merely asserted present. Faithful to G3(b): the query is per *retrieval kind*, not per IRI — a single `hasAddress` query ratifies the edge across both its Person and Org domains (one edge, one query, even with a union domain); no separate query per domain/range pairing. (c) **chain defer** — `dependsOnTransaction`/`chainMembers` are pure ODR-0007 §S007-Q4 comment-ware with no exemplar; deferred-with-recorded-reason until a chain exemplar + its query land. With (a)+(b)+(c) the gate is sound and I vote FOR. **Davis (DA) withdraws his Q2 dissent in full on this amendment.**

**Q3 — `opda:Name` class-vs-value. REVISE → structured value, outside the relationship gate. Ballot AGAINST a navigated `owl:Class opda:Name` gated as a §R1 edge; FOR emitting it as a structured value-node.** ODR-0006 §"Kind layer" declares `opda:Name` "once" as a *structured datatype*; a name has no independent identity/lifecycle and is never a query endpoint, so it fails the class-hood test (SWWO ch. 11; reification earns its keep only when you attach assertions to / query the node, ch. 10). Emit `opda:hasName` → an `opda:Name` value-node with `opda:givenName`/`opda:familyName` datatype properties (optionally `sh:node`-grouped WITHOUT class identity), but treat it as Category-F attribute structure (ODR-0022 §1 row F), NOT a relationship-layer edge for the completeness gate. **Name is also excluded by the Q2 endpoint-IC rule itself** — it is −I (value-determined, no bearer-independent IC), so the Category-F reading and the symmetric-IC test are the same ruling. **Promotion trigger, named (co-signed with Hendler):** promote `opda:Name` to a reified class ONLY on the **addressability trigger** — a concrete need for a stable name-referent a value provably cannot serve (`opda:NameChangeEvent` relating former/successor named nodes, or name-level provenance) — **NEVER on entity-resolution**, which is value-comparison that resolves to the *bearer* (the name being evidence, not the query endpoint) and is served perfectly by the structured value. (Addressability ≠ identity; Guarino's separate name-moment-IC bar is unmet in this domain.) **Corpus-state note (Hendler reports, Cagle-verified — not independently checked by me):** the addressability trigger is already LIVE as the corpus stands — `opda:NameChangeEvent` (`opda-agent.ttl:34`, a SHACL-AF construct) + a deed-poll exemplar — so the EMITTED branch today is the *promoted* one (former/previous-name resolving to `opda:Name` nodes, an ADR-0048 emitter follow-on). This does not change the ruling; it confirms the trigger is real and *discriminating* — it fires exactly where addressability is needed and the default-value-outside-§R1 rule still governs every name *without* such a trigger. (Implementers should verify the cited line against the live tree before relying on it.)

**Q5 — OWL+domain/range vs SHACL-only; rangeless ban. AFFIRM (refined per Davis).
