# Baker — Session 037 verdicts

*Tom Baker (DCMI; co-author of the DCMI namespace policy and the SKOS Reference). Lens: namespace/URI policy, vocabulary governance, metadata-term stewardship, SKOS concept-scheme design.*

---

## Q1 — Org/standard split + base path → **AFFIRM** (FOR)

**Baker:** `https://w3id.org/opda/pdtf/` is the correct base, and it is correct for governance reasons, not merely aesthetic ones. The DCMI namespace policy (DCMI Namespace Policy, 2007) draws a hard line between the *agency that maintains terms* and the *term sets it maintains*: DCMI itself owns `http://purl.org/dc/` and partitions maintained vocabularies beneath it (`/dc/terms/`, `/dc/dcmitype/`, `/dc/dcam/`). That two-level shape — **maintenance authority / maintained vocabulary** — is exactly `opda` (the authority) over `pdtf` (the vocabulary it publishes). A bare `https://w3id.org/opda/` would conflate authority with vocabulary and leave you no room to publish a *second* standard under the same authority without a disruptive migration; a bare `https://w3id.org/pdtf/` would orphan the vocabulary from its steward and forfeit the w3id.org redirect that DCMI's own experience (PURL → w3id) shows is the durable layer. The org/standard two-segment base is the most-defensible, least-regret choice. I would record one amendment as a non-blocking note: the WG should reserve `https://w3id.org/opda/` as a *registry* landing page (an index of standards opda stewards) so the authority IRI is itself dereferenceable, per the DCMI practice of making the agency URI resolve to a description of the agency.

## Q2 — Hash vs slash → **REVISE leaning REJECT** (AGAINST the slash rule; revert to hash now — sharpened after Gandon cross-talk)

**Baker:** I now ballot AGAINST rule 2 (slash, overriding ODR-0004). My initial position was a conditional REVISE/FOR — "slash is OK if the WG funds the resolution infrastructure." Gandon supplied the fact that shows I was answering the wrong question: **ADR-0006's own reopening trigger for the hash decision is "≥1,000 terms OR a named consumer needs per-term content-negotiation," and neither is met** (~147 terms; no named consumer). The council has already written down the conditions under which slash earns its keep, so rule 2 fails its own stated test — the override of ODR-0004 lacks grounds. The W3C "Best Practice Recipes for Publishing RDF Vocabularies" (Berrueta & Phipps, 2008) confirms the direction: **Recipe 3 (hash)** is for *small, stable* vocabularies that fit one document — one GET returns every term, fragments resolve client-side, zero redirect infrastructure, the namespace document *always* resolves; **Recipe 4/6 (slash)** is for *large or selectively dereferenced* vocabularies, paying the 303/per-term-resolution cost. opda's ~147-term core is squarely Recipe-3, and ODR-0004 §Rules.1 ratified hash 9–0 (Knublauch DA) on that reasoning. flat-slash takes the *cost* side of Recipe 4/6 while having Recipe-3 *scale* — the worst trade. **Q5 strengthens this, it does not weaken it (Gandon's sharpening):** the multi-artefact mass (data-dictionary 582, 31 profiles, instance data) lives in `/harness/`, not in the term namespace — so the Recipe's "partitioned → Recipe 4" branch is already satisfied at the `/pdtf/` vs `/harness/` boundary, NOT inside the term space. `/pdtf/` is therefore mono-artefact-small (the ~147 classes/properties + SKOS schemes, one co-evolving document), which makes the case for hash on `/pdtf/` *stronger*, not merely defensible. (The one future seam to watch: if the SKOS concept schemes grow large or get dereferenced selectively, that is the natural module boundary and a reopening trigger — but at current scale they ride in the same document fine.) **The DPV precedent argues against the proposal, not for it:** DPV is "slash-to-the-module, hash-within-module" (`/dpv/pd#Address`) — the Cool-URIs (Sauermann & Cyganiak, W3C Note 2008; httpRange-14 TAG resolution) "whole vocabulary in one document" pattern applied per module, where the module IS the dereferenceable document. opda is not adopting per-module documents, so DPV licenses HASH for opda's core, not flat slash. **Amendment:** revert to the ODR-0004 single hash namespace now; adopt slash only when a reopening trigger actually fires, at which point copy DPV's slash-to-module + hash-within hybrid — never flat slash. (Note: the sibling hm `hm.com/ns/` slash is not a counter-example — hm partitions by bounded-context sub-namespace, which is the slash-to-module half of the DPV pattern, not flat-slash-per-term.)

## Q3 — Versioning → **AFFIRM** (FOR)

**Baker:** No version segment in the IRI, version carried by `owl:versionInfo`, is the correct and well-precedented choice. This is exactly DCMI's practice and DPV's: the *term URI is permanent and version-independent* — `dcterms:title` has meant the same concept since 2008 and its URI never carried a version — while versioning is expressed by `owl:versionInfo` / `owl:priorVersion` literals and dated *release artefacts*, not by minting new term URIs. The DCMI namespace policy is built on the principle that **a term URI, once published with a given meaning, is never reassigned and never versioned in the string**; semantic evolution that would change meaning forces a *new term*, not a versioned URI. Embedding `1.0.0` in term IRIs (as the current `…/opda/1.0.0/` and `…/opda/property/1.0.0/` forms do) is the anti-pattern — it guarantees that every consumer's stored data points at a stale, version-pinned URI the moment 1.0.1 ships. Retiring the version segment from term IRIs is the single highest-value fix in this proposal. Keep dated/versioned *release snapshots* as separate retrievable artefacts (a `owl:versionIRI` pointing at an immutable copy is fine and is good practice), but the canonical term IRI stays version-free.

## Q4 — Modules / flat namespace → **REVISE** (FOR, with a recorded test)

**Baker:** Flattening is sound *for this vocabulary*, but the proposal should record *why* so the decision is auditable rather than assumed, because it diverges from both cited prior-art projects (hm `/ns/pf/`, `/ns/sds/`; DPV `dpv/pd`, `dpv/legal`). **The grounding is operational, not identity (Pandit's correction, which I accept).** It is tempting to rest the case on the SKOS Reference (§4.6): scheme membership is asserted by `skos:inScheme`, *not* by URI prefix — a `skos:ConceptScheme` is an RDF resource and concepts belong to it by triple, never by string-matching their IRI. That is true and it disposes of the *identity* worry (one flat namespace + `skos:inScheme`, and for OWL terms `rdfs:isDefinedBy`, carries scheme identity perfectly — opda's 45 SKOS schemes can co-exist under one flat `/pdtf/` precisely because membership ≠ prefix). But identity is the wrong axis: what DPV's and hm's module *URI segment* actually buys is **three operational affordances `skos:inScheme` cannot provide** — (a) independent **dereference** (`dpv/legal/eu/gdpr` resolves to just the GDPR document, not the whole graph), (b) independent **versioning clock** (the GDPR module re-releases on EU-law time, off `pd`'s clock), (c) independent **citation/import** (`owl:imports` only that module). Wholesale deprecation of a module is really (b)+(c): you retire a release+import unit. Those are **document/release boundaries**, not membership assertions — which is exactly why a URI segment, and not a triple, is the right carrier *when a driver for (a)–(c) exists*. opda has no such driver: its 6 modules are file splits of a single co-versioned standard with one governance owner, none independently dereferenced/versioned/imported. So the segment has nothing to buy, and flattening is correct. **Amendment — adopt Pandit's reopening criterion (sharper than the symptom-level test I first proposed):** the ADR should state *"introduce a module segment only when a sub-vocabulary acquires an independent versioning clock OR is adopted/imported independently"* — the DPV trigger stated as a condition, naming the driver (a release/import unit) rather than the symptom. This is one face of a single reopening principle the ADR can state once and reuse for Q2's SKOS-scheme watch-item: **a sub-vocabulary earns its own URI segment exactly when it becomes an independent dereference / version / import unit.** This is the URI-level mirror of [[opda-classification-over-inheritance]]: partition *logically* (by `skos:inScheme` membership) until an *operational* release boundary forces partitioning *structurally* (by URI prefix).

## Q5 — Standard-entity vs physical-resource split → **REVISE** (FOR, conditional on a recorded placement procedure — hardened after Davis (DA) cross-talk)

**Baker:** The `/pdtf/` (the standard) vs `/harness/` (about/supporting the standard) partition is sound linked-data and sound metadata governance — it is **not** over-engineering, and "one namespace + `rdf:type`" is the inferior alternative *for the normative core*. The distinction tracks a real and durable line the DCMI/FRBR-influenced tradition takes seriously: *the published abstract vocabulary* (whose terms must be maximally stable, citable, never disturbed) vs *resources about or supporting it* (instance data, test fixtures, build artefacts, ODR/ADR records) whose churn and lifecycle differ entirely. Co-locating volatile build/instance/test resources in the *term* namespace pollutes the one space that must stay quiet and citable. Keeping the standard's namespace pristine is precisely how DCMI protects `/dc/terms/` and W3C protects its vocabulary namespaces from the working drafts and test suites that orbit them. The split also gives a clean dereference story without `rdf:type` interrogation: `/pdtf/` → normative ontology, `/harness/` → supporting material. **One framing correction (Pandit):** `/harness/` is *not* the analogue of keeping `dpv-owl`/`dpv-skos` serialisations separate — those are content-negotiated *representation variants of the same denoted entity*. `/pdtf/`÷`/harness/` is a split by **entity kind** (standard entity vs apparatus *about* the standard) — a **citability boundary** ("is this THE standard, or material about it?"), not a representation boundary. This keeps Q5 consistent with Q7: serialisation/expression differences (OWL-vs-SHACL) must *never* move a namespace boundary, whereas entity-kind/citability differences do.

**Davis (DA) is right on one point, and it forces me from AFFIRM to REVISE:** a namespace boundary with *no written decision procedure* is a standing tribunal — "is it a *standard entity*?" as a bare intuition re-litigates forever, and the SHACL case (Q7) proves the intuition under-determines (a shape node is in `opda:`, is normative, yet describes physical data). With no consumers yet, the split's benefit is partly hypothetical while the litigation cost is immediate. So the split is only defensible if the procedure is **recorded and binding**, not left to per-case intuition. Where I diverge from Davis's "one namespace + `rdf:type`, defer the split" (Dodds & Davis, *Hierarchical URIs + Natural Keys*): his approach has zero litigation cost because the wall is in a re-classifiable triple — but the one distinction a *citing* consumer must resolve **without** fetching-and-type-querying is "is this part of the normative standard?", and that is a genuine governance/identity boundary that earns its URI segment. The asymmetry Davis understates: an `rdf:type` triple is freely re-classifiable, but *which side of a URI wall a resource sits on* is not — DCMI namespace policy holds a published term URI is never reassigned, so deferring the split to post-publication is the expensive path, not the cheap one. The three "disputes" (Q6/Q7/Q8) are the boundary being *calibrated once at design time* — exactly when calibration is cheap — not the boundary failing.

**Amendment (the binding procedure the ADR must record):**
1. **Placement test:** *Would a consumer ever cite this resource AS part of the normative standard, or deprecate/version it as a unit of the standard?* Yes → `/pdtf/`; No → `/harness/`. (Resolves all three live cases deterministically: shape nodes → `/pdtf/`; data-dictionary → `/harness/`; profiles → `/pdtf/profiles/`; and future kinds — a published normative crosswalk → `/pdtf/`, a test fixture → `/harness/`.)
2. **Fallback for genuine ambiguity:** default to `/harness/` (the re-classifiable side — this is where I bank Davis's point), and **promote to `/pdtf/` only by council; never demote a published `/pdtf/` URI.**
3. **One-directional dependency:** nothing in `/pdtf/` may `rdfs:isDefinedBy` or normatively depend on `/harness/`, or the boundary becomes fictional.
4. **Path segment, not a second authority (Davis's residual, resolved on governance grounds — my call):** the `/pdtf/`÷`/harness/` boundary is realised as **a path segment under one `/opda/` authority/PICG redirect, NOT two coordinate namespace roots.** The governance test for a coordinate authority (DCMI namespace-policy delegation reasoning) is whether the two could plausibly diverge in *stewardship* — different maintainer, change-control body, PICG ownership, independently transferable. They cannot: `/harness/` holds opda's *own* governance/build apparatus (ADRs, ODRs, exemplars, instance/test data, named graphs), all authored/versioned/retired by opda *as steward of pdtf* — co-stewardship is structural, not incidental, so a second authority root is unearned (*Linked Data Patterns*, Hierarchical URIs: a segment under one authority unless the authorities genuinely diverge). This does **not** weaken the boundary-in-the-URI: the citation-test resolves identically and a consumer reads "normative" off the `/pdtf/` *path segment* either way — path-vs-coordinate is purely a redirect-surface/deploy-side choice (one PICG rule, no sibling drift), which ADR-0006 already decouples from URI policy.

With (1)–(4) recorded, FOR. Without a recorded procedure, I join Davis's single-namespace REVISE outright — an uncalibrated wall *is* a tribunal. (Davis withdrew the standing-tribunal attack and single-namespace REVISE once the procedure was recorded; his retained path-vs-coordinate refinement is resolved by (4) and is captured in the ADR-0006 amendment as Q5 disposition §4.)

## Q6 — data-dictionary placement → **AFFIRM** (FOR)

**Baker:** The 582 data-dictionary entries belong in `/harness/`, and this follows directly from Q5. These entries are *physical PDTF data fields* — descriptions of where data sits in the property pack, keyed like `propertyPack.environmentalIssues.flooding` — they are not abstract published standard *terms* with independent semantic identity in the ontology sense. They are reference/registry resources *about* how the standard's data is laid out, which is exactly the `/harness/` lifecycle: high-churn, instance-shaped, build-regenerated. Placing them in `/pdtf/` would be the category error the SKOS Reference warns against — conflating a *concept* (an abstract unit of thought in a scheme) with a *data-field locator*. If a given data-dictionary entry turns out to denote a genuine controlled-vocabulary concept (e.g. an enumerated flooding-risk value), *that concept* should be minted as a `/pdtf/` SKOS concept and the data-dictionary entry should `dct:references` it — but the dictionary entry itself stays in `/harness/`. Correct as proposed.

## Q7 — SHACL shapes placement → **REVISE** (FOR)

**Baker:** The proposal's instinct — shape *documents* in `/harness/`, shape *nodes* somewhere principled — is right; my amendment is on *where the nodes live*. Shape NODES that are part of the published, normative standard (the base shapes and the profile shapes that the standard *defines as its conformance criteria*) are **standard entities** and belong in `/pdtf/`, alongside the classes and properties they constrain, because conformance shapes are part of what the standard *is* — they are as normative as the OWL axioms. I would **reject the dedicated `/shacl/` namespace**: it fragments the standard's identity across two namespaces for a serialisation/technology reason (SHACL-vs-OWL) that has nothing to do with governance — the same mistake as versioning-in-URI, just on a different axis. The W3C vocabulary-publishing tradition keeps the *meaning* of a standard in one namespace regardless of the languages (RDFS/OWL/SHACL) used to express it. So: **shape nodes → `/pdtf/` (in the `opda:` term namespace), shape/profile documents (the physical TTL files served) → `/harness/`.** This also keeps Q7 consistent with Q5's one-directional rule: a `/pdtf/` shape may `sh:targetClass` a `/pdtf/` class with no cross-boundary dependency. Amendment: drop `/shacl/` from consideration; shapes are not a separate vocabulary, they are the standard's constraint layer.

## Q8 — Profiles → **REVISE** (split the answer; FOR)

**Baker:** Profiles need disaggregating, and the proposal treats them as one thing when they are two. A per-form overlay **profile shape** — the SHACL node that *says what conforms for baspi5* — is a **standard entity**: it is opda asserting, normatively, "this is the conformance profile for the baspi5 form," and per Q7 it belongs in `/pdtf/` (e.g. `…/pdtf/profiles/baspi5` for the profile *resource*, with its shape nodes in the same term namespace). The profile **document** (the served TTL) and any **instance/test fixtures** for that profile are physical resources → `/harness/profiles/`. The discriminator is the same one DCMI uses for Application Profiles (Singapore Framework for Dublin Core Application Profiles, Nilsson/Baker/Johnston 2008): **a profile is itself a first-class, citable specification artefact** — people cite "the BASPI profile" the way they cite the standard — so its identity must live in the stable standard namespace, not the churning harness. The `dct:source` link to the external form question is exactly right and should be preserved (`dct:source` / `dct:references` to the form authority's question). **Amendment:** `/pdtf/profiles/<form>` for the profile resource + its shape nodes; `/harness/profiles/<form>` for the document and fixtures; keep `dct:source` to the external form. This makes the 31 profiles independently citable standard artefacts — which, unlike the 6 modules in Q4, they genuinely are, so here the URI segment *earns* its keep.

### Cross-cutting note for the synthesis

Three of my revisions share one root principle, which the ADR should state once rather than rediscover per-question: **put governance/identity boundaries in the URI, put logical/serialisation distinctions in the triples.** Org-vs-standard (Q1), standard-vs-harness (Q5), and per-profile identity (Q8) are *governance* boundaries → they earn URI segments. Module membership (Q4), OWL-vs-SHACL expression (Q7), and version (Q3) are *not* governance boundaries → they go in `skos:inScheme` / `rdfs:isDefinedBy` / `owl:versionInfo`, never in the URI string. Q2 is the one genuinely open question and turns entirely on whether the WG will fund slash resolution; absent that commitment, ODR-0004's hash stands.
