# Session 029 (ODR-0023 R2) — Holger Knublauch, SHACL / realisation voice

*Opening.* I co-edited SHACL because the operationally load-bearing layer in a working data system is the one that runs at the boundary — the shapes that say what MUST be present, what value-space is admissible, what cardinality holds *in this context* — and SHACL was deliberately specified to do that against a **fixed, separate TBox**, never to redraw the class graph (SHACL §1.4, *Relationship between SHACL and RDFS/OWL*; the shapes graph and data/ontology graph are distinct). The engineering question in front of R2 is therefore narrow and answerable: name one query or one validation that **fails today** and that minting `property-qualities` / `property-modes` / `legal-estate-attributes` as three w3id.org namespaces would make **pass**. I have read the emitted corpus through that lens and the answer is none. The 94-odd Property+LegalEstate leaves already carry per-property SHACL property-shapes, their per-form requirement variation already lives in the ODR-0010 overlay profiles (the overlay *is* the form), and their bearer is already pinned by `rdfs:domain`. A module split adds three namespace IRIs, three import edges, and a permanent split-maintenance burden over leaves whose constraints are already realised — and buys, on the present evidence, exactly zero new entailment and zero new validation. That is the textbook definition of Allemang's word: decorative. I align with Allemang and Davis on the disposition and I will carry the realisation question (Q5) concretely: keep the UFO insight, but in the leanest vehicle that ODR-0010 already sanctions — a gUFO `rdf:type` annotation in the separate annotation graph — never a module, and never a SHACL shape.

---

### Q1 — Load-bearing? Does a NAMED consumer query or validator need the Quality-vs-Mode (or legal-estate) entailment as a typed set?

**REJECT.** No tabled consumer query and — my specific contribution — **no SHACL validator** needs the typed set. Every constraint a form actually enforces (EPC band ∈ scheme, council-tax band present under BASPI5, tenure value admissible) is already expressed as a property-shape `sh:path`/`sh:in`/`sh:minCount` keyed on the **bearer node**, exactly the ODR-0010 §Rules.1–2 canonical mapping (`required → sh:minCount 1`; `enum → merged sh:in`); none of those shapes selects `?x a gufo:Quality`, and turning a hypothetical Quality typing off changes no validation report. A module split is load-bearing only if a shape would `sh:targetClass` the UFO meta-type or a query would `SELECT` on it — neither exists. This is just the §Q6a reasoner-independence test (UNION-over-children must equal the entailed parent, else decorative) read at module grain, and the corpus fails it.

### Q2 — Valid partition key? Is UFO meta-category fit for a permanent w3id.org module boundary, given straddlers/re-sorters?

**REJECT** (as a boundary key). A SHACL target must resolve a node to exactly one shape deterministically; a w3id.org module IRI is a far harder, near-permanent commitment, and the partition key behind it must be single-valued per leaf. §Q5a fails this in its own table: `priceQualifier` and `marketingTenure` sit in one cell typed "**Mode / Quality Value**" — one leaf, two axes — and `ownershipType` is a Quale whose `rdfs:domain` is `opda:LegalEstate`, straddling Allemang's axis and Kendall's. A key that lands a single leaf on the slash between two module IRIs is not a partition; it would force re-homing across published namespaces every time the metaphysical reading is revisited. The one single-valued, machine-checkable key on these leaves is `rdfs:domain` — which is exactly what the walk used and what a SHACL `sh:targetClass` already keys on.

### Q3 — Module reality: does ODR-0008c collapse to `rdfs:domain opda:LegalEstate`? Do a/b need a typing pass? What of the ~70 other-class leaves?

**AFFIRM** (the brief's reading is correct). ODR-0008c **is** `rdfs:domain opda:LegalEstate` (the 25 tenure/lease/charge leaves of ODR-0024 R1) — already a one-line selection (`SELECT ?p { ?p rdfs:domain opda:LegalEstate }`) and a one-line SHACL target (`sh:targetClass opda:LegalEstate`); wrapping that in a module is ceremony around an existing filter. ODR-0008a/b **cannot be drawn**: the 69 `opda:Property` leaves carry no gUFO meta-type (Allemang confirmed zero `gufo:` triples in the emitted TTL), so a/b would mint the namespace *before* the content that justifies it. And ~70 leaves sit on `opda:Search`/`Transaction`/`NearbyFacility`/`Organisation`/`RiskAssessment` — out of axis-scope, owned by ODR-0008d/0006/0007. A cut that addresses under 60% of the leaves and leaves the rest to other modules is a partial overlay, not a modularization; from a shapes standpoint all 164 are already cleanly addressable by bearer-keyed targets without any of it.

### Q4 — Disposition + precise trigger.

**REVISE → DEFER a/b; REJECT 0008c as a standalone module; keep ODR-0008 monolithic; mint nothing.** Affirm the session-025 framework, stewards, and per-axis criterion as standing. The realisation-grade re-open trigger, conjunctive so it cannot fire on metaphysics:

> **ODR-0008a/b spawn only when BOTH hold, per axis:** (i) a gUFO meta-category typing pass has actually run and committed `?x a gufo:Quality` / `gufo:Mode` to the corpus (a typed set exists as data); AND (ii) **either** a named consumer query selects on that meta-type and its answer changes with the typing removed, **or** a named SHACL validator must `sh:targetClass` the meta-type to enforce a constraint the bearer-keyed shapes cannot. Absent both, the axis is decorative and MUST NOT spawn.

**ODR-0008c is rejected outright as a module** (definitionally a `rdfs:domain` filter / `sh:targetClass` target the corpus already provides), not deferred. Kendall's **attachment-class** four-way (Property vs LegalEstate vs …) is the genuinely useful live cut — but it is the bearer-Kind axis, already realised by `rdfs:domain` and SHACL targeting, **not** the UFO axis; it belongs with the bearer-Kind work, and I would still ask Kendall what dereferenceable entailment a distinct namespace buys that `sh:targetClass opda:LegalEstate` does not. This mirrors ODR-0024 R4's discipline (the School/HealthCareFacility Subkind split held-as-live on a consumer-query trigger, band constraints SHACL-scoped per overlay in the meantime) applied to the UFO axis.

### Q5 — (My central question.) If NOT spawning, preserve the UFO distinction WITHOUT minting modules — gUFO `rdf:type`, SHACL, `dct:subject`, or not recorded?

**REVISE → record it as a per-property `gufo:` `rdf:type` annotation in the *separate annotation graph*, gated to uncontested leaves; NOT a SHACL construct, NOT a module.** This is the leanest faithful vehicle and ODR-0010 already prescribes the home. Ranking the four options on the realisation criterion (what does a consumer/validator actually need, at what maintenance cost):

1. **gUFO `rdf:type` triple per property — CHOSEN, lowest faithful cost.** `opda:currentEnergyRating a gufo:Quality`, `opda:groundRent a gufo:Mode` makes the meta-category **recoverable by a single SPARQL `?p a gufo:Quality`** with no namespace, no import edge, no split maintenance — and it reverses for free if a typing turns out wrong. Critically it lands **in the separate annotation graph, NOT the shapes graph**: ODR-0010 exiled advisory annotations from the shapes graph (≈7–2) precisely so a SHACL processor "must find only interpretable constraint components plus known annotation properties," and §Q7a clause 3 + CI test 3 (`ASK { GRAPH opda:annotations { ?s a sh:NodeShape } }` returns FALSE) make that a checked boundary. A `gufo:` `rdf:type` is a classification triple, not a constraint, so it belongs there — alongside `opda:aiHint`, keyed to the property IRI. **Gate it to the uncontested leaves** (`gufo:Quality` on the EPC/council-tax/built-form Quale-in-Region leaves) and **omit the straddlers** (`priceQualifier`, `marketingTenure`) and re-sorters (`ownershipType`) until adjudicated — annotating a leaf the §Q5a table itself slashes across two axes would record a falsehood.

2. **SHACL construct — REJECT.** A node-shape or `sh:class gufo:Quality` constraint implies a **validation consumer that does not exist**: SHACL exists to fail data at a boundary, and no data is malformed for lacking a Quality typing. Minting per-axis shapes also re-incurs exactly the structure cost Q1 found unwarranted, and worse, drops a non-constraint into the shapes graph in direct violation of ODR-0010's purity rule and §Q7a CI test 3. SHACL is the right vehicle the moment a **named validator** must enforce the Quality/Mode distinction at the data boundary — that is my withdrawal condition — but absent that consumer it is the wrong tool, and I will not reach for it just because I am the SHACL voice.

3. **`dct:subject` — acceptable fallback only.** Lighter still and dependency-free if even gUFO typing is contested, but it under-specifies (`dct:subject` is generic aboutness, not a meta-ontological `rdf:type`), so it is faithful-enough but not the faithful *best*; prefer gUFO typing where uncontested, fall back to `dct:subject` only where the gUFO type itself is disputed.

4. **Not recorded — REJECT as the floor but note it is nearly free.** The framework + criterion already stand in session-025/ODR-0023 R2 and the straddler evidence in §Q5a, so the insight is not lost if nothing is annotated. But a cheap recoverable annotation that costs one reversible triple per uncontested leaf is worth the negligible price over pure prose.

**Cost/benefit vs a namespace split, stated flatly.** *Annotation:* N reversible triples in an existing graph, queryable today, zero import management, zero IRI commitment, reverses for free. *Module split:* three permanent w3id.org IRIs, three `owl:imports`/profile-composition edges to maintain, leaves re-homed across published namespaces on every re-sort of a straddler, and a build/CI surface that must keep the overlay composition (ODR-0010's documented graph-union, which "MUST be specified and maintained as code") consistent across the split — all to deliver, today, no entailment and no validation that the bearer-keyed shapes + `rdfs:domain` do not already deliver. The annotation discharges the insight at the lowest faithful cost; the split capitalises a decorative distinction into permanent infrastructure. Derive, don't declare — and realise in the leanest vehicle that does the job.

---

**One-line verdict.** Nothing fails today that a module split would fix — the constraints already live in per-property SHACL shapes and ODR-0010 overlay profiles, the bearer is already pinned by `rdfs:domain`, and no validator or query selects on a UFO meta-type. **REJECT spawn-now; DEFER a/b on the conjunctive (typing-pass AND query-or-validator-bites) trigger; REJECT 0008c as a standalone module; preserve the insight as gated gUFO `rdf:type` annotations in the *separate annotation graph* (ODR-0010), never a SHACL construct and never a module.**
