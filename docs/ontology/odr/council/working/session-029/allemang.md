# Session 029 (ODR-0023 R2) — Dean Allemang, `property-qualities` steward

*Opening.* I set this gate in session-025, so let me hold myself to it. A model element does not earn its keep because an upstream foundational ontology says two things are *different in kind*; it earns its keep when a consumer query needs the distinction it draws — when **turning the axis off changes an answer**. That is the whole burden of the competency-question-driven method we argue for in *Semantic Web for the Working Ontologist* (Allemang, Hendler & Gandon, 3rd ed., 2020): you model the questions you must answer, and you **derive, don't declare** — a structure is justified by the inferences it lets a working consumer draw, not by the metaphysical pedigree of its labels. I came to R2 prepared to spawn if the walk had produced a UFO-typed leaf-set plus a named load-bearing query per axis. It produced neither. What the walk produced is a *bearer-domain* partition (`rdfs:domain`), and I have now confirmed against the live corpus that there is **no `gufo:`/`ufo:` prefix declared anywhere in the emitted ontology and not a single `rdf:type gufo:Quality`/`gufo:Mode` assertion on any leaf** — the lone "UFO: …" string is one `skos:scopeNote` prose note on `opda:RiskAssessment`. So the partition the spawn-rule presupposes does not exist as data. The framework is sound; the spawn is not earned. I REJECT spawning now, and I will give the cleanest re-open trigger I can.

### Q1 — Load-bearing? Does a NAMED consumer query need the Quality-vs-Mode (or legal-estate) entailment as a typed set?

**REJECT.** No tabled query needs "all Qualities of this Property as a typed set" *distinct from* "all Modes." Every competency question on the table — give me the EPC band, the council-tax band, the built form, the tenure of this estate — is answered today by a bearer-scoped property pattern (`?p opda:currentEnergyRating ?b`) or, if you want the whole descriptive bag, by `?p ?prop ?v` over the `rdfs:domain opda:Property` set the walk already emitted. My session-025 criterion is explicit and I will not soften it for my own axis: a sub-module spawns *only* when turning the axis off changes an answer. Turn off a hypothetical `gufo:Quality` typing and re-run every query in the corpus and **not one answer changes**, because not one query selects `?x a gufo:Quality`. This is exactly the reasoner-independence discipline of §Q6a hoisted to the module level — and *SWWO* 3rd ed. is blunt that an entailment with no query behind it is decorative weight, not modelling. Quality-vs-Mode is real; real is not the test.

### Q2 — Valid partition key? Is UFO meta-category a RIGID, stable key fit for a permanent w3id.org module boundary?

**REJECT** (as a *boundary* key — AFFIRM only as analytic commentary). A w3id.org module boundary is a near-permanent published commitment; the key that draws it must be rigid and re-sort-free. UFO meta-category is neither, on this very corpus. ODR-0008 §Q5a already shows the instability in its own binding table: `priceQualifier`/`marketingTenure` are typed "**Mode / Quality Value**" — a *straddle the table itself records with a slash* — and Davis's `ownershipType`-as-Quale-of-LegalEstate is genuinely ambiguous between row 1 (Quale-in-Region) and the Mode reading. A key that puts a single leaf on the *slash between two modules* cannot cut a permanent boundary; you would be re-homing leaves across published module IRIs every time the metaphysical reading shifts. *SWWO* (3rd ed., on modelling for change) warns precisely against committing infrastructure to distinctions that the data will make you re-litigate. The one stable, observable key in play is `rdfs:domain` — the bearer Kind — which is exactly what the walk used.
