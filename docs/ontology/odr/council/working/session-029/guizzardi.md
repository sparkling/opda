## Q1 — Load-bearing? Does a named consumer query need the Quality-vs-Mode (or legal-estate) entailment as a typed set?

**REJECT (as currently met).** I can construct the *shape* of such a query — "return every property whose Quale-in-Region is worse-than-D on an ordered region" needs quality-space reasoning a flat `xsd:string` cannot give (Masolo et al. 2003 *WonderWeb D18* §4.3, the Quale-in-Region region structure I rely on for Quality individuation) — but constructing it is not exhibiting it, and Allemang's S025 test is exact: *turning the axis off must change an answer, else it is decorative.* No such query is attested in ODR-0024 or the corpus. Worse, the entailment that *would* be load-bearing (ordered-region comparison over an EPC scheme) is delivered today by the **SKOS scheme's `skos:broader`/region structure on the value**, not by a *module boundary on the property* — so even the one plausible query is answered without spawning. The distinction is real (Guizzardi 2005 Ch. 4); its *load-bearingness for a module seam* is **unshown**.

## Q2 — Valid partition key? Is UFO meta-category a rigid, stable key fit for a permanent w3id.org module boundary?

**REJECT — and here I must engage Guarino directly, because this is an OntoClean rigidity question and the evidence is against me.** A module boundary on w3id.org is a *rigid* commitment — once minted it cannot non-disruptively re-sort. But two of the council's own exemplars are not rigid under the UFO cut: **`priceQualifier`** is tagged in ODR-0008 §Q5a as *"Mode / Quality Value"* — it straddles two of my three axes — and under UFO a value that straddles Mode and Quality is precisely the signature of a *Relator hiding a Quality* (the price-qualifier inheres in the *listing/sale relator*, S007 territory, not in the Property), i.e. it is a **modelling error or an under-analysed Relator**, not a stable axis member (Guizzardi 2005, the Relator/Mode/Quality trichotomy; OntoClean anti-rigidity, Guarino & Welty 2002). **`ownershipType`** is a Quale of `LegalEstate` — so does it belong to `property-qualities` (it is a Quale) or `legal-estate-attributes` (its bearer is the estate)? The two candidate keys — *UFO meta-category* and *bearer Kind* — **disagree on this very leaf**, which is the definition of a non-functional partition. You cannot mint a rigid namespace around a key whose own worked examples re-sort under the next analysis pass. The cut is a sound *analytic lens*; it is not a *rigid sortal partition* of the property universe.

## Q3 — Module reality: does ODR-0008c collapse to `rdfs:domain opda:LegalEstate`? Do a/b need a gUFO pass not yet done? What of the ~70 other-class leaves?

**REVISE — the three "axes" are not one object.** (i) **ODR-0008c `legal-estate-attributes` collapses to `rdfs:domain opda:LegalEstate`** — the 25 LegalEstate leaves are *already* partitioned by the walk's bearer-domain (HARD EVIDENCE 3–4). But that is **Kendall's four-way attachment-class split** (Property vs LegalEstate vs …), which S025/Davis flagged as held-as-live and *distinct from the UFO axis* — calling a `rdfs:domain` facet "the UFO Mode/Quality axis" would be a category mistake about what typed the leaves. (ii) **ODR-0008a/b genuinely do not exist yet:** the 69 `opda:Property` leaves were never sorted Quality-vs-Mode; that is a **fresh per-leaf gUFO typing pass the walk did not perform** (HARD EVIDENCE 4). You cannot spawn `property-modes` when no leaf carries `gufo:Mode`. (iii) The **~70 other-class leaves** (Search 23, Transaction 15, NearbyFacility 8, …) sit on classes the three-axis split does not address at all — they belong to ODR-0008d / ODR-0006 / ODR-0007 (and ODR-0008d already typed its artefacts as **Information Objects**, not via a module wall — see Q4). So the split's real scope is at most ~94 of 164, and within that only ~69 are even candidates for the a/b UFO cut, none yet typed.

## Q4 — Disposition: spawn / defer / reject?

**REVISE → DEFER with a sharpened, two-part trigger; do not spawn now.** The UFO-faithful sequence is **type first, split only if a query then needs it** — and the corpus already set this precedent in ODR-0008d, where the five artefact classes + `opda:RiskAssessment` were typed as UFO **Information Objects via `rdf:type`/scopeNote, not via a module boundary** (ODR-0008d Rule 1(a), Rule 3; "UFO typing in this corpus is done via class/rdf:type, not module boundaries"). A UFO meta-category is a property of a *universal* — in gUFO it licenses an assertion `ex:builtForm rdf:type gufo:Quality` (Almeida et al. 2019, the gufo:Quality/gufo:Mode/gufo:Relator vocabulary) — it does **not**, on its own, license a separate *namespace*. **Sharpened trigger for the eventual a/b spawn (replacing Davis's S025 condition, now testable and unmet):** spawn `property-qualities` and/or `property-modes` only when **BOTH** (1) a per-leaf **gUFO typing pass** has run over the ~69 Property leaves, emitting `gufo:Quality`/`gufo:Mode` `rdf:type` triples (with straddlers like `priceQualifier` *resolved* — re-analysed as a Quality of the S007 listing Relator, not parked as "Mode/Quality"), **and** (2) a **named consumer query** is exhibited whose answer-set differs under the Quality-vs-Mode entailment for that specific axis. `legal-estate-attributes` is **not** a UFO-axis spawn — route it to Kendall's attachment-class four-way, on its own trigger, never conflated with the Mode/Quality cut.

## Q5 — If not spawning, how to preserve the insight?

**AFFIRM — gUFO `rdf:type` annotations, exactly as ODR-0008d did.** Do not lose the distinction; *record it where UFO says it lives* — on the universal. Emit per-property `rdf:type gufo:Quality` / `gufo:Mode` annotations (and `gufo:Relator` for the resolved `priceQualifier`) into the descriptive TTL, mirroring ODR-0008d's Information-Object typing. This is strictly stronger than `dct:subject` tagging and strictly cheaper than a module: it makes the meta-category **machine-queryable and reasoner-visible** (a future consumer query *can* then ask "give me the Modes of this estate," which is the very artefact Q1/Q4 require to fire the spawn), it forces the straddler resolution UFO demands *before* it can pollute a namespace, and it keeps the cut reversible. The gUFO annotation pass is, in fact, the precondition I named in Q4(1): doing it both preserves the insight *and* generates the typed set without which no honest spawn judgement is possible. Type the leaves first. Split only when a query makes the seam load-bearing.
