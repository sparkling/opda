## OPENING

### Citation grounding I rely on (web-verified before asserting)

- **OWL 2 Primer §2** — the open-world assumption: "in the case of an OWL 2 document it may simply be missing (but possibly true)". Absence of an edge is not absence of truth.
- **OWL 2 Primer §4.2** — subclass relationships *must be explicitly stated*; they are not derivable from labels or co-membership. So "no `rdfs:subClassOf`" is a deliberate modelling fact, not an omission to be papered over.
- **OWL 2 Primer §8.1 / §9** — annotation properties "have no formal meaning" under the Direct Semantics. An `opda:ufoCategory` string is an annotation: it carries **no entailment**.
- **SKOS Reference §3.1** — a `skos:Concept` is "an idea or notion; a unit of thought" — a *value*, modelled as an **individual**, not a logical class.
- **SKOS Reference §3.5.1** — the spec makes "no additional statement about the formal relationship between the class of SKOS concepts and the class of OWL classes." The two registers are deliberately held apart.
- **SKOS Reference §8.6.6** — `skos:broader` is **NOT transitive** and does **not** assert subclassing; it is a vocabulary-navigation link, not a T-Box axiom.
- **SHACL Core §2.1.3 / §4.6.1 (`sh:in`)** — a shape is a *constraint on a focus node's data graph*, not an assertion about the class. `sh:in` enumerates *permitted values of a path*; it is in the shapes graph, not the ontology graph.
- Hendler, **"A Little Semantics Goes a Long Way"** (widely-cited maxim; cf. Hendler & co-authors, *Semantic Web* corpus) — expose the semantics that *help the consumer act*; do not surface the whole model because you can.

### The decisive empirical fact I verified myself

`sh:in` in `opda-shapes-merged.ttl` comes in **two kinds**, and the split changes the Q1 answer:
- **10 literal-valued** `sh:in ( "None" "ADSL copper wire" … )` with `sh:datatype xsd:string` — these point at **NO scheme**. There is no scheme node on the other end. They are inline enumerations of permitted strings.
- **2 IRI-valued** `sh:in ( <…/scheme/currency/GBP> <…/scheme/currency/EUR> … )` (currency, peril) whose members **are** `skos:Concept`s `skos:inScheme` an actual scheme.

So a blanket "draw class→scheme edges from every `sh:in`" would manufacture edges to schemes that **do not exist** for 10 of 12 cases. Only currency + peril have a real scheme on the far end — and those two **already** have object properties (`currency→Concept`, `peril→Concept`) per the EVIDENCE findings.

### Q1 — class→vocabulary bridge from `sh:in` — **REVISE / FOR** (narrowly)

A `sh:in` constraint is **not** an ontological relation, and drawing it as one in the general case is a category error. SHACL Core §2.1.3 is explicit that a shape constrains a *data graph*; it says nothing about the class as a universal. To render every `sh:in` as a class→scheme edge would assert structure that is **not in the model** — precisely what OWL 2 Primer §4.2 warns against (relations must be asserted, not conjured from co-occurrence).

BUT the EVIDENCE's framing over-counts the opportunity. The honest, faithful subset is small: where a `sh:in` enumerates **IRI members that are `skos:Concept`s of a known scheme** (currency, peril only), the *model already carries* the linkage through the object properties `currency→Concept` and `peril→Concept`. The defect there is not a missing edge — it is that those two object properties point at a **bare generic `Concept`** instead of the **specific scheme**. That is a model-extractor fix, not a SHACL-derived fiction.

**My REVISE:** Do **not** synthesise class→scheme edges from literal `sh:in` (no scheme exists; it would be invention). **Do** fix the two real cases by making the extractor resolve `currency`/`peril` object-property ranges (or their `sh:in` IRI members) to their **owning scheme**, and carry it as a clearly-labelled **derived view edge** `opda:usesScheme` — *badged as derived/SHACL-sourced, never asserted into the TTL*. This respects SKOS §3.5.1 (the scheme is a vocabulary register, the edge says "this class draws coded values from this vocabulary") without claiming an `rdfs:range` that isn't there. A little semantics: surface the 2 real bridges that help a consumer ("where do currency values come from?"); do not fabricate 10 phantom ones.

### Q2 — honesty of the 25 floating classes — **AFFIRM / FOR**

This is open-world honesty, not a bug. OWL 2 Primer §2: a missing edge is "possibly true but missing" — it is *not* a defect to be filled. OWL 2 Primer §4.2: subclass links must be *explicitly asserted*; the model deliberately asserts none (ODR-0011 / session-036, "classification over inheritance"). And `opda:ufoCategory` is an **annotation** (OWL 2 Primer §8.1: "no formal meaning") — so there is, correctly, **no edge** from a class to a gUFO category, because there is no axiom. To draw `rdf:type`-to-gUFO-category edges would be to render an annotation as if it were a logical relation: a category error in the other direction.

Caveat I will defend: the disconnection is *honest*, but honesty is not the same as *legibility*. The relator/role classes are disconnected on the canvas because the **relator-mediation** that connects them lives in object properties whose domain/range the extractor does have. Where a *real asserted* object property connects a Role to its Relator or bearer, that edge should already draw — if it doesn't, that's an extractor gap worth a look, distinct from inventing structure. I'll defer the relator-mediation modelling question to Guarino/Guizzardi's seat.

### Q3 — layer separation vs one canvas — **REVISE / FOR** (this is my decisive question)

A single untyped node-link graph that unions `owl:Class` + `skos:Concept`/`Scheme` + `sh:NodeShape` + gUFO categories **is the wrong instrument** when ~90% of nodes are SKOS — and it is a **register error**, not merely a colour-noise problem. The three live in different semantic registers: OWL classes are universals (T-Box), SKOS concepts are vocabulary *values* modelled as individuals (SKOS §3.1), SHACL shapes are *constraints on data* (SHACL §2.1.3). Co-plotting them on one canvas with one edge-grammar implies a relational homogeneity that the specs explicitly deny (SKOS §3.5.1 holds `skos:Concept` and `owl:Class` apart on purpose).

But "wrong instrument" does **not** mean "rip it out." It means: the right design is **explicitly layered, separately-legible views** — which the page has *already started*: SKOS is off by default (40 classes + 28 edges), behind a "Show SKOS layer" toggle; colour encodes node *type*. That is the correct direction. My REVISE pushes it one step further along its own grain:

1. Make `layer` (`owl` / `skos` / `shacl` / `external`) a **first-class facet** in the extractor output (it currently has node `type` but no `layer` grouping), so engines can group/box by register, not just colour.
2. Treat SKOS as a **tree/sunburst**, OWL as a **class/relation diagram** — the page footer *already* names the right specialist instruments (WebVOWL for OWL, SKOS Play! for concept-scheme trees). The unified canvas is fine as *one* view among tabs, but the **default** consumer view should be the OWL T-Box (classes + object properties + the 2 real `usesScheme` bridges), with SKOS and SHACL as deliberately separate layers a consumer opts into.

This is "a little semantics goes a long way" applied to visualization: show the consumer the register they came for, let them add layers, do not flatten three ontological kinds into one undifferentiated hairball.

### Q4 — surfacing the mass / "too few entities" — **AFFIRM / FOR** ("40 is correct-by-doctrine"), with a presentation REVISE

"40 classes" is **correct-by-doctrine**, not a defect. Coded values are SKOS individuals (SKOS §3.1), not subclasses; promoting them to OWL classes would *manufacture* T-Box universals out of vocabulary values — the exact confusion SKOS §3.5.1 is written to prevent, and it would explode a clean 40-class T-Box into hundreds of spurious classes. The 225 datatype properties are **attributes** (literal-valued), and the 325 shapes are **constraints** (SHACL §2.1.3) — neither is a node in a *class* graph. So I AFFIRM keeping datatype properties off the node-link canvas and shapes off as nodes.

The honest fix for "too few / too much hidden" is **presentational, on the right instrument**: datatype properties belong on the **class detail page** as an attribute list (they already are, per ontology-model.json `attributes`), and as an optional **count badge** on the class node ("Person · 7 attrs") so the consumer sees the mass exists without drowning the canvas in literal leaves. Shapes likewise stay a node decoration (the existing `hasShape` ring) plus a count. This communicates the depth without a category error. The graph's job is the *relational T-Box*; the attribute/constraint mass is shown where it is legible — not by inflating the graph.
