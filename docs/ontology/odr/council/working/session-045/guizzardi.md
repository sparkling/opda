## OPENING — per-question positions

### Q1 — class→vocabulary bridge from `sh:in`

**REVISE — ballot: FOR (recover the linkage) / AGAINST (drawing it as an ontological *relation* `opda:usesScheme`).**

The honest answer splits exactly along the two `sh:in` shapes (fact 3), and the split is a UFO class-vs-value distinction, not a tooling detail.

- A **literal-valued `sh:in`** on a quality-value property (`tenureKind`, `ownerType`, `builtForm`, `evidenceType`) is a **constraint on the admissible region of a Quality** borne by the Kind (DOLCE Quale-in-Region; Masolo et al. *WonderWeb D18* 2003 §4.3, the apparatus ODR-0030 Rule 2 calls "register-deference"). It is **not** a relation between the bearer Kind and the scheme. There is no relator mediating "Property" and "TenureKindScheme"; the scheme is the *value space the quality ranges over*, and "ranges over a value space" is not an ontological edge between two endurants — it is the typing of an intrinsic moment. To draw `Property —usesScheme→ TenureKindScheme` as a first-class graph relation **reifies a value-space membership into a relator-shaped edge that has no relator** — the inverse of the discipline this corpus enforces everywhere else (ODR-0027; ODR-0031's "classified-under, never is-a"). So: **AGAINST** minting/asserting `opda:usesScheme` in the *ontology* or presenting it as a peer of object properties.
- BUT the linkage is real *derived, documentary* information a reader needs, and it is **recoverable**: for literal `sh:in`, by matching the literals against the scheme's `skos:notation` (a reconstruction, only as sound as the notation convention — and the prose already asserts the binding); for IRI-valued `sh:in`, directly via `skos:inScheme`. So **FOR** surfacing it — but as a **derived, clearly-labelled "constrains / draws values from" affordance**, edge-styled distinctly (dashed, a SHACL-layer relation, never the same visual class as an `owl:ObjectProperty`), and computed in the extractor from `sh:in` (not asserted in the TTL). The honest predicate name is something like `opda:constrainedByScheme` / a viz-only `drawsValuesFrom`, **never** `usesScheme` phrased as if the Kind *uses* the vocabulary the way it *mediates* a relator.

The category-error guard: this is the same error as dressing the Relator-founds edge as an inert annotation (ODR-0031 Rule 5 declined that). A SHACL value-constraint and an ontological relation are different layers; recover the former, label it as the former, do not promote it to the latter. *Citation: Guizzardi 2005 Ch. 4 §4.3 (Quality / Quale — value vs relation); SHACL Core §4.6.1 (`sh:in` is a value constraint on a property shape); Masolo et al. WonderWeb D18 2003 §4.3 (Quale-in-Region); ODR-0030 Rule 2; ODR-0031 Rule 8 (anti-patterns — no `sh:in` shape in a relation role).*

### Q2 — the unconnected role/relator classes; honesty of the disconnection

**AFFIRM (the disconnection is an honest reflection of a rigid, IC-driven model) — with one REVISE — ballot: FOR.**

The 25 orphans are **two different things conflated by the brief, and only one is "honest model, leave it":**

1. **The role/relator family floating is doctrinally correct and should largely STAY.** `Buyer`, `Seller`, `Proprietor` are **anti-rigid Roles/RoleMixins** (Guizzardi 2005 Ch. 4 §4.4). An anti-rigid type has **no monadic outgoing structural edge of its own** — its very nature is that it is *founded by* a relator and *borne by* a Kind; the entity's substantive relations hang off the **Kind** and the **Relator**, not the Role. A Role that "connects" to lots of things directly would be the **Role-as-Kind anti-pattern** ODR-0027 §R3 / hm-ODR-0010's 13 fixes exist to prevent. So a sparse, peripheral Role node is the **graph signature of a correctly anti-rigid Role** — drawing more edges into it would be *less* faithful, not more. **AFFIRM:** do not invent edges to make Roles look "connected."
2. **`rdf:type`-to-gUFO-category as a real edge: REJECT, hard.** This directly violates the **co-signed absolute red line** of ODR-0031 Rule 2/8 — the gUFO alignment is referenced-not-imported and **never reasoned over**; `opda:ufoCategory` is an inert `owl:AnnotationProperty`; badges read "classified-under, **never is-a**." Surfacing `Buyer —rdf:type→ gufo:RoleMixin` as a graph edge re-materialises exactly the quarantined typing the corpus spent sessions 040/041 making inert. The category IS already shown — as the node's **colour facet** (`ufoCategory`, `ontology-graph.mjs:65`). That is the correct, inert presentation. **AGAINST** any rdf:type-to-gUFO edge.
3. **Relator-mediation as a real edge: REVISE — and this is the one place the brief has a genuine point, but it must be earned, not asserted.** ODR-0031 Rule 5 declined the founds-edge **as inert markup** because it was *not realised in the artefact* — and fact 2 confirms it still isn't: there is no `opda:mediates`/`roleOf` triple, only `numberOfSellers` (a datatype aggregate) and prose. UFO is unambiguous that a Relator **existentially mediates** the roles it founds (Guizzardi 2005 Ch. 4 §4.4 — a relator is a relational endurant whose `mediation` to each relatum is an existential-dependence relation; OntoUML's `mediation` is a first-class meta-relation). So the mediation **genuinely exists ontologically**. The question is whether it should be **asserted as a triple** (and then it would extract for free). My position: **this is a real modelling gap worth a small, bounded fix — emit `opda:mediates` (or per-overlay `roleOf`/`playedBy`) object properties for the Relator spine** (`Proprietorship —mediates→ Proprietor`; `Transaction —founds→ Seller`/`Buyer`), because (a) it is the **one place UFO is load-bearing** by ODR-0030's own finding, (b) it converts narrated mediation into queryable structure, and (c) it would connect the orphans *correctly* (via the relator, the way UFO says they connect) rather than by faking Role-level edges. This is **NOT** the Q3-style "make the graph prettier" move and **NOT** the rdf:type-to-gUFO move — it is asserting a relation that **really exists and the corpus admits exists** but never wrote down. **Caveat (honest):** this is an ontology change, not an extractor change; it touches the Relator spine ODR-0030/0031 guard closely; it must stay design-time-asserted and SHACL-validated, never reasoned (consistent with ODR-0029). It is a proposal for the operator + a Relator-spine council, not a unilateral edit. If the operator/Guarino judge the narrated-mediation sufficient, the fallback is the Q1 treatment: surface mediation as a **derived** edge the extractor reads off the relator's bearer comments — but I prefer the honest assertion.

So: the disconnection is **mostly honest** (Roles are correctly peripheral; gUFO typing is correctly an inert facet not an edge), with **one real, bounded gap** — the existential mediation the Relator genuinely bears is unwritten. *Citation: Guizzardi 2005 Ch. 4 §4.4 (Relator, mediation, role-founding, anti-rigidity of Roles); ODR-0027 §R3 (Roles never subClassOf); ODR-0030 §Outcome (Relator spine is the one load-bearing locus; `numberOfSellers` proof); ODR-0031 Rule 2/5/8 (gUFO never reasoned; founds-edge declined as inert markup; "classified-under not is-a").*

### Q3 — layer separation vs one canvas

**AFFIRM (separate the layers) — but defer the *instrument* choice to the visualization experts — ballot: FOR.**

From the UFO lens: the OWL T-Box (Kinds/Roles/Relators + their relations), the SKOS value-spaces (Quale regions, register vocabularies), and the SHACL shapes are **three different ontological registers**. The T-Box is *what kinds of things exist and how they relate*; the SKOS layer is *the value-spaces qualities range over*; the SHACL layer is *the constraints on instance data*. Conflating them on one untyped node-link canvas — where 90% of nodes are `skos:Concept` value-tokens — **buries the categorial structure (40 Kinds/Roles/Relators) under the value-space**, which is precisely backwards: the value tokens are the *least* ontologically structural content (they are quality values, not endurants). So I **AFFIRM** an explicit `opda:layer` view dimension (T-Box / SKOS / SHACL) and **AFFIRM** that the SKOS scheme-stars are better shown as **trees/sunbursts per scheme** than as 49 disconnected stars on the relational canvas (a `skos:ConceptScheme` is a hierarchy or a flat set — a tree/list instrument fits its structure; SKOS Primer 2009 §2). Whether this is "toggleable layers on one canvas" vs "separate instruments" is an HCI judgement I defer to Cagle/Davis/Allemang — the **ontological** claim is only that the three registers must be *distinguishable*, and that a single flat node-link graph is the wrong default for the SKOS mass. *Citation: SKOS Primer (W3C Note 2009) §2 (concept schemes as hierarchies/collections — tree-shaped); Guizzardi 2005 Ch. 4 (the endurant T-Box is categorially distinct from quality value-spaces); ODR-0011 §8a (SKOS schemes carry the value-spaces, distinct from the OWL Kinds).*

### Q4 — surfacing the mass / "40 classes is too few"

**AFFIRM "40 classes is correct-by-doctrine" (REJECT "too few entities" as a defect) — REVISE on presentation — ballot: FOR.**

This is the question my lens is most decisive on, and the answer is unambiguous: **"too few entities" is a category misunderstanding, not a defect.** 40 Kinds/Roles/Relators is the *correct cardinality* for this domain precisely **because** the corpus ratified classification-over-inheritance (ODR-0027): every coded discriminator (`tenureKind` Freehold/Leasehold/Commonhold, `ownerType`, `builtForm`, `evidenceType`, the council-tax bands, EPC bands…) is a **quality value / SKOS concept, NOT a subclass** — by doctrine, and grounded in my own work. Under the OntoClean cascade (ODR-0011 §8a, which I co-authored the gUFO grounding for in session-036): `+R ∧ −I` → classify (a coded facet, no subclass); only `+R ∧ +I ∧ −D` earns a subclass and `+R ∧ +I ∧ +D` is a Relator. A naive ontology that minted `Detached ⊑ BuiltForm ⊑ …`, `Freehold ⊑ Tenure`, one class per code, would have *hundreds* of classes — and that is the **atomic-class-collapse / subclass-per-value anti-pattern** (Sales & Guizzardi OntoUML anti-pattern catalogue; ODR-0011 §Alternatives rejects `owl:oneOf`/subclassing exactly). So **40 is the count a disciplined model SHOULD have**; "too few" measures the model against the anti-pattern. **AFFIRM** correct-by-doctrine.

On surfacing the mass (the 225 datatype properties + 325 shapes + 323 concepts): from the UFO lens these are **qualities/modes and their value-spaces + constraints**, not endurants — so they are **correctly NOT graph nodes in the endurant relational view**. A datatype property is an intrinsic moment's projection onto a value (Guizzardi 2005 Ch. 4 §4.3); rendering 225 of them as nodes would flood the endurant graph with non-endurant content (the same error as the SKOS flood, Q3). So **AGAINST** promoting datatype properties to graph nodes/edges in the default relational view. **BUT REVISE on communication:** the thinness *symptom* is real and is a **presentation defect, not a model defect** — the answer is (a) **node badges / a quality-count decoration** on each Kind ("Property: 40 attributes, 12 shapes") so the mass is *legible without being structural*, and (b) lean on Q3's layer separation so the SHACL and SKOS mass live in their own instruments where their cardinality is informative rather than noise. The deliverable for Q4 is a **communication fix** (badges + the foundational-ontology / classes-page prose explaining *why* 40 is correct), not a re-modelling. *Citation: Guizzardi 2005 Ch. 4 §4.3–4.4 (qualities/modes vs sortals — values are not new Kinds; the subclass-per-value collapse anti-pattern); Sales & Guizzardi, OntoUML anti-pattern catalogue (atomic-class collapse); ODR-0027 §R1 (`isMemberOf` default, `isA` only on distinct IC); ODR-0011 §8a + §Alternatives (coded SKOS not subclass trees).*
