# Session 030 — Ian Davis (deployment realism / scope discipline / publish-first)

- **Panellist** (Cagle holds the DA seat). Lens: linked-data deployment at scale, UK-Gov data patterns, publish-first, mint nothing the model does not genuinely need.
- **Framing accepted:** R3 is a directing-authority override of the held trigger. No deferral, no gate. I decide on the merits, against what BASPI5 actually carries today. Output is an ODR + an ADR; no code this session.
- **The decisive data fact I carry in:** the source array `propertyPack.buildInformation.roomDimensions.rooms[]` has **no positional token** — no `roomId`, no stable name, no slot key. The array index is form ergonomics (render order in a UI), not an identifier. Therefore **no room re-identification key exists in practice**, and a `opda:Building` node would be minted with **zero distinguishing payload** beyond what `numberOfFloors`/`opda:area`/`builtForm` already carry flat on `opda:Property`. Three leaves of 239. That is the whole estate under discussion.

## Q1 — Room: class, by-value structure, or flat properties? — **REVISE (toward by-value structure; reject class)**

REJECT the class; the honest cut is a **by-value structure** (anonymous node), not flat properties and not a Kind. Flat properties fail the deployment test the moment you co-vary: `length`/`width`/`roomName` are a **repeating group** — a property can have N rooms, and flat `opda:length`/`opda:width` on `opda:Property` cannot hold the tuple together (which width pairs with which length?). That is the one thing the data forces. But a *class* (`opda:Room` with a minted IRI) is more than the data can pay for: minting a stable IRI asserts a re-identifiable individual, and BASPI5 gives me **no token to re-identify it across two surveys** — the `rooms[]` index is not a key, it is layout order. So I land exactly where `opda:MonetaryAmount` (ODR-0024 R3) landed: an **anonymous by-value node** (`opda:RoomDimension` or similar) carrying `length`/`width`/`roomName`, attached to `opda:Property`, no minted identity, value-equality semantics. It round-trips the repeating group losslessly and mints zero unwarranted individuals — which is the publish-first answer when the source carries values, not entities.

## Q2 — Building: class now? — **REJECT**

REJECT. A `opda:Building` class would have **zero instances** at ship: no leaf in the 239 dereferences a building as a thing, the eventual IC consumer (ODR-0005 §3a hard-case 4 "Replacement") names a *building* but is **latent and unexercised** — nothing in BASPI5 today exercises building-vs-Property divergence. `numberOfFloors`/`opda:area`/`builtForm` already sit flat on `opda:Property` and answer every building-shaped question losslessly (S026 established this; it still holds). Minting `opda:Building` now is a class with no members and no query — the textbook over-mint. UK-Gov publish-first discipline: you publish the entity when a consumer dereferences it, not in anticipation. The Replacement witness makes the eventual IC *genuine* (I affirm that, with Guarino), which is exactly why the zero-cost re-home path matters — but genuine-eventually is not warrant-now.

## Q3 — Mereology / attachment (transitivity?) — **REJECT (no mereology this session)**

REJECT minting `opda:hasRoom`/`opda:partOf` plumbing and REJECT any transitive part axiom. With Room as a by-value node and Building absent, there is **no two-level part hierarchy to be transitive over** — the structure attaches directly to `opda:Property` by a single `opda:hasRoomDimension`-style predicate (matching how `opda:MonetaryAmount` attaches: one bearer, one object property, value node at the end). Transitivity is a property of a part-of chain; we have no chain. Asserting `owl:TransitiveProperty` over a one-hop attachment buys nothing and risks the classic deployment failure — a transitive part-of that later entails room⊑property⊑estate inferences nobody validated. **No mereology, no transitivity; a flat by-value attachment on `opda:Property`.** If Building ever promotes, the part predicate is minted *then*, with its characteristics decided against a real query — not pre-wired against a node that does not exist.

## Q4 — length/width/roomName (units; roomName non-rigid) — **AFFIRM**

AFFIRM. `length`/`width` are measured Qualities and need a **length unit** — and the brief is correct that `UnitOfAreaScheme` does **not** cover them (it is area, not length). The deployment-honest move is to mint/seed a `UnitOfLengthScheme` (metre as the seeded default, extensible — mirroring the `CurrencyScheme`-GBP-default idiom in ODR-0024 R3) and attach unit on the value node, OR carry a `sh:datatype` + explicit-unit convention; either ships and round-trips. `roomName` is correctly flagged **non-rigid** (ODR-0024 R10 already says so): "Bedroom 2" is a label, not an identity principle — it must be a plain `xsd:string` annotation on the value node, never a key, never an IRI fragment. This is the load-bearing reason Room is a *value* and not a *Kind*: its only candidate name is non-rigid, so there is nothing to identify it *by*. The geometry (`length`×`width`) is the payload; the name is decoration.

## Q5 — Realisation + minimality — **AFFIRM (minimal realisation; one anonymous value node, no class, no scheme beyond length-unit)**

AFFIRM minimality. The complete shipping unit is: **one anonymous by-value node** (`opda:RoomDimension`) with `opda:length`/`opda:width` (length-unit-bearing) + `opda:roomName` (`xsd:string`, non-rigid), attached to `opda:Property` via one object property, with a SHACL node-shape making `length`/`width` present and constraining the unit — exactly the `opda:MonetaryAmount` shape (required magnitude + scheme-restricted unit) from ODR-0024 R3. **Zero classes minted, zero IRIs for rooms, one new length-unit scheme.** This closes the last 3 of 239 (full Category-G coverage 236→239) without minting a single individual the data cannot identify. Engaging **Cagle (DA)**: this is reference-shaped data carried as a value, not a class — your "reference data, not classes" instinct and my publish-first instinct converge here. **Allemang**: this is the "keep it a structured datatype, don't over-promote" answer, with the one concession that the repeating group forbids *flat* properties — it must be a value *structure*. **Knublauch**: the whole commitment is a SHACL node-shape over a lean anonymous node; no OWL class axioms, no reasoner obligations — the shape *is* the contract.

## Cut list (what ships this session — the minimum)

1. **Mint `opda:RoomDimension` as a by-value structure** (anonymous node, `opda:MonetaryAmount` precedent): `opda:length` + `opda:width` (length-unit-bearing) + `opda:roomName` (`xsd:string`, non-rigid). **No IRI minting for room instances.**
2. **One object property** `opda:hasRoomDimension` (working name), `rdfs:domain opda:Property`, `rdfs:range opda:RoomDimension`. One hop, no part-hierarchy.
3. **Seed `opda:UnitOfLengthScheme`** (metre default, extensible — `CurrencyScheme`/GBP idiom). `UnitOfAreaScheme` is untouched (it does not cover length).
4. **One SHACL node-shape** on `opda:RoomDimension`: `length`/`width` required; unit `sh:in` the length scheme. Mirrors the MonetaryAmount shape.
5. **Coverage** closes 236→239 (the 3 held `roomDimensions.rooms[]` leaves). ODR-0024 R10's hold is **discharged-as-value**, not promoted-as-class.

**Do NOT mint:** `opda:Room` class · `opda:Building` class · `opda:hasRoom`/`opda:partOf` mereology · any `owl:TransitiveProperty` · any building-level scheme · any room identity key. None has a member, a query, or a re-identification token in BASPI5.

## Withdrawal conditions (tied to real source-data changes — promote Room/Building only when the source changes, not when the model wants to)

- **Promote `opda:Room` to a Kind** ⟸ a future BASPI5/PDTF revision introduces a **stable room token** (`roomId`, persistent room name, or survey-stable slot key) AND a named consumer query **re-identifies a room across two dated surveys** by that token. The S026 *re-identification* trigger, restated against the data: no token in the source ⇒ no class. The day the source mints a token, the by-value node re-homes onto a minted `opda:Room` with that token as its key — cheap, because the geometry properties already exist on the value node.
- **Promote `opda:Building`** ⟸ a leaf appears that **dereferences a building as an individual distinct from the Property** (e.g. multi-building title, building-level UPRN, or a building survey carrying building-scoped identity) AND the ODR-0005 §3a "Replacement" divergence is **exercised by a real query** (building persists while Property identity changes). Until a leaf forces building≠Property in the data, the flat `numberOfFloors`/`opda:area`/`builtForm` on `opda:Property` stand.
- **Add mereology/transitivity** ⟸ *both* Room and Building exist as Kinds (above) AND a query traverses Building→Room part-of. Mint the part predicate and decide its characteristics then, against that query — never pre-wired.
- **Standing discipline:** each withdrawal condition names a **source-data change** (a new token, a new leaf, a multi-building artefact), not a modelling preference. Absent a source change, the by-value cut is the published answer. This is the publish-first contract: the source grows the entity; the model follows.
