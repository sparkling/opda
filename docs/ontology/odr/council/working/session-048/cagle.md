## Opening positions

### Q1 — Disjoint domain/range form: (a) repeated `rdfs:domain` "any-of" + `sh:or`, (b) `owl:unionOf` + `sh:or`, or (c) honest SHACL-only?

**Position: REJECT (b). Prefer (c) SHACL-only; (a) only as a documentation-comment fallback, never as an `rdfs:domain` triple. Ballot: AGAINST.**

Grounded reasoning:

1. **SHACL already carries the *exact* semantics — by OPDA's own ruling.** ODR-0029 §R3 states the `domain`/`range`-as-SHACL pattern "carries the **exact** semantics of `rdfs:domain`/`range` evaluated closed-world." When two layers carry the *exact same* semantics and only one is evaluated, the unevaluated one is not "a second layer of meaning" — it is a redundant copy with an independent failure mode (drift). This is the SHACL-first thesis I have argued in print: *"Validate the shape, document in the shape"* — the node/property shape with `sh:or ( [ sh:class :Person ] [ sh:class :Organisation ] )` IS the machine-legible, human-legible, tool-legible statement of the bearer disjunction (W3C SHACL Rec, §2.1 NodeShape / §4.6.1.2 `sh:or`). A `sh:or` over `sh:class` is read correctly by *every* SHACL processor; an LLM reads `sh:or (Person Organisation)` as "either" at least as naturally as it reads a domain triple.

2. **(b) `owl:unionOf` is affirmatively dangerous in OPDA — it is on the *excluded* list precisely because it invites the reasoning OPDA opted out of.** ODR-0025 §R2 lists `owl:unionOf`/`intersectionOf`/`complementOf` as forbidden from the closure: *"DL classification; existential/universal reasoning; outside RL."* Authoring a blank-node `owl:unionOf` class expression as "documentary" plants a live DL trap in a corpus that today has **zero** blank-node class expressions. The moment any external tool — or a future maintainer who forgets the doctrine, which is *the exact session-047 failure mode this ADR claims to cure* — points a reasoner at the class graph, `rdfs:domain [ owl:unionOf (Person Organisation) ]` is sound, evaluable OWL and will classify. (a)'s "any-of" reading is *not* even standards-correct: under RDFS 1.1 §2.3.1 multiple `rdfs:domain` is **conjunction** — every subject is inferred to be *all* listed classes at once. ODR-0025 §R2 names this verbatim: *"a property with N declared domains infers each subject is all N classes at once."* So (a) ships a triple whose *only* correct standard reading is the intersection bug, relabelled "any-of" by local convention. That is not documentation; it is a documented landmine.

3. **The AI-signal claim is the weakest link and it cuts the other way.** The premise is "OWL axioms are machine-legible documentation SHACL doesn't carry." But for disjunctions the SHACL *is* the carrier and it is more precise: `sh:or` is unambiguously disjunction; `rdfs:domain × N` is ambiguously intersection-relabelled-as-union. An AI grounding on the asserted graph that reads a domain triple gets the *wrong* reading unless it also reads the doctrine ODR. We would be authoring a signal that is correct only if the reader has already internalised a non-standard convention — i.e. not a signal, a shibboleth.

**If the council insists on a documentary surface**, the honest form is an `rdfs:comment` / `skos:scopeNote` in prose ("bearer is Person or Organisation; enforced in SHACL `sh:or`") — which is exactly what `opda:playedBy`'s comment already does at `opda-agent.ttl:239`. That is real documentation with zero entailment surface. Authoring it as `rdfs:domain` triples buys nothing the comment doesn't and adds the intersection-misread risk.

**Disposition placeholder:** HOLD pending peer rebuttal; will reduce to a fallback ranking (c) > (a) > (b) if the council rejects pure SHACL-only.

### Q2 — Author `owl:FunctionalProperty` / `InverseFunctionalProperty` as documentary AI-signal (+ `sh:maxCount` / `dash:uniqueValueForClass`)?

**Position: REJECT. Author domain/range-equivalent constraints in SHACL only. Ballot: AGAINST.**

Grounded reasoning:

1. **These are the two constructs OPDA singled out as the most dangerous, by name, twice.** ODR-0025 §R2 excludes `owl:FunctionalProperty`/`InverseFunctionalProperty` with the rationale *"Trigger `owl:sameAs` identity-merge"*, anchored to ODR-0005 §R5 (no `owl:sameAs` — "catastrophic in master data"). ODR-0026 §R2 repeats the exclusion. Authoring `owl:InverseFunctionalProperty` on an identifier (NI number, LEI, Companies House number — exactly the "identifying value is an ontological fact" case the proposition targets) is authoring the *premise* of an `owl:sameAs` merge as a standing, evaluable axiom. In a master-data ontology for property transactions, two records that share an IFP value get merged by any conformant OWL reasoner — the single most destructive operation possible on this data. ADR-0049 calls this "documentary." It is documentary *only* under OPDA's opt-out; the axiom itself is a loaded gun pointed at the data, safetied by nothing but a doctrine that the same ADR admits was forgotten once already.

2. **`dash:uniqueValueForClass` is the correct, evaluated, scoped tool — and it already exists.** The uniqueness-of-identifying-value fact has a purpose-built SHACL constraint: DASH's `dash:uniqueValueForClass` (TopQuadrant DASH constraint-components vocabulary; the documented convention is "no two focus nodes of the given class share a value for this property"). That is *exactly* InverseFunctional scoped to a class, validated closed-world, with no `owl:sameAs` merge — it flags the collision instead of silently fusing the two entities. For single-valuedness, `sh:maxCount 1` is the Functional analogue (W3C SHACL Rec §4.2.1). OPDA already uses `sh:maxCount 1` for identity-key surfaces (`opda-agent-shapes.ttl:148,167,211`). So the constraint is present, evaluated, and safe. Adding `owl:FunctionalProperty`/`owl:InverseFunctionalProperty` on top adds *nothing enforced* and re-arms the merge.

3. **"Scoped to ontological-fact single-valuedness" is not a scope a reader can see.** The proposition would have us author the characteristic only where single-valuedness is "an ontological fact." But the axiom in the TTL carries no marker of that intent — `owl:FunctionalProperty` reads identically whether the modeller meant "ontological fact" or "current data shape." The scope lives in a human's head and a council record, not in the artefact. That is precisely the gap ADR-0049 §Context says caused session-047. We would be re-creating the disease while prescribing it as the cure.

**Disposition placeholder:** HOLD; this is my strongest dissent — the `owl:sameAs`-merge re-arm is a category difference from domain/range, not a degree.

### Q3 — Author `owl:disjointWith` as documentary axiom (+ `sh:not`/`sh:xone`) under a scoping rule?

**Position: REJECT as OWL axioms; the disjointness *check* already exists and should stay SHACL/consistency-gate only. Ballot: AGAINST.**

Grounded reasoning:

1. **OPDA already *checks* disjointness without authoring a single `owl:disjointWith`.** ADR-0035's negative consistency gate detects disjointness violations as a validation step (ODR-0025 §R1: *"A disjointness consistency check is permitted but is a validation gate, not an entailment — it produces no domain triples"*; ODR-0029 §R1). The data-quality stake the proposition cites is *already covered*. So Q3 is purely: should we *also* author the axiom? The answer is no — it is the disjointness analogue of Q1: a second copy of an already-enforced fact, with a live DL surface (`owl:disjointWith` + a shared instance entails `owl:Nothing` / inconsistency under OWL-DL — W3C OWL 2 Primer §3.1).

2. **The scoping rule's own exclusion proves the hazard is real and adjacency-close.** The proposed rule says "NEVER between anti-rigid roles sharing a bearer's identity." But OPDA's relationship layer is *built on* exactly such roles — Seller/Buyer are `opda:RoleMixin`, Proprietor is `opda:Role`, all anti-rigid, all sharing the bearer's identity (`opda-agent.ttl:239` documents this disjunction reasoning). The carve-out exists because the very next disjointness someone would naively author (Seller disjoint Buyer? Person disjoint Organisation as bearers?) is *wrong* under UFO and would falsify legitimate co-typing and co-bearing. A scoping rule whose first clause is "don't apply it to the things this ontology is mostly made of" is a rule that mostly says "don't." `sh:xone` / `sh:not` already let you forbid an *unwanted* co-typing exactly where you mean it, closed-world, instance-by-instance (W3C SHACL §4.6.2 `sh:xone`, §4.6.1.1 `sh:not`), without arming a TBox inconsistency.

3. **`owl:disjointWith` is the single worst "documentary" axiom for AI grounding.** An LLM reading `Person owl:disjointWith Organisation` will treat it as a hard partition and refuse to reason about an entity that is legitimately *both* under a different facet (OPDA's whole classification-over-inheritance doctrine — facets, not subclass trees). The signal actively misleads the consumer the proposition is trying to serve.

**Disposition placeholder:** HOLD; the existing consistency gate already delivers the data-quality value the proposition seeks, with no TBox-inconsistency surface.
