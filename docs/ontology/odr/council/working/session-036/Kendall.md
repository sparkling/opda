## OPENING

### The FIBO lens, stated up front

FIBO does **not** "always classify." FIBO runs SKOS classification schemes (e.g. instrument-classification, entity-type code lists) *alongside* OWL class hierarchies, and the choice between them is principled, not stylistic. The OMG/EDMC guidance I have argued for years (and in *Demystifying OWL for the Enterprise*, Uschold 2018, which I edited the series around; and my own *Ontology Engineering*, Kendall & McGuinness 2019, §on classification-vs-subsumption) is: **a subclass earns its place when downstream logic must quantify over the kind as a type — when the kind carries distinct relational structure (its own properties/restrictions), participates in disjointness/covering axioms that do work, or is the target of class-scoped constraints that cannot be re-expressed as value constraints.** Otherwise a coded value on one class is the lower-coupling, higher-governability choice. So I am sympathetic to the directing authority's *prior* — "prefer classification, default away from subclass proliferation" — but a **blanket** "always" is as wrong as "always subclass," and I will say so in Q1.

### The empirical keystone I verified (decisive for Q2)

opda's **own flagship "Substance Kind label" precedent already prefers classification over inheritance** — and ODR-0011 §8a's prose overstates what was built:

It is a **TRIPLE inconsistency** — I found it; **Allemang independently verified all three legs against the artefact** (his discipline: a teammate file/line claim gets checked, not trusted) and sharpened it past my original two-leg statement:

(Correct modelling outcome; live contradiction with the rule-text.)
- **LEG 3 — the property comment re-asserts the false promise.** `opda:tenureKind` (`opda-property.ttl:696–702`) is `owl:DatatypeProperty` + `rdfs:domain LegalEstate` + SHACL `sh:in ("Freehold" "Leasehold" "Commonhold")` (`opda-property-shapes.ttl:96–97`) — a coded-only facet; yet its `rdfs:comment` (`:699`) says "Each member binds to its corresponding OWL sub-class via `skos:exactMatch` per ODR-0011 §8a." It does not.

**The artefact got the MODEL right and the RULE-TEXT/DOCUMENTATION wrong.** `tenureKind` fails earn-its-keep on **both** conjuncts (Allemang): no Freehold-only property (Δ fails) AND a freehold→leasehold conversion is the *same* legal estate (anti-rigid → fails the R+ gate; a tenure subclass would be a rigidity violation). **Crucially (Allemang's cargo-cult point): the false unconditional MUST is not inert — it is an *active hazard* that would instruct future modellers to mint the very empty subclasses ODR-0024 §R10 (`opda:Building`) refused.** So adopting the principle does not merely *generalise* §8a — it *forces a correction of a live §8a defect.*

**So opda's de-facto rule is already the directing authority's rule.** Where a kind-distinction is a *label over one Substance Kind* and carries no distinct relational structure, opda emits a **coded value with a SHACL `sh:in` gate on the one bearer class**, and *does not* build the subclass tree — even when an ODR's prose anticipated one. `tenureKind` is the proof: tenure-change is named as a hard case in `LegalEstate`'s own IC (`opda-property.ttl:54`), yet it is handled by *changing a coded value*, not by *re-typing across a subclass migration* — exactly because tenure subclasses would carry no properties Freehold-but-not-Leasehold and would make tenure-change a rigidity violation (an estate cannot change its `owl:Class`). This is precisely the load-bearing test applied correctly, and it landed on classification.

This finding **reframes session-036**: the question is not "should opda adopt a new rule" — opda already follows it for `tenureKind`/`ownerType`. The question is whether **evidence is the exception** (a place where inheritance *is* load-bearing) and whether ODR-0011 §8a's overstated prose should be corrected to match the practice.

### Q1 — The principle

**AFFIRM, as a default with a sharp load-bearing test — REVISE the wording to reject "always."** Codify in **ODR-0011 §8a** (it already owns the Substance-Kind-label↔subclass binding rule and is where the `tenureKind` overstatement lives), cross-referenced from ODR-0008 §Q5a. Not a brand-new ODR — the rule is an amendment to existing pattern-ODRs, and minting a new top-level principle ODR for something three ODRs already half-state would itself violate the simplicity the principle preaches.

**The load-bearing test (a subclass is mandatory, not a coded value, when ANY holds):**
1. **Distinct relational signature** — the sub-kind is the `rdfs:domain`/`rdfs:range` of ≥1 property the siblings cannot bear (a real disjoint property profile, not a descriptive facet that merely happens to co-occur). *FIBO test: does the subtype anchor its own object-property edges?*
2. **Distinct UFO sort** — the sub-kind is a *different fundamental ontological category* from its siblings (e.g. one is a Relator and the others are Information Objects), so collapsing them onto one class would assert a false common nature. *(OntoClean/UFO: Guizzardi 2005 Ch.4; Guarino & Welty 2009.)*
3. **Class-scoped axiom that does work** — the sub-kind participates in a disjointness, covering, or equivalence axiom that the entailment regime *evaluates* and that changes an answer-set or a validation report. *(In opda specifically, ODR-0025/0026: an axiom the closure never evaluates fails this test by definition — it is documentation, and documentation is better carried by `skos:exactMatch` + `rdfs:comment`.)*
4. **Rigidity-safe** — instances do **not** migrate between the sub-kinds over their lifetime (else a subclass forces an unlawful `owl:Class` change; a Phase/Role/coded-value is the correct vehicle). *(This is what kills tenure subclasses and what must be checked for evidence.)*

A kind-distinction failing all four → **classification** (coded SKOS facet + `sh:in` on the one bearer Kind, `sh:targetSubjectsOf` the facet property so enforcement is entailment-free — opda's `ownerType`/`tenureKind` idiom). Passing ≥1 → inheritance is admissible *for that distinction*, but the test is applied **per sub-kind**, not to the family as a block.

*Citation grounding:* ODR-0011 §8a (the existing Substance-Kind-label↔subclass binding rule, and the `tenureKind` precedent that already realises it as classification); ODR-0024 §R4 (range-less generic object-property-per-value = "namespace landmines" → coded value — the established opda precedent against class/property-per-value); ODR-0024 §R5 (the structural C-vs-G categoriser: an enumerated value-space is a *coded range*, not a class tree); ODR-0008 §Q5a (datatype+`sh:in` for non-§8a enums; SKOS scheme for §8a labels — already a classification-first table); FIBO practice (SKOS code lists alongside OWL); Kendall & McGuinness, *Ontology Engineering* (2019), classification-vs-subsumption.

## Refinement (verified after opening — the one honest complication to my own position)

I checked whether `DocumentEvidence`'s `⊑ opda:AttachedDocument` gives it a distinct property signature `ElectronicRecordEvidence` lacks. **It does** — and this is the strongest counter to a clean "collapse Doc+ER together":

- The filing-metadata properties `opda:documentDate` / `documentTypeCode` / `filedUnder` / `retrievedOn` are real (`opda-descriptive.ttl:276–670`), all `rdfs:domain opda:AttachedDocument`. `DocumentEvidence ⊑ AttachedDocument` (`opda-claim.ttl:53`) inherits them; `ElectronicRecordEvidence ⊑ opda:Evidence` **only** (`:63`) does not. So Document and ElectronicRecord are **not** symmetric after all — Document carries a (documentary) inherited signature.
- **But three things bound how much this rescues inheritance:** (a) it is `rdfs:domain`, which under ODR-0026 §R2 is **documentary, not evaluated** — fails my test 3, and does not SHACL-constrain instances at the subclass level (a `sh:property` shape would). (b) The signature flows from `AttachedDocument` ("is it a filed document?"), **not** from anything *evidential* — it distinguishes "filed-document evidence" from "API-record evidence," which is exactly the acquisition-method axis the `opda:evidenceType` facet already codes. (c) Confirmed across the **whole** tree: **no property is `rdfs:domain opda:DocumentEvidence` or `rdfs:domain opda:ElectronicRecordEvidence`** — the only evidence subclass with its *own* declared property is `VouchEvidence` (`attestedBy`). `ElectronicRecordEvidence` has **zero** own or inherited distinguishing properties — it is a pure type-tag, structurally identical to `opda:Evidence`.

**This sharpens (not weakens) my ruling:**
- `VouchEvidence` — keep (own property + distinct sort). Unchanged.
- `DocumentEvidence` — its load-bearing content is the **`⊑ AttachedDocument` subsumption** (ODR-0024 §R7's ratified document-bearer hook), NOT a sibling-distinguishing *evidence* axiom. The right model is: `DocumentEvidence` survives **as the RoleMixin intersection "AttachedDocument playing the Evidence role"** (which it already is, `:53`) — that subsumption is genuine and load-bearing for the document-bearer lineage. It is *not* a peer of a structureless `ElectronicRecordEvidence`.
- `ElectronicRecordEvidence` — the **weakest** case in the whole family: a pure type-tag, no own/inherited property, same UFO sort as Document. This is the one that most clearly should be **facet-only** (`opda:evidenceType = Electronic-Record` on `opda:Evidence`) unless the unbuilt §R5 `record.source.name` obligation gives it a real signature.

So the honest per-subclass result is **not** "1 (Vouch) vs 3" and **not** "collapse Doc+ER as a pair" — it is: **Vouch (keep, distinct sort) · Document (keep *as the AttachedDocument∩Evidence role*, its load-bearing fact is the bearer-subsumption) · ElectronicRecord (facet-only candidate — the genuinely inert tag).** The directing authority's prior cuts cleanest against `ElectronicRecordEvidence` specifically.
