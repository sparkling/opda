# Session-036 — Isaac/Miles working position (SKOS concept-vs-class boundary)

**Lens:** Antoine Isaac / Alistair Miles, W3C SKOS Reference editors. The concept-scheme vs OWL-class boundary; `skos:broader` ≠ `rdfs:subClassOf`; the §3.5.1 "no statement" rule; the mapping properties (§10). 

**Proposition (directing-authority):** *"Always prefer classification over inheritance where you can."*

---

## OPENING

### The SKOS-grounded reading of the directive

The directive is **sound as a stated default, but its word "always" must be read as a rebuttable presumption, not an absolute** — and SKOS itself is the authority for *why* it is rebuttable. I take "classification" to mean: model the discriminator as a **controlled, dereferenceable value-space** — a `skos:ConceptScheme` whose members are `skos:Concept`s — and key behaviour off *membership in that value-space* (a coded value on a property), rather than off **`rdfs:subClassOf` position in a TBox subsumption tree**. Read that way, the directive is **the exact thesis of ODR-0011** (every closed JSON `enum` → a `skos:ConceptScheme`, *never* an `owl:oneOf` bag, *never* a subclass tree; ODR-0011 §Decision, §Alternatives). It is already corpus law for ~40 schemes. The proposition is therefore **AFFIRMED at the level of the default** — but the precise governed answer the SKOS Reference compels is richer than "classification *instead of* inheritance," and that precision is the decisive contribution of this lens.

### The §3.5.1 test — the spec forbids the strict either/or

The single most load-bearing fact for Q1 is **SKOS Reference §3.5.1** (heading: "SKOS Concepts, OWL Classes and OWL Properties"): the SKOS data model **makes no statement** about whether a `skos:Concept` is, is not, or may be an `owl:Class` — verbatim, *"this specification does not make any additional statement about the formal relationship between the class of SKOS concepts and the class of OWL classes."* §3.5.1 goes further and gives a worked example of a resource asserted to be **both** a `skos:Concept` and an `owl:Class`, stating explicitly that this is **"consistent with the SKOS data model."** A resource may be both without contradiction. (This is the well-known "classes-as-instances" / metamodelling accommodation; SKOS deliberately declines to police it, which is *why* OWL-class-aligned schemes coexist with concept-only schemes in practice.)

Therefore **"classification *vs* inheritance" is a false dichotomy** at the level of the formalism. The governed answer to "prefer classification" is very often **not** "drop the class" but **"a `skos:ConceptScheme` `skos:exactMatch`-bound to the OWL class(es)"**: classification *for the value-space / glossary / form-control layer*, the OWL class *for the structural / constraint-bearing layer*, with the two kept in lockstep by a **mapping property** (`skos:exactMatch`, SKOS §10) — **NEVER `owl:sameAs`** (ODR-0011 §8a; ODR-0005 Anti-pattern §5). This is exactly what FIBO does and exactly what `opda:TenureKindScheme` / `opda:OwnerTypeScheme` / `opda:PropertyTypeScheme` already do: a "Substance Kind label" scheme whose every member `skos:exactMatch`-es its OWL sub-class.

### The decisive boundary — `skos:broader` is NOT `rdfs:subClassOf` (SKOS Primer §2.3.1)

The error the directive is *really* guarding against has a precise name in our own literature. **SKOS Primer §2.3.1** and **SKOS Reference §8** (Semantic Relations) are explicit: `skos:broader`/`skos:narrower` assert a **thesaurus-style generalisation between *concepts*** and carry **no formal entailment** — a narrower concept is *not* inferred to be a subclass, an instance, or a part of its broader concept; the relation is deliberately weak (it is not even asserted transitive — `skos:broaderTransitive` is the separate super-property). `rdfs:subClassOf`, by contrast, licenses **instance-level subsumption**: every member of the subclass is entailed a member of the superclass, and the subclass inherits every domain/range/property-restriction of the parent.

So the **concept-vs-class test** I put to the council is a *consequence test*, not a taste:

> **Use a `skos:Concept` in a scheme (classification) when the discriminator is a value in a closed/curated value-space and you want NO instance-level subsumption or property-inheritance consequences from picking one value over another.** Picking "Detached" over "Semi-detached" must entail nothing about an instance beyond "its built-form quale = Detached."
>
> **Use a true `rdfs:subClassOf` (inheritance) only when the subtype is an OWL class that bears OWL-level structure the value cannot carry** — a *distinct property signature* (domain/range/cardinality the siblings do not share), *its own data-realisable identity criterion*, or membership-reasoning a consumer actually exercises. A subtype that carries none of these is a concept masquerading as a class.

This test is **already operationalised in the corpus** as ODR-0008 §Q4a's three-criterion class-promotion gate (promote to a Class iff (a) authority-retrieved provenance, (b) distinct lifecycle, or (c) distinct PII regime). The negative cases prove the directive: `opda:Room` (8–0) and `opda:Building` (7–1) were **refused** class status in session-030 precisely because they had **no data-realisable IC** — they collapsed back to a scheme value or an anonymous by-value structure. That is "prefer classification" enforced by identity criterion, not by fiat. And `opda:PerilScheme`'s own scope note states the rule verbatim: *"a delimited value-space is a scheme, not a class tree … NEVER an `rdfs:subClassOf` hierarchy."*

### Q2 — Evidence: the session-035 dual layer is RIGHT; concept-only would be WRONG

Applying the consequence test to evidence settles it, and it settles it **for the dual layer that session-035 already emitted** — not for concept-only, and not for subclass-only.

Run the three discriminators through the test:

- **The OIDC4IDA `evidence.type` axis {Document, Electronic-Record, Vouch} is a closed, regulator-cited value-space.** Picking "Document" is a *value choice with no inheritance consequence of its own*. By the test → **classification**. This is `opda:EvidenceMethodScheme` carried by the `opda:evidenceType` facet. Correct. (And note: it was *minted as a scheme from the start*, S009 — the scheme was never the defect; the defect, retired at S035, was the parallel `owl:equivalentClass` short-name alias, which is a *different* construct — see below.)
- **But the three evidence subtypes are NOT mere value-distinctions — they carry disjoint OWL property signatures and divergent identity.** `opda:attestedBy rdfs:domain opda:VouchEvidence` (`opda-claim.ttl:121`) exists on Vouch and **nowhere else**; `opda:DocumentEvidence rdfs:subClassOf opda:AttachedDocument` (`:53`) pulls in the document filing-metadata signature Electronic-Record and Vouch do **not** bear. **`VouchEvidence` is categorially a different fundamental thing** — an Agent-founded attestation Relator (`a opda:Relator`, `:92`; `prov:wasAttributedTo` an Agent, two-relata dependence), not an Information-Object sibling of the document evidences. A value on a single facet **cannot** carry "this kind, and only this kind, has an `attestedBy` join to a `prov:Agent`." By the test these subtypes **pass the inheritance warrant** — distinct property signature + (for Vouch) distinct UFO category. → **keep the subclasses.**

So evidence is the case where **both layers are warranted and the answer is the §3.5.1 dual layer**: the `EvidenceMethodScheme` concepts for the governed, extensible, regulator-cited value-space + the `…Evidence` OWL subclasses for the disjoint structure, **bridged member-for-member by `skos:exactMatch`** (`DocumentEvidence skos:exactMatch …/evidenceMethod/Document`, `:55`; etc.). I **verified all three `skos:exactMatch` links are emitted** (`opda-claim.ttl:55,65,98`) against the three scheme concepts (`opda-vocabularies.ttl:1108–1130`). This is **textbook ODR-0011 §8a "Substance Kind label"** and it is correct as it stands.

**Concept-only (drop the subclasses) would be WRONG** for a SKOS-specific reason beyond the property-signature loss: it would **split one concept scheme across two membership disciplines** — some members concept-only, others concept-plus-subclass — and it would **orphan the ODR-0009 per-subtype obligations** onto a value the SHACL must re-discriminate by hand. SKOS's own integrity (§3.5.1 permits dual; it does not *require* dissolving the class) gives no warrant to delete a class that bears real OWL structure. The scheme is the *value-space*; it is not a replacement for a class that has a property signature.

**Refinement from cross-talk (Guarino's adjudication — the warrant is per-MEMBER, and the rigid anchor lives in the bearer, not the anti-rigid evidence-role).** My opening said "keep the subclasses" flatly; that is right for Document and Vouch but over-stated for ElectronicRecord, and Guarino's split is sharper:
- **DocumentEvidence:** `⊑ AttachedDocument ⊓ Evidence` is a *role∩bearer intersection* over the **present** rigid Information-Object Kind `opda:AttachedDocument` (`:32`); the rigid backbone is there, the disjoint filing signature is real → settled-keep.
- **VouchEvidence:** the rigid anchor IS the `opda:Relator` (rigid by sort) + the +D two-relata Agent↔Claim dependence → settled-keep.
- **ElectronicRecordEvidence:** has **no distinct property signature today AND no minted neutral bearer** (`opda:ElectronicRecord` does not exist — the S035 Finding-5 asymmetry). This is the case my Δ-only test, alone, would wrongly demote to concept-only. The resolution (which I accept): its IC is **pending, not absent** — it is the IC of the not-yet-minted `opda:ElectronicRecord` bearer (source-register + retrieval-activity individuation). The decisive SKOS point, mine to make: **§3.5.1 permits a class to coexist with a concept but nowhere COMPELS dissolving a class; and a class whose IC is *pending* is categorially distinct from a −I value-distinction with NO IC.** Concept-only demotion is correct only for the latter. So ElectronicRecord is **documentary-retained under ODR-0026 §R2** (OWL class kept unevaluated, zero closure cost, as the placeholder), with the **§R7-bearer-mint as the realising trigger** (minting `opda:ElectronicRecord` supplies the rigid anchor, witnesses the IC, and grounds the `record.source` Δ in one move). This is the +I disjunct from Q1 doing its work: identity (present-or-pending) trumps property-signature parsimony.

### Q1 (the principle) and the alias distinction — a synonymy need, not a classification question

There is a subtlety the council must not blur: the construct session-035 **retired** was **not** a subclass and **not** a misuse of classification. It was `owl:equivalentClass` aliases (`opda:DocumentEvidence ≡ opda:Document`) — a **controlled-vocabulary *synonymy* need ("Document" is an alternate name for "Document Evidence") mis-encoded as a TBox identity axiom.** SKOS Reference §5 (Lexical Labels — the `skos:prefLabel`/`skos:altLabel`/`skos:hiddenLabel` vocabulary; an `altLabel` is exactly "an alternative lexical label for a resource") is the correct vehicle, and that is exactly the remedy now in the artefact (`skos:altLabel "Document"`, `:54`). This is a **fourth** category distinct from all three of the directive's terms: not inheritance, not classification, but **labelling**. Recording it sharpens the principle — "prefer classification over inheritance" must be stated alongside "and prefer *labelling* (`skos:altLabel`) over *identity axioms* (`owl:equivalentClass`/`owl:sameAs`) for mere synonymy." The corpus already lives both halves.

## VERDICTS (FINAL — post-cross-talk with Baker + Guarino; both threads converged)

- **Q1 — AFFIRM — ballot: FOR.** "Prefer classification over inheritance" is AFFIRMED *as a rebuttable default* and is the standing thesis of ODR-0011 (closed enum → `skos:ConceptScheme`, never `owl:oneOf`/subclass tree). The SKOS concept-vs-class **test** is a *consequence test* (SKOS Reference §3.5.1 — the model makes "no statement" forcing one over the other; a resource asserted both `skos:Concept` and `owl:Class` is "consistent with the SKOS data model"; SKOS Primer §2.3.1 / SKOS Ref §8 — `skos:broader` carries no subsumption, `rdfs:subClassOf` does). **Canonical statement (the +I disjunct made explicit, per Guarino's correction of my too-narrow opening):**
  > **Use a `skos:Concept` (classification) when picking the value entails NO subsumption, NO distinct property-inheritance, AND NO distinct re-identification under a separate identity criterion. Use a true `rdfs:subClassOf` (inheritance) when the subtype bears a distinct property signature, OR a distinct identity criterion (+I — including a *pending* one located in a latent bearer Kind, held documentary under ODR-0026 §R2, not dissolved), OR exercised membership-reasoning.**

  The +I disjunct is decisive and must not be dropped: `opda:Person`/`opda:Organisation` share a `name`/`address` signature yet have distinct ICs (biological-continuity vs legal-charter persistence) — the narrow Δ-only test would wrongly collapse them to one class + a `personhoodKind` facet; +I rightly keeps two Kinds. **Cite Guarino (OntoClean +I/+R) + ODR-0005 §2a** (Substance Kind = Sortal, Rigid, supplies own IC — the IC is the test, independent of property count). The default and the exception **co-vary across almost the whole corpus** (`builtForm`/`tenureKind`/`ownerType` and every Quale-in-Region/Phase scheme are genuinely −I → correctly concepts/facets); they diverge only at the thin-Δ-but-+I seam. Operationalised already as ODR-0008 §Q4a (the Room 8–0 / Building 7–1 refusals = "prefer classification" enforced by absence-of-realisable-IC). **Labelling corollary:** `skos:altLabel` (SKOS §5), never `owl:equivalentClass`/`owl:sameAs`, for synonymy — a *fourth* category distinct from the directive's three terms.

  **Subclass-worthiness is per-MEMBER, not per-set (Guarino's adjudication — guards the directive against an over-retention misread).** A closed/regulator-fixed value-space confers *zero* subclass warrant by itself (`tenureKind` is closed and −I → facet-only); **closed-set completeness is the WRONG justification for retaining a subclass, and "rigid origin" is only the +R gate, not the warrant — the warrant is +I (distinct individuation).** Guarino's signed sentence, cite verbatim: *"Subclass-worthiness is per-member, not per-set: each member must independently clear +I. A closed/regulator-fixed value-space confers no subclass warrant by itself (tenure is closed and −I → facet-only). Where a member's +I is genuine but its distinguishing property signature is not yet emitted (latent Δ), identity trumps property-signature parsimony and the subclass is retained as documentary (ODR-0026 §R2) — on the member's own latent IC, never on set-completeness."*

- **Q2 — REVISE — ballot: FOR (the session-035 dual layer; REJECT "facet instead of inheritance" and REJECT blanket concept-only).** Evidence is the case where the default is *rebutted on warrant*, and the warrant differs **by member** — the rigid anchor lives in the bearer (Guarino), not in the anti-rigid evidence-role-subtypes, so resolve per bearer:
  - **DocumentEvidence — dual layer clean, fully warranted.** `⊑ AttachedDocument ⊓ Evidence` (`:53`) is a *role∩bearer intersection* over a **present** rigid Information-Object Kind (`opda:AttachedDocument`, `:32`, §R7) — it carries the disjoint filing signature a value cannot. Rigid anchor present.
  - **VouchEvidence — warranted; the rigid anchor IS the Relator.** `a opda:Relator` (`:92`) is rigid by sort; the +D two-relata Agent↔Claim dependence (`attestedBy`, `:121`) is structure a value-slot categorially cannot hold.
  - **ElectronicRecordEvidence — DOCUMENTARY-RETAINED, not concept-only, with a realising trigger** (Guarino's adjudication, which I accept and which corrects my opening's set-level instinct). It has **no distinct property signature today AND no minted neutral bearer** (the S035 Finding-5 asymmetry — `opda:ElectronicRecord` does not exist). But its IC is **pending, not absent** — located in the not-yet-minted `opda:ElectronicRecord` bearer (individuation by source-register + retrieval-activity). SKOS §3.5.1 permits dual but **nowhere compels dissolving a class**, and a class with a *pending* IC is categorially distinct from a −I value-distinction; so under ODR-0026 §R2 the role-subclass is kept unevaluated at zero closure cost as the placeholder. **Re-open / realising trigger:** mint the neutral `opda:ElectronicRecord` bearer (complete §R7 symmetry), which simultaneously supplies the rigid anchor, witnesses the IC, and grounds the `record.source` Δ.

  Across all three, the OIDC4IDA kind is the `opda:EvidenceMethodScheme`/`opda:evidenceType` **classification**, bridged member-for-member by `skos:exactMatch` (SKOS §10 — a *retrieval-interchangeability* assertion, NOT `owl:equivalentClass`/`owl:sameAs`; co-extension enforced by SHACL `EvidenceTypeValueShape` `sh:in`, never a DL axiom — Baker). Verified emitted: `opda-claim.ttl:55,65,98`; concepts `opda-vocabularies.ttl:1108–1130`. Blanket concept-only would orphan the per-subtype OWL obligations and split one scheme across two membership disciplines.

- **Q3 — AFFIRM — ballot: FOR.** Ratify the session-035 disposition as the worked exemplar of the directive; it is implemented and correct in-tree. **Two refinements adopted from cross-talk:** (i) close the ODR-0009 §Deferred `EvidenceMethodScheme → EvidenceKindScheme` rename as **"no-op — withhold PERMANENTLY,"** not "deferred-pending" (Baker's DCMI/Cool-URIs verdict: an IRI is opaque denotation, not a label; deprecation is reserved for a referent change, which did not occur; ODR-0011 §5a minor-edit; renaming = ODR-0024 §R4 namespace landmine); (ii) add **one concrete artefact delta** — a one-line `skos:scopeNote`/`rdfs:comment` on the scheme recording *"IRI retains the historical 'Method' token by design; the scheme denotes the evidence-KIND axis — do not rename (opaque-IRI discipline, session-036)"* (Baker's safeguard), so a future steward does not "fix" the stale token. The ElectronicRecord §R7-bearer-mint trigger (Q2) is the one substantive open item, correctly held documentary.
