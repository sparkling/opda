### Q1 — The principle, from a SHACL/forms standpoint

The principle holds, with a sharp operational test. A SHACL processor targets in two ways (W3C SHACL Rec §2.1.3): **class-based** (`sh:targetClass`, §2.1.3.1) and **value/predicate-based** (`sh:targetSubjectsOf`/`sh:targetObjectsOf` §2.1.3.3, `sh:targetNode`, implicit). The two are not equivalent under entailment:

- **`sh:targetClass C` is entailment-relative.** §2.1.3.1 defines the target as the set of nodes that are `rdf:type C` *in the data graph* — and the spec explicitly notes (§1.5, "SHACL and RDFS/OWL inference") that whether subclass members are included "depends on whether the data graph is interpreted under an entailment regime." So `sh:targetClass opda:Evidence` matches an `opda:VouchEvidence` instance **only if `rdfs:subClassOf` type-propagation has been materialised**. Under opda's own regime that propagation *is* materialised (ODR-0025 §R1 rule #2) — but a **SHACL-Core consumer who loads `opda-claim-shapes.ttl` without opda's `config/opda-owl-rl-safe.rules` closure gets nothing**: the supertype shape silently no-ops on every leaf-typed instance. ODR-0026 §R2/§R3 and ODR-0013's "open-world/closed-world guard" both warn precisely this: *"SHACL shapes MUST NOT assume `equivalentClass` (or `domain`/`range`) entailment — target the actual asserted types"* (ODR-0026 Consequences). The same hazard attaches to subclass-targeting whenever the instance is typed at a different level than the shape.
- **`sh:targetSubjectsOf opda:evidenceType` is entailment-free.** §2.1.3.3: the target is every subject of a triple with that predicate — a purely syntactic match, no `rdf:type`, no closure. It fires identically in TopBraid, pySHACL, jena-shacl, and a hand-rolled validator, with or without a reasoner.

**The test:** *Does the obligation survive a SHACL-Core consumer with no entailment regime?* If keying on a coded value makes the constraint fire where keying on a class would silently no-op, prefer the value. That is the operational content of "prefer classification" for a SHACL author, and it is exactly why opda already emits `ownerType` (ODR-0024 §R6) and `evidenceType` as `targetSubjectsOf`-value-shapes rather than `targetClass` shapes.

**Forms add a second, independent reason.** DASH generates an editor from a **property shape** (`dash:editor`/`dash:viewer` on a `sh:PropertyShape`, per the DASH vocabulary and ODR-0013 §"DASH rendering"). A coded facet *is* a property shape: `opda:evidenceType` + `sh:in (…)` renders as exactly one `dash:EnumSelectEditor` — the same control the in-tree BASPI5 profile emits 30-plus times for every `sh:in` enum (`baspi5.ttl:142, 193, 222, 254, …`). **A class hierarchy gives DASH nothing to render.** There is no `dash:editor` for "choose your `rdf:type` among four subclasses"; subclass membership is not a property, so the kind-choice is invisible to the form-generator. Under inheritance, an author picks the evidence kind by *minting an instance of the right class* — which no generated BASPI/TA6 form can offer as a field. Classification makes the discriminator a first-class form control; inheritance makes it un-renderable.

`Q1: AFFIRM — ballot: FOR — A coded facet keyed via sh:targetSubjectsOf (W3C SHACL §2.1.3.3) is entailment-free and DASH-renderable as a single dash:EnumSelectEditor (DASH; ODR-0013 §DASH-rendering); class-targeting (§2.1.3.1) is entailment-relative (§1.5) and gives a form-generator no property to render. The operational test: prefer the value where it makes the obligation fire on a SHACL-Core consumer with no closure (ODR-0026 §R2/§R3 "target the actual asserted types"). This is opda's own emitted ownerType idiom (opda-agent-shapes.ttl:47–51).*

### Q2 — Does classification validate per-subtype obligations as well as / better than the subclasses?

This is the heart of it, and the answer is **yes, demonstrably better — and opda already ships the proof in a sibling profile.**

The hard per-subtype obligation is **Vouch ⇒ `attestedBy` a `prov:Agent`** (`opda-claim.ttl:121`, `attestedBy rdfs:domain opda:VouchEvidence`). Two ways to validate it:

**(a) Inheritance (the in-tree `VouchEvidenceShape`):** `sh:targetClass opda:VouchEvidence ; sh:property [ sh:path opda:attestedBy ; sh:class prov:Agent ; sh:minCount 1 ]`. This *works* — but only on a node asserted (or materialised) to be `opda:VouchEvidence`. An author who records `[] a opda:Evidence ; opda:evidenceType "Vouch" ; …` — the coded form the corpus now blesses (S035; `opda:evidenceType` is *"the governed, extensible discriminator that replaced the retired aliases"*, `opda-claim.ttl:111`) — and who omits the leaf type is **never targeted**, so the missing `attestedBy` passes silently. The obligation is keyed to a fact (`rdf:type VouchEvidence`) that the new, blessed authoring path does not require the author to state.

**(b) Classification (value-guarded conditional):** one node shape `sh:targetSubjectsOf opda:evidenceType`, with the per-kind obligation expressed as a **material implication in pure SHACL Core** — `P→Q ≡ ¬P ∨ Q`:

```
sh:or (
  [ sh:not [ sh:property [ sh:path opda:evidenceType ; sh:hasValue "Vouch" ] ] ]    # not-Vouch → vacuously ok
  [ sh:property [ sh:path opda:attestedBy ; sh:minCount 1 ; sh:class prov:Agent ] ] # Vouch     → must attest
) .
```

This is `sh:or`/`sh:not`/`sh:hasValue`/`sh:property` — all SHACL Core §4, no `advanced=true`. (I correct my own first instinct here: `sh:qualifiedValueShape` is the wrong tool — it counts values *of a path*; the obligation we want is an implication *on the focus node*, which is the `sh:or` idiom above. The `sh:xone` + `sh:qualifiedValueShape` form in ODR-0013:59 / ODR-0010 §3 is for a **discriminated union of whole branches** — the right shape when the kinds have *disjoint* obligation sets, e.g. the BASPI5 seller-capacity `sh:xone` below. For a single per-kind add-on obligation the `sh:or`-implication is tighter.) Either way the Vouch branch fires on the **coded value**, with no dependency on the leaf `rdf:type` and no entailment. It catches the exact instance (a) misses.

**This is not theoretical — it is the in-tree house style.** The canonical MVP profile already discriminates a per-subtype obligation on a *coded value, not a class*: `opda:Baspi5_SellersCapacityShape` (`baspi5.ttl:65–70`) is `sh:targetClass opda:Seller ; sh:xone (…)` whose Personal-Representative/Attorney branch (`_:b12c3d2fd5c1c`, `baspi5.ttl:173–177`) keys on `sh:path opda:hasAssertedCapacity ; sh:in ("Personal Representative for a Deceased Owner" …)` and *then* demands `sh:minCount 1` on `opda:hasEvidencedAuthority` (`_:bdd6345faa412`, `:756–760`, B1.3.2-3). The seller's *capacity* is a coded facet; the conditional obligation (PR/Attorney ⇒ evidenced authority + attachments) dispatches on the **value**, not on a `opda:PersonalRepresentativeSeller` subclass that opda pointedly never minted. ODR-0010 §3 (Knublauch's own canonical mapping, 12-0) and ODR-0013's `oneOf`→`sh:xone`+`sh:qualifiedValueShape` row legislate this: discriminated unions dispatch on the discriminator **value**. Evidence-kind is the same shape of problem with the same answer. The corpus is already self-consistent on the value-keyed side; the subclass-targeted `VouchEvidenceShape` is the **outlier**.

**Does classification lose anything for validation?** No. Whatever a subclass-targeted shape can assert about Vouch, a value-keyed conditional asserts identically — `sh:class`, `sh:minCount`, `sh:datatype`, even nested `sh:node` — because the constraint *body* is the same property-shape; only the *target/activation* differs. The only thing the subclass carries that the value cannot is **structure-bearing in the class graph** (disjoint property signatures, `attestedBy rdfs:domain VouchEvidence`) — and S035 settled that the subclasses are **retained** for exactly that (documentary `rdfs:domain` under ODR-0026 §R2; UFO Relator/Information-Object distinction). My Q2 verdict does not touch the subclasses' existence. It touches **which target the *enforcement shape* keys on.** Validation should key on the value; the class graph keeps the ontology.

`Q2: AFFIRM — ballot: FOR — Classification validates the Vouch⇒attestedBy obligation strictly better: a value-guarded material implication (sh:or(¬P ∨ Q), pure SHACL Core §4) on sh:targetSubjectsOf opda:evidenceType fires on the corpus's now-blessed coded authoring path (opda:evidenceType, opda-claim.ttl:111) with zero entailment dependency, where the in-tree sh:targetClass opda:VouchEvidence shape (opda-claim-shapes.ttl:66) silently passes a Vouch recorded by value without the leaf type. The constraint body is identical; only activation differs. I find no per-subtype property constraint that ONLY a subclass sh:targetClass expresses. Precedent is in-tree: Baspi5_SellersCapacityShape (baspi5.ttl:65–70, 173–177) already dispatches a per-subtype obligation on the hasAssertedCapacity value via sh:xone, never on a subclass (ODR-0010 §3; ODR-0013:59 oneOf→xone+qualifiedValueShape for disjoint-branch unions).*

### Q3 — Disposition vs the in-tree implementation

The in-tree implementation is **half-right and incomplete**, and S035's own routing (ADR-0012: *"emit the per-subtype enforcement shape (`sh:qualifiedValueShape`, SHACL Core)"*) is **not yet satisfied**. Disposition:

1. **Emit the missing `opda:EvidenceFacetShape`** that `opda-claim.ttl:71` already advertises. One node shape, `sh:targetSubjectsOf opda:evidenceType`, carrying each per-value obligation as an `sh:or(¬P ∨ Q)` implication (SHACL Core §4 — no `advanced=true`; the ADR-0012 routing says Core, and Core suffices). Branch one: `evidenceType "Vouch"` ⇒ `attestedBy` `sh:minCount 1` `sh:class prov:Agent`. (The `opda-claim.ttl:71` comment names `sh:qualifiedValueShape`; the emitter should use the `sh:or`-implication for single add-on obligations and reserve `sh:xone`+`sh:qualifiedValueShape` for genuinely disjoint-branch unions per ODR-0013:59 — I'd correct the comment to "sh:or-guarded on opda:evidenceType" to match what is actually emitted.) This is the shape the corpus has owed since S009 and re-owed at S035. The comment currently advertises a shape that does not exist; either build it or strike the comment — and Q1/Q2 say build it.
2. **Retarget the Vouch obligation from subclass to value.** Move the `attestedBy`/`prov:Agent` constraint out of `VouchEvidenceShape` (`sh:targetClass opda:VouchEvidence`) and into the `EvidenceFacetShape` Vouch branch (keyed on `opda:evidenceType "Vouch"`). **Keep** `EvidenceTypeValueShape` (the `sh:in` membership gate) — it is correct and complements the facet shape (one says "the value is legal", the other says "given the value, here are the obligations"). This makes the *enforcement* uniformly value-keyed, matching `ownerType` and `Baspi5_SellersCapacity`.
3. **Whether to *delete* `VouchEvidenceShape` (the `targetClass` form): I say retire it, with one caveat.** Functionally it becomes redundant once (2) lands — two shapes asserting the same `attestedBy` obligation on overlapping targets is the kind of double-mint S035 spent a whole session deleting (the alias). **But** there is a real interoperability argument from S035's synthesis (Isaac-Miles's `skos:exactMatch` reconciliation, S035 §"residual fork"): a consumer may legitimately dispatch on `rdf:type opda:VouchEvidence` *or* on the coded value and should get the same answer. A `targetClass` shape serves the rdf:type-dispatching consumer. **My disposition: collapse to the value-keyed `EvidenceFacetShape` as the single enforcement home, and if a named consumer needs rdf:type-side enforcement, re-add `VouchEvidenceShape` as a thin `sh:node opda:EvidenceFacetShape`-style delegation rather than a duplicated constraint body.** Do not maintain two independent copies of the obligation. (This is the same "don't duplicate, delegate" discipline ODR-0010 §2 applied to merged `sh:in`.)
4. **DASH:** the `EvidenceFacetShape`'s `opda:evidenceType` property shape gets `dash:editor dash:EnumSelectEditor` + `dash:viewer dash:LabelViewer`, fed by the `sh:in` over `EvidenceMethodScheme` — identical to every other coded facet in `baspi5.ttl`. The evidence-kind choice becomes a renderable form field; under the pure-subclass model it never could be.

**Net:** the proposition is **AFFIRMED for the enforcement layer** — key SHACL on the coded value; it is entailment-free, DASH-renderable, and catches the case the subclass shape misses. The subclasses **stay** (S035, structure-bearers; not my lens to relitigate). The concrete defect this session must fix is that `EvidenceFacetShape` is promised and unbuilt, and the one per-subtype obligation that shipped shipped on the wrong target.

`Q3: REVISE — ballot: FOR — The in-tree implementation is incomplete against its own S035/ADR-0012 routing: opda:EvidenceFacetShape (promised at opda-claim.ttl:71) is unbuilt, and the Vouch⇒attestedBy obligation shipped subclass-targeted (opda-claim-shapes.ttl:66) — the entailment-relative form (SHACL §2.1.3.1) that S035/ODR-0026 §R2 warned against. Disposition: emit EvidenceFacetShape as the value-keyed enforcement home (sh:targetSubjectsOf opda:evidenceType + per-value sh:or(¬P ∨ Q) implication, SHACL Core §4; sh:xone+sh:qualifiedValueShape only for disjoint-branch unions); retarget the Vouch obligation onto evidenceType "Vouch"; retain EvidenceTypeValueShape (the sh:in gate); collapse the duplicated VouchEvidenceShape to a thin sh:targetClass…sh:node delegation only if a named rdf:type-dispatching consumer requires it; correct the opda-claim.ttl:71 comment to name the shape actually emitted. Keep the subclasses (S035). Add dash:EnumSelectEditor on the facet (ODR-0013 §DASH).*

### Rebuttal (one pass)

**The cross-talk converged the panel; my rebuttal is not a counter but the integration of Guizzardi's one valid residue, which I concede and fold in.**

Guizzardi is right, and it is the sharpest point in the session: value-keyed enforcement fires on the value and is **blind to the asserted `rdf:type`**, so `EvidenceFacetShape` alone cannot reject the two *incoherence* cases —
- `[] a opda:DocumentEvidence ; opda:evidenceType "Vouch"` (the Information-Object class carrying the Relator's discriminator), or
- `[] a opda:VouchEvidence` with `evidenceType "Document"` or absent (the Relator class with the wrong/no code).

`skos:exactMatch` (`opda-claim.ttl:55/65/98`) *documents* `VouchEvidence ⟺ "Vouch"`, but under ODR-0026 §R2 it is **not evaluated**, so nothing *enforces* the agreement. This is exactly the drift the dual layer exists to prevent, and it is the one gap where leaving enforcement purely value-keyed *would* erode something — not the Relator itself (that survives in the class graph), but the guarantee that the coded label still means what the class says. I had treated `skos:exactMatch` as making the two dispatch paths *interoperable* (S035 synthesis); Guizzardi correctly notes interoperability under a non-evaluated bridge is **documented, not enforced**. Concede.

**The coherence shape lands cleanly in SHACL Core — with one honest asymmetry I'll state precisely, because it actually strengthens my Q1 thesis rather than dilutes it:**

- **Direction A (class → value), fully clean.** `sh:targetClass opda:VouchEvidence ; sh:property [ sh:path opda:evidenceType ; sh:hasValue "Vouch" ]`. Pure Core (`sh:hasValue`, SHACL §4.5). Fires on any node *asserted* `a opda:VouchEvidence` and demands the code agree. Catches case 2. (It is entailment-relative *in target* — but here that is harmless and correct: it should fire precisely on the asserted Relator class.)
- **Direction B (value → class consistency), also Core, via the existing facet shape's negative branch.** Add to `EvidenceFacetShape` (on `sh:targetSubjectsOf opda:evidenceType`) a guard: `sh:or ( [ sh:not [sh:property[sh:path opda:evidenceType; sh:hasValue "Vouch"]]] [ sh:not [ sh:class opda:DocumentEvidence ] ] [ sh:not [ sh:class opda:ElectronicRecordEvidence ] ] )` — "if the code is Vouch, the node is not asserted to be a *conflicting* sibling class." Pure Core (`sh:or`/`sh:not`/`sh:class`/`sh:hasValue`). Catches case 1.

**The honest asymmetry:** Direction B's `sh:class` test *is* entailment-relative (SHACL §4.1, `sh:class` checks `rdf:type` reachability under the active regime). That is fine and in fact *the point* — coherence checking is the **one place where consulting `rdf:type` is exactly what you want**, because the whole job is to compare the type against the value. So accepting Guizzardi's residue does not undercut my Q1 verdict; it *sharpens* it: **enforcement of the per-kind obligation keys on the value** (entailment-free, the §R2-safe path), while **coherence between the two layers** is the dedicated, separate shape that *deliberately* consults the class — they are different jobs and should be different shapes, each keyed on the target appropriate to its job. This is the same "two shapes, two jobs" discipline Cagle and I already drew between the `sh:in` gate and the facet shape. It is a *third* shape (`opda:EvidenceClassCoherenceShape`), not a reason to move the obligation back onto the class.

So the disposition gains one item and loses nothing:

5. **Emit `opda:EvidenceClassCoherenceShape`** — Direction A (`sh:targetClass` per subclass ⇒ `sh:hasValue` the matching code) + Direction B folded into the facet shape's value-guard (code ⇒ not a conflicting sibling class). Core only, no AF. Severity `sh:Violation` (an incoherent type/value pair is an identity-contract breach under ODR-0013's rubric — the node claims to be two different evidence kinds at once). This **closes the drift gap** that S035's `skos:exactMatch` documented but ODR-0026 left unevaluated, and it does so *in the layer that should hold it* (SHACL, not OWL `equivalentClass` — which ODR-0026 §R2 forbids evaluating anyway). With it, the dual layer cannot silently diverge.

**Net effect of the cross-talk on my position:** Q1 and Q2 unchanged and *reinforced* (the coherence case is the exception that proves the rule — value for obligations, class only where you are deliberately checking the class). Q3 gains item 5 (the coherence shape). The proposition "prefer classification" is **AFFIRMED for the enforcement and forms layers**, with the precise, SHACL-grounded boundary that the *one* legitimate use of class-targeting here is inter-layer coherence — exactly because that is the one constraint a coded value cannot express about itself. Zero residue; the DA is fully aligned.

`Coherence addendum (folds into Q3): REVISE — ballot: FOR — Accept Guizzardi's class↔value coherence residue: value-keyed enforcement is rdf:type-blind, so emit a dedicated opda:EvidenceClassCoherenceShape — Direction A (sh:targetClass opda:VouchEvidence ⇒ sh:hasValue "Vouch", SHACL §4.5) + Direction B (the facet shape's value-guard: code ⇒ sh:not conflicting sibling class, SHACL §4.1/§4.5), Core only, sh:Violation.
