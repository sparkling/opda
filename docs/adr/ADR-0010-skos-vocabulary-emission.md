# SKOS vocabulary emission

## Context and Problem Statement

[ODR-0011](../ontology/odr/ODR-0011-enumeration-vocabularies.md) ratifies the SKOS substrate: every JSON enum becomes a `skos:ConceptScheme`; per-member `skos:prefLabel @en` + `skos:notation` + `skos:definition @en`; per-scheme **UFO meta-category declaration** (seven-category framework â€” Quale-in-Region; Role label; Phase label; Method/plan code; Quality Region; Substance Kind label; Quality Value).

This ADR ratifies the **engineering emission** of those schemes as `opda-vocabularies.ttl`. The substrate must land before any module ADR (ADR-0011) because every `kind: pattern` module references SKOS schemes for category-like attributes.

**Inputs to the emission:**

- ~160 enum leaves from `pdtf-transaction.json` + overlay schemas (built-form; council-tax band; EPC band; tenure kind; central-heating fuel type; participant status; assurance level; etc.).
- ODR-0011 Â§8a seven-category UFO framework (which category each scheme commits to).
- Per-scheme `dct:source` to glossary or regulator (ODR-0011 Â§4a verbatim-citation for regulator-governed concepts).
- ODR-0011 Â§5a SHACL-AF deprecation-chain rule (Cagle).
- A9 per-kind discipline output (UFO category + IC + artefact realisation).

**Decision space:**

- **Scheme-level structure**: per-scheme `skos:ConceptScheme` + per-member `skos:Concept`.
- **Naming convention** for scheme URIs (`opda:BuiltFormScheme`) and member URIs (`opda:builtForm/Detached`).
- **Per-member metadata** depth (prefLabel + notation + definition; optional broader/narrower; deprecation lifecycle).
- **UFO category declaration** mechanism (`dct:source` + `skos:scopeNote`).
- **Cross-vocab mapping** placement (deferred per Phase-3.5 audit; admission case-by-case).

## Decision Drivers

* **ODR-0011 ratification is dense** â€” every member metadata field, every UFO category, every regulator-citation rule is in `## Rules` or `## Operational specifications`. Engineering applies; it does not re-deliberate.
* **A9 per-kind discipline** ([ODR-0001](../ontology/odr/ODR-0001-linked-data-council-methodology.md) Â§What an ODR records) requires the UFO category + IC + artefact realisation MUST land for `kind: pattern` schemes. ODR-0011 is the meta-ODR; each scheme is itself a pattern instance.
* **Regulator-governed concepts** ([ODR-0011 Â§4a](../ontology/odr/ODR-0011-enumeration-vocabularies.md)) require verbatim-citation discipline for PII categories (DPV-PD scope-note discipline; Pandit's S011 Q4 amendment).
* **Deprecation lifecycle** ([ODR-0011 Â§5a](../ontology/odr/ODR-0011-enumeration-vocabularies.md) â€” Cagle SHACL-AF deprecation-chain rule): retired members carry `dct:isReplacedBy` / `prov:wasDerivedFrom` / `dct:modified` per case.
* **Cross-vocab mapping is case-by-case admission** per Phase-3.5 audit. Initial emission does NOT include `skos:exactMatch` to external vocabs; admission triggers convene per-scheme mini-sessions.

## Considered Options

* **A â€” One `skos:ConceptScheme` per JSON enum, separate file per scheme.** Pro: per-scheme isolation. Con: ~160 files; file-naming churn; downstream `owl:imports` overhead.
* **B â€” One unified `opda-vocabularies.ttl` with all schemes (chosen).** Pro: single file; one `owl:imports` from each module; CI byte-identity straightforward. Con: large file (~3000 lines projected); but TTL is text â€” large is fine.
* **C â€” Per-module vocabularies** (vocabularies.ttl per module ADR). Pro: scope-local. Con: cross-module reused schemes (e.g. `participantStatus` used by Agent + Transaction) duplicated; reconciliation overhead.

## Decision Outcome

Chosen option: **B â€” One unified `opda-vocabularies.ttl`**, because cross-module reuse is high (the same Phase label scheme drives Agent + Transaction; the same Quale-in-Region scheme drives Property + Address Variants), single-file emission has lower import overhead, and CI byte-identity is mechanically straightforward.

### Per-scheme metadata MUST-haves (per ODR-0011 Â§1a + Â§8a + A9)

For every scheme, the emission MUST land:

1. `skos:prefLabel @en` â€” exactly 1 (ODR-0011 Â§S14/S15).
2. `dct:title @en` (catalogue display).
3. `skos:definition @en` (one canonical definition).
4. `dct:source` to authoritative origin (W3C spec / OPDA TF / business glossary / data dictionary / external regulator â€” five-line precedence per ODR-0004 Â§7a).
5. `opda:ufoCategory` declaration (one of seven; ODR-0011 Â§8a).
6. `skos:scopeNote @en` with UFO + DOLCE backing reference (Guizzardi 2005 chapter / Masolo D18 section).
7. `opda:hasSteward` (per S008 Q2 module-owner-proposes; ODR-0011 Â§1a).

For every member, the emission MUST land:

1. `skos:inScheme` (scheme URI).
2. `skos:prefLabel @en` â€” exactly 1.
3. `skos:notation` â€” exactly 1 (canonical machine-readable form).
4. `skos:definition @en` â€” exactly 1 (per ODR-0011 Â§S14/S15).
5. `dct:source` to per-member canonical origin.

Deprecated members additionally carry one of three lifecycle predicates per ODR-0011 Â§5a:

- `dct:isReplacedBy <new-member>` (clean supersession).
- `prov:wasDerivedFrom <prior-form>` (lineage-only; no supersession).
- `dct:modified <date>` (in-place modification with audit trail).

### SHACL-AF rule emission (Cagle's deprecation-chain rule)

Per ODR-0011 Â§5a + ODR-0017, the deprecation discipline is enforced by a SHACL-AF rule. The rule emits into `opda-shapes.ttl` (per ADR-0012 emission); the rule body is:

  }

Materialised at `sh:Info` severity (ODR-0017 non-blocking).

### Consequences

* Good, because the SKOS substrate is one file, atomically emitted; downstream module ADRs cite it once via `owl:imports`.
* Good, because A9 per-kind discipline output is mechanical â€” every scheme carries `opda:ufoCategory` + `skos:scopeNote` UFO + DOLCE citations.
* Good, because regulator-cited schemes (CouncilTaxBand, EPC band, etc.) honour ODR-0011 Â§4a verbatim-citation discipline at emission time.
* Good, because deprecation-chain SHACL-AF rule (Cagle) emits into the shapes graph for non-blocking quality monitoring.
* Good, because per-scheme steward attribution (per S008 Q2) lands in metadata, supporting downstream amendment proposals.
* Bad, because the file is large (~3000 lines projected for ~160 schemes). Mitigation: rdflib + canonical serialiser handle large TTL fine; reviewer cognition is the cost, not parser.
* Bad, because `skos:exactMatch` to external vocabs is deferred (per Phase-3.5 audit). Some downstream consumers may request cross-vocab mapping earlier than expected. Mitigation: per-scheme admission via Author-only mini-sessions per Phase-3.5 audit Item 2.
* Neutral, because some schemes have ratified UFO categories (per ODR-0011 Â§8a explicit list) and others are inferred from the data dictionary; the inference rule is mechanical (data-dictionary leaf shape â†’ category).

The ADR is honoured when all six hold:

1. **Emission lands.** `opda-gen emit-vocabularies` produces `opda-vocabularies.ttl` in `source/03-standards/ontology/`.
2. **Byte-identity CI green.** Regeneration produces zero diff.

4. **Every scheme carries UFO category.** SPARQL: `SELECT ?s WHERE { ?s a skos:ConceptScheme . FILTER NOT EXISTS { ?s opda:ufoCategory ?c } }` returns empty.
