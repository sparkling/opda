# Manual build-time remark + rehype plugins

## Context and Problem Statement

The manual's markdown content (228 files at `docs/manual/`) was generated by the `/diagramming` skill's post-creation export workflow. Mermaid blocks are stored as `<details><summary>Mermaid Source</summary>\n\`\`\`mermaid\n‚Ä¶\n\`\`\`\n</details>` paired with an `<img>` reference to the offline PNG (per the skill's contract). The PNGs at `docs/manual/<tier>/diagrams/<doc>/*.png` are offline-export artefacts; the live Astro site renders Mermaid client-side via `public/ui/client.js` (per ADR-0015 ¬ß"Mermaid integration").

For the Astro build pipeline to render the manual correctly, two transformations are needed at build time:

1. **`remarkUnwrapMermaidDetails`** ‚Äî walks markdown AST; for each `<details><summary>Mermaid Source</summary>` block containing a `mermaid` fenced code block, replaces the entire `<details>` (and the immediately-preceding `<img>` reference) with `<div class="mermaid">‚Ä¶mermaid source‚Ä¶</div>` so the existing client.js loader picks it up.
2. **`rehypeFrontmatterUriExtraction`** ‚Äî for entries of `kind: entity` / `scheme` / `exemplar` (per the Zod schema in ADR-0016), extracts the OPDA URI from the markdown body (e.g. the first `### opda:<EntityName>` in Physical-Ontology classes.md sections; the page title for entity files) and writes it into the entry's data as `entityUri`. Enables the Cross-Tier Links footer (per ADR-0017's `CrossTierLinks.astro`) without re-parsing markdown at render time.

Per the [implementation programme plan ¬ß4](../plan/manual-astro-integration.md), this is **Phase 3** ‚Äî runs in parallel with Phase 2 (ADR-0017 components) once Phase 1 (ADR-0016) is green.

Realises ADR-0015 `¬ßConfirmation` criteria 5 (`Diagram.astro` unchanged ‚Äî site reuses the existing loader; no parallel setup), 7 (ELK-laid-out diagrams render correctly via the unwrapped `<div class="mermaid">`), 8 (no `docs/manual/_export/` files referenced from `src/`).

## Decision Drivers

* The PNG `<img>` references in the markdown are local-export-only ‚Äî they must NOT appear in the emitted HTML (would 404 since `docs/manual/_export/` is gitignored and never shipped).
* The `<details>` source IS the canonical mermaid ‚Äî unwrap and live-render so dark-mode toggling works (existing `client.js` re-runs `mermaid.run()` on theme change).
* Plugins live at `src/lib/remark/` per the Astro project convention.
* Wired into `astro.config.mjs` via the `markdown.remarkPlugins` + `markdown.rehypePlugins` arrays.
* Plugins are tested via the standard remark-test pattern (small fixture markdowns + expected AST output).
* Both plugins together are <100 lines; small surface area to maintain.

## Considered Options

* **A ‚Äî Two custom plugins (`remarkUnwrapMermaidDetails` + `rehypeFrontmatterUriExtraction`)** ‚Äî chosen per ADR-0015.
* **B ‚Äî Use an existing remark plugin (e.g. `remark-mermaidjs`)** ‚Äî those plugins typically render mermaid to SVG at build time, which (a) duplicates the existing client.js setup, (b) loses dark-mode coordination, (c) doesn't handle ELK consistently.
* **C ‚Äî Hand-edit each markdown to remove `<img>` + unwrap `<details>`** ‚Äî defeats the regenerability discipline.
* **D ‚Äî Astro integration package wrapping both plugins** ‚Äî over-engineered for 100 lines of code.

Specific to this ADR:

1. Both plugins exist at `src/lib/remark/unwrap-mermaid-details.ts` + `src/lib/remark/frontmatter-uri-extraction.ts`
2. `astro.config.mjs` wires both plugins in the markdown pipeline
3. Build emits **zero** `<details><summary>Mermaid Source</summary>` blocks in the dist HTML (every one transformed to `<div class="mermaid">`)
4. Build emits **zero** `<img>` references pointing at `docs/manual/<tier>/diagrams/` paths (all stripped)
5. Build emits **zero** references to `docs/manual/_export/` from any `src/` or `dist/` file
6. ELK-flagged mermaid blocks (those with `config: layout: elk` in their YAML frontmatter) render via the ELK plugin client-side ‚Äî confirmed by manual smoke test on a vocabulary scheme membership graph
7. Per-entry `entityUri` field populated for every `kind: entity` / `scheme` / `exemplar` entry; `getCollection('manual')` returns entries with `entityUri` non-null where expected
8. Validation report at `docs/adr/validation/ADR-0018-validation-report.md`
