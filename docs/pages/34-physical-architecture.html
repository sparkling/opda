Physical architecture

The PDTF data standard exists, physically, as three Git repositories of JSON Schema files, an OpenAPI specification, a small JavaScript merge engine, and a set of JSON-LD and DID artefacts. This page maps those artefacts and how they fit together — the concrete codebase the overlays, data dictionary and ontology are all derived from.

What "physical" means here

Modelling covers the semantic layer — what the data means. This page covers the physical layer — the files, packages, versions and engines that the data standard is actually shipped as. Three repositories, all under source/03-standards/: @pdtf/schemas, @pdtf/api and @pdtf/trust-framework.

The three repositories

PDTF is not a single artefact. It is three repositories with distinct responsibilities and release cycles, all published under the trust.propdata.org.uk domain that every $id and context URL points at.

Inside @pdtf/schemas

Almost all of the physical schema content lives in one repository. Its layout separates the base schema, the overlay files, the modular extension overlays, the verified-claims wrapper, and the engine (index.js) that composes them. A small v2/ tree is still shipped for backward compatibility.

The base transaction schema

pdtf-transaction.json defines a single root object — the transaction. Its biggest member by far is propertyPack, an object of roughly 55 property groups covering everything a buyer, seller, conveyancer or lender needs to know: address, price, ownership, construction, energy, searches, disputes, fixtures, and so on. Five of propertyPack's members are arrays — titles, searches, documents, surveys and valuations — which model cleanly as related entities.

Overlays and extensions

The base schema is deliberately permissive — almost nothing is required. The overlays are what make it strict and form-specific. Each overlay is a draft-07 schema that mirrors the base's shape but only specifies what it adds or constrains: a form-reference attribute on each field (so a value traces back to a section number on a paper form), required arrays, form-specific titles, and oneOf discriminators. The 16 extension overlays are sub-pieces of NTS2 — they let an adopter stage their migration from NTS 2023 to NTS 2025 one feature at a time.

The merge and validation engine

index.js is the only executable code in the schemas repository, and it is small. Three functions do the work:

  • getTransactionSchema(schemaId, overlays[]) — composes the base schema with the chosen overlays and extensions.
  • getValidator(schemaId, overlays[]) — returns a compiled AJV validator for that combination.
  • validateVerifiedClaims(claims, schemaId, overlays[]) — checks that each claim's JSON Pointer resolves to a real path in the merged schema.

Composition is deepmerge with a custom arrayMerge strategy: required arrays are unioned, oneOf arrays are concatenated, and everything else follows ordinary object-merge rules. Validation is AJV configured for draft-07, with discriminator support and ajv-formats for date and URI formats. Merged schemas and validators are cached.

Verified claims

A transaction document says what is true about a property. A verified claim says who attested it, how, and with what evidence. The pdtf-verified-claims.json wrapper is the v3 provenance model. Each claim entry is keyed by a JSON Pointer into the merged transaction schema — for example /propertyPack/priceInformation/price — and is backed by a verification block whose evidence is one of three types: document, electronic_record or vouch. Supporting files are linked by cryptographic digest.

The exchange API

The @pdtf/api repository is the wire protocol — an OpenAPI 3.0.3 specification, shipped as four files from 1.1.0 through 2.0.0. It defines how participant services exchange data, organised into four tags: managing Transactions, reading and writing Claims, webhook Subscriptions for real-time updates, and trust-framework Discovery.

The trust layer (PDTF v2)

The @pdtf/trust-framework repository is where PDTF is heading. Instead of a verified-claims wrapper around a single transaction document, PDTF v2 issues per-entity W3C Verifiable Credentials — one credential each for a Person, Organisation, Property, Title or Transaction. Issuers and subjects are identified by did:web DIDs; credentials are signed with a DataIntegrityProof; access is governed by a termsOfUse policy; and a registry.json plus a JSON-LD context tie the vocabulary together.

Two generations of provenance

The schemas repository and the trust-framework repository represent two generations of the same idea — attaching trust to property data. Both are physically present in the codebase today; v3 verified claims are what production uses now, and the W3C VC model is the roadmap direction.
